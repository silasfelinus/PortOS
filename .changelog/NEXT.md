# Unreleased Changes

## Added

- **[pipeline-audio-phase-4c-2-4c-3-4d-2-local-oss] The pipeline audio stage can now generate background music locally, and voice-over lines are mixed into the final episode.** Background Music gains a "Generate (MusicGen)" option that renders a text-prompted track on-device via MusicGen on Apple Silicon — pick a model tier (Small / Medium / Large) and a clip length, and the result lands in your music library attached to the issue, no upload or API key needed (one-time setup: `INSTALL_MUSICGEN=1 bash scripts/setup-image-video.sh`; until then the panel shows an install hint). Each voice-over line also gets a "Start (s)" field that places it on the episode timeline, and when the final cut is stitched those rendered lines are now mixed in at their offsets — with any background music automatically ducked beneath the dialogue so speech stays clear. (Commercial music engines like Suno are intentionally not wired up yet.)

- **[conflict-journal-media-collections] Media collection name/description/cover edits are now protected when two machines change the same collection.** If a peer's edit to a collection's name, description, or cover image would overwrite a change you made independently, your version is archived to Sharing → Conflicts so you can review and restore it — the same safety net universes and series already had. Images filed into the collection were never at risk (both sides' images are always kept); only the collection's text/cover fields could silently lose an edit, and now they don't.
- **[story-builder-issue-llm-seeding] Story Builder can now generate issues straight from the arc.** The issues step gains a "Generate issues from arc" button that breaks every season of your plot arc into episodes and creates one issue per episode — seeded with each episode's logline and synopsis — without leaving the builder for the Pipeline. Locked or empty seasons are skipped and reported rather than blocking the rest, and you can still plan issues in the Pipeline as before.
- **[brain-bucket-chip-reorder] Reorder bookmarks within a bucket by dragging.** On the Brain Links bucket board you can now drag a chip to a new spot inside its own bucket — a marker shows where it will land as you drag — instead of only being able to move chips between buckets or reorder the buckets themselves.
- **[local-llm-large-narrative-models] Local LLM install picker now suggests a large narrative / long-context tier.** Added Mistral Large 2 123B, Command R+ 104B, Llama 3.3 70B, Qwen3 30B-A3B, and Google Gemma 4 (31B dense + 26B-A4B MoE, both 256K context, vision) to the curated catalog — workstation-class (64–128GB unified memory) models suited to whole-manuscript editorial review where prose quality and a long context window matter most. Verified Ollama pull ids and sizes; the Gemma 4 builds also ship MLX variants (`gemma4:31b-mlx` / `gemma4:26b-mlx`) for faster inference on Apple Silicon. Qwen3 235B is intentionally omitted (142GB won't fit 128GB).
- **[context-aware-editorial-budgeting] Manuscript editorial passes now scale to the target model's context window — whole-manuscript when it fits, chunked when it doesn't.** A new `contextBudget` helper plus a `contextWindow` provider field (inferred large for frontier CLI/cloud providers, from `numCtx` for local Ollama) drive all three passes: the completeness pass ("Finish the draft"), manuscript-fix ("Generate fix"), and per-issue editorial analysis. Big-context models (Claude 4.8 ~1M, GPT 5.5) now receive the entire manuscript in one call instead of being capped at the old hardcoded 200K/48K-char limits; models too small to hold the book chunk by issue — completeness feeds each chunk a digest of prior-chunk findings for cross-chapter continuity and merges with first-wins dedupe, manuscript-fix concatenates per-chunk edits — instead of silently truncating. The completeness/fix API responses now report `chunked` + `chunkCount`, and the Manuscript Editor shows a "Reviewed in N chunks" badge (with an explanatory tooltip) when a review had to chunk. Design record: `docs/plans/2026-05-31-context-aware-manuscript-editorial.md`.

- **[video-resume-restore-keyframes] Video Gen first/last-frame mode now has a multi-keyframe picker, and a mid-render reload restores it.** With an LTX-2 model selected you can switch on multi-keyframe interpolation and anchor 2–8 gallery images at specific frame positions for the model to interpolate between, instead of only a single first and last frame. Reloading the page while such a render is in flight now repopulates the keyframe picker — each anchor's image and frame position — along with the rest of the form, rather than losing it.
- **[scene-level-wardrobe-picking-per-scene] Storyboard scenes can now pin each character to a specific outfit.** Each scene in the Storyboards stage gains a collapsible "Wardrobe" picker listing the characters the scene mentions; choose one of a character's saved wardrobes and that outfit's description is added to the character's look for that scene's image and video renders, on top of (never replacing) their canonical physical description. Picks are per-scene and explicit — nothing is guessed — and selecting "default outfit" clears the pin.
- **[ai-assignment-inventory] Settings now has an AI Assignments admin page.** It inventories provider/model pins across the provider registry, embeddings, Voice, Messages, Brain, Universes, Story Builder, Pipeline series, CoS schedules, Loops, Feature Agents, and social agents, with per-row edits plus a bulk provider migration control.
- **[manuscript-multi-edit-fixes] Manuscript "Finish the draft" fixes can now propose and apply multiple anchored edits at once.** The manuscript-fix LLM contract moved from a single `{ find, replace }` to `{ edits: [{ issueNumber, find, replace, note }] }`, so a broad or unanchored editorial comment can generate several concrete insertions — across multiple issue sections when needed — instead of falling back to manual editing. The editor's comment card renders each suggestion as an inline diff with a per-edit checkbox and editable replacement, and accept applies the selected edits atomically through a new serialized batch write (`updateStagesWithLatest`) that snapshots prior text into runHistory and re-checks each anchor inside the per-series write lock. The prompt template embeds manuscript/premise/anchor content in distinct (non-backtick) fences with an explicit "treat as quoted source" guard; migration `060-manuscript-multi-edit-fix-prompt.js` auto-upgrades the prior single-edit prompt on existing installs. Backward compatible: legacy single-edit `{ find, replace }` accepts still resolve to one target.
- **[provider-fallback-model] Provider fallbacks can now pin a model, not just a provider.** Each AI provider's editor gains a "Fallback Model" selector beside "Fallback Provider" (populated from the chosen fallback provider's model list; blank = use that provider's own default), persisted as a new `fallbackModel` field. `providerStatus.getFallbackProvider()` returns the configured model alongside the provider, and every fallback path runs it: `createRun`'s pre-flight swap, `runPromptThroughProvider`'s inner `createRun` and runtime-retry paths (staged LLM + all prompt callers), and the CoS-agent lifecycle (provider- or task-level pin overrides the per-task model selection).

## Changed

- **[series-merge-season-rehoming] Merging two series now keeps each issue in its season.** When you merge a duplicate series into another, issues that belonged to a season are placed into the surviving series' matching season (paired by season number) instead of all landing un-grouped for you to re-file by hand. Issues that had no season stay un-grouped, as before.
- **[conflict-journal-deep-diff] The Sharing → Conflicts tab now shows which sub-entry changed, not a wall of JSON.** For structured fields — universe categories and canon (characters/places/objects), series arc and seasons, issue stages — a conflict now breaks down into one focused before/after diff per changed entry, labelled by its name or key (e.g. just the character "Alice", or the "tone" category), instead of diffing the entire field as a single JSON blob. Restoring and merging still operate on the whole field.

- **[conflict-journal-issues] Sync conflicts on individual issues are now captured and restorable, like universes and series.** When two of your machines independently edit the same issue while offline and then sync, the older edit is no longer silently overwritten — it's archived in Sharing → Conflicts where you can review what differs and restore your version (covering the issue's title, status, arc placement, length, and stage content such as prose and comic pages). Previously only universe and series conflicts were journaled; child issues rode last-write-wins silently.
- **[manuscript-editorial-jump-pinned-overlay] Jumping to an editorial comment in the Manuscript editor now keeps that comment on screen.** Clicking "Jump" (or the issue link) on an editorial feedback item still scrolls the manuscript to its anchor, but the comment is now also pinned as a floating, dismissable overlay over the manuscript — so its note, suggestion, and Generate-fix/Accept/Dismiss actions travel with the scroll instead of being lost off-screen (previously the sidebar card stayed put on desktop and was pushed far below the fold on mobile's stacked layout). The overlay reuses the existing comment card, auto-closes when the comment is accepted or dismissed, and dismisses on Escape or its close button.
- **[rich-text-editor-for-prose-stage-replace-plain] The pipeline prose stage now edits in the same reading-comfortable editor as the Writers Room.** Drafting a prose stage uses a serif, roomy-line-spacing, spellchecked editor with a heading-hint placeholder instead of the old code-style monospace box — so prose reads like prose while you write it. The script-shaped stages (comic script, teleplay) keep their monospace editor.
- **[inline-confirm-row-primitive]** Consolidated the duplicated Brain delete-confirmation rows onto one shared component, so inline confirm prompts stay visually and behaviorally consistent (no user-facing change).
- **[inline-confirm-row-border-b-variant] Delete confirmations across Notes, the wiki browser, the daily log, and link buckets now share one consistent look.** The four full-width "are you sure?" bars moved onto the same shared confirm component as the rest of the app, so their buttons and text now match every other inline delete prompt.
- **[confirm-ux-button-pair-popover-audit] Inline "confirm/cancel" delete buttons across Apps, Feature Agents, local-LLM models, time-capsule snapshots, character reference sheets, and catalog types now look and behave the same.** Six places that each had their own slightly different little confirm buttons now use one shared control, so confirming a delete feels identical wherever you do it.

- **[coalesce-fallback-provider-failure-storm] AI provider failovers no longer thrash the disk when many requests hit a down provider at once.** When a provider goes down with several requests in flight, PortOS now benches it and picks the fallback a single time for the whole burst instead of repeating that work per request, so failover stays fast and quiet under load.
- **[capabilities-cheap-memory-count] Capabilities overview counts Brain memories more cheaply.** The frequently refreshed status overview now reads the active-memory total directly, keeping it lighter as memory history grows.
- **[media-collections-per-record-store] Media collections now persist per-record instead of in one monolithic file.** `data/media-collections.json` is split into `data/media-collections/<id>/index.json` (plus a type-level `index.json` stamping the storage `schemaVersion`) via the shared `createCollectionStore` layout — the same per-record shape universes / series / issues already use. Every collection write (`addItem`, `bulkUpdateCollectionItems`, merges, tombstone GC) now serializes on its own record queue, so high-frequency render-filing no longer rewrites the whole ~200 KB document per image or blocks unrelated collections' writes. Migration `059-split-media-collections.js` performs the one-time split (idempotent, backs the legacy file up as `.bak-059`), the boot-time verifier checks the on-disk version, and the dataSync snapshot watcher tracks the directory so per-record edits still invalidate the checksum cache. The wire sync payload is unchanged, so collections still sync across peers of mixed versions.
- **[chrome-canary-followups] Custom Chrome setup writes browser config through the shared atomic writer.** The Canary setup script now uses `server/lib/fileUtils.js#atomicWrite` for browser-config updates, so it gets the same temp-file/rename behavior and Windows fallback as the rest of PortOS config persistence.
- **[setup-browser-canary-headless] The PortOS-managed browser now runs visible (headed) by default.** New installs — and opting into Chrome Canary at setup — launch a window you can see, since the whole point of a dedicated managed browser is to watch its automation. On update, existing installs still carrying the old headless default are flipped to headed; if you'd previously turned headless *off* (or any non-default value) it's left untouched. If you actually want headless, set `"headless": true` in the browser config again.
- **[media-job-throttled-progress-persist] Live media-job progress is now debounce-persisted to disk so a mid-render restart recovers the last snapshot.** `runJob` schedules a ~250ms-debounced `persist()` after each `progress`/`statusMsg` mutation — coalescing the dozens-per-second diffusion progress events instead of writing per event — and drains it in `terminate()` before the terminal transition, mirroring the `appendAgentOutputLines` batching pattern. The terminal sink (`terminate`) is now async and made idempotent via a `terminalStarted` guard, which also closes the watchdog `mod.cancel` race the async drain would otherwise widen.
- **[media-job-store-progress-on-job-record] Render queue rows now hydrate live progress from the job record.** The media job queue now stores `progress` and `statusMsg` on the running job whenever image/video runners emit progress or status events, exposes those fields through `GET /api/media-jobs/:id` and list responses, and preserves them in persisted queue snapshots. The Image / Video render queue uses that snapshot to show a compact progress bar during polling, so non-SSE consumers no longer reset in-flight jobs to 0% until the next live event arrives.
- **[shared-bounded-concurrency-mapper] Bounded-concurrency worker pool extracted to a shared helper.** The cursor-based worker-pool idiom that was hand-rolled in `embeddings.js`, `catalogExtraction.js`, and `routes/imageVideoModels.js` is now a single tested `server/lib/mapWithConcurrency.js` (order-preserving, concurrency-bounded), with all three call sites migrated and behavior unchanged.
- Bumped the bundled slashdo submodule (`lib/slashdo`) to latest `main` (`11cb89c`).
- **[media-job-progress-persist-debounce-count-test]** Hardened the test suite so the media-job progress-save coalescing can't silently regress to a per-event disk write.
- **[unify-media-job-sse-hook-imagegen-videogen]** Image Gen and Video Gen now share one live-render progress subscriber, consolidating duplicated streaming code with no change to behavior.
- **[importer-progress-hook]** The Importer's live analyze-progress checklist now runs on a shared, tested progress hook instead of page-local socket wiring — no change to behavior.
- **[mediacollections-deterministic-loadone-fastpath]** Universe/series media-collection lookups now resolve their linked bucket in one direct read instead of scanning every collection, trimming the work done on each cross-machine sync push (no user-facing change).
- **[data-versioning-centralize-cosagents-settings] Saving settings is now crash-safe and serialized.** `settings.json` is written through the shared atomic temp-file/rename writer and overlapping saves run one at a time, so a mid-write crash can no longer truncate the file and two near-simultaneous settings changes can no longer clobber each other (no user-facing change).
- **[setup-data-import-drift-tables-from-migrations]** The setup script's "a prompt migration is pending" warning now derives its hash tables directly from the migration files instead of a hand-maintained copy, so the warning can't silently fall out of sync when a new prompt migration ships (no user-facing change).
- **[migration-043-line-ending-precedent]** Documented in the story-builder prompt seed migration that a future migration amending those prompts must switch to the hash-driven update pattern (no user-facing change).
- **[cos-on-demand-mark-app-review-started-dedupe]** When several on-demand improvement requests for the same app are processed in one CoS dispatch cycle, the app's activity record is now marked review-started once instead of once per request (no user-facing change).
- **[shared-tolerant-fetch-json-helper]** Hardened the HTTP clients that talk to local model backends, the managed browser, and the background agent runner: a non-JSON/HTML error response now surfaces the real error instead of crashing with "Unexpected token <", via one shared tolerant JSON-body reader (no user-facing change).
- **[tolerant-fetch-json-additional-clients]** Extended that same tolerant JSON-body reader to the remaining fetch clients (calendar/message sync, the memory and insights LLM calls, the local-LLM playground, Civitai, Telegram, and the code-review runner), so a non-JSON/HTML error body surfaces the real error instead of crashing — and the code-review runner now reports the server's actual error text instead of a misleading "no content" (no user-facing change).
- **[insights-callprovider-nonjson-test]** Added regression-test coverage pinning the insights LLM client's non-JSON-error guard so a malformed provider response can't silently overwrite the cached cross-domain narrative or theme analysis, and consolidated the hand-rolled fetch-`Response` test mocks across the server suites onto one shared helper (no user-facing change).
- **[cos-agent-output-write-amplification] Streamed CoS agent output no longer rewrites the entire agent-state file once per line.** A shared ~250ms output batcher now coalesces the runner's per-line output events and direct-CLI stdout/stderr into single state writes (draining before an agent is marked complete), cutting agent-state write churn on chatty agents from dozens of writes/sec to a few — with no change to the live output tail (no user-facing change).
- **[subagent-spawner-top-level-side-effects]** The CoS agent spawner now initializes explicitly at server startup instead of as a side effect of being imported, so importing it (in tests or elsewhere) no longer re-arms its event listeners and timers (no user-facing change).
- **[wardrobe-appearance-id-route-validation] Rendering a storyboard scene with a stale outfit pick now fails clearly instead of silently ignoring it.** If a scene's wardrobe selection points at a character or outfit that no longer exists (e.g. it was deleted after the pick was made), the render returns a clear error instead of quietly rendering the character without that outfit.

## Fixed

- **[voice-taskcomplete-announce-once-guard] The voice assistant no longer repeats "your coding task is done" if a finished task is edited again.** Each voice-dispatched coding task now announces its completion (or failure) once; a later edit to an already-finished task won't re-trigger the spoken line, while a task that genuinely changes outcome — say a blocked task that later succeeds — still speaks its new result.
- **[cos-runner-client-non-json-response-handling] CoS agent and run failures now surface the runner's actual error instead of a cryptic "Unexpected token <" crash.** When the background agent runner returns a non-JSON error response (for example while it is restarting mid-request), PortOS reports the runner's real message rather than choking on the unparseable body.
- **[cos-task-id-uniqueness-validation] Reordering a CoS task list with duplicate task IDs no longer silently drops tasks.** If a hand-edited or corrupted task file holds two tasks sharing the same ID, reordering it previously kept only one and discarded the rest; every task is now preserved, with any collision given a distinct ID.
- **[banner-icon-aria-hidden] Screen readers no longer announce the decorative icon on alert banners.** The warning/error/info banners' leading icon is now hidden from assistive tech by default, so only the message text is read aloud instead of an unhelpful icon name preceding it.
- **[media-job-sse-transient-reconnect-resilience] A brief network blip mid-render no longer fails the whole image or video generation.** The live progress stream now distinguishes a transient connection hiccup — which the browser silently reconnects, picking up where the render left off — from a real disconnect, so a momentary Wi-Fi/Tailscale drop while a render is in flight no longer aborts it with a "Lost connection to server" error.
- **[ollama-context-truncation] Local-Ollama runs no longer silently truncate long prompts to ~4K tokens.** PortOS sends stage prompts to Ollama's OpenAI-compatible `/v1/chat/completions`, which defaults to a ~4K context window and silently drops anything past it — so whole-manuscript editorial passes (the completeness pass concatenates up to 200K chars ≈ 50K tokens) were really only reviewing the tail of the manuscript with no error raised. The toolkit runner now sends a per-request `num_ctx` when the provider config carries one (a new optional `numCtx` provider field, honored by Ollama and ignored by other OpenAI-style endpoints), and `ensureBackendProvider` defaults the Ollama provider to a generous window at boot (`OLLAMA_NUM_CTX` env override, default 32768). LM Studio is unaffected (it sets context at model load). Cloud providers were never affected. This is the urgent half of the editorial context work; the full context-window-aware full-manuscript↔chunked strategy is planned in `docs/plans/2026-05-31-context-aware-manuscript-editorial.md` (`[context-aware-editorial-budgeting]`).
- **[provider-fallback-model-leak] Falling back to another provider no longer leaks the primary's model id onto it.** When a stage's primary provider was unavailable (e.g. Codex benched after an OpenAI content-safety refusal), the model resolved against the primary — `codex-configured-default` — was carried verbatim onto the fallback provider, so the run logged `LM Studio/codex-configured-default` and then 400'd because LM Studio has no such model (and Claude Code rejected it the same way). `stageRunner` now re-resolves the model against the *fallback* provider (honoring its configured `fallbackModel`, else its own default / args-baked model) instead of forwarding the primary's resolved model, and the toolkit `createRun` run record + log line do the same.
- **[ltx2-fflf-skips-last-image-resize-when-both-frames-set] ltx2 FFLF now resizes both anchor frames.** The two-keyframe ltx2 FFLF path passes both `--image` and `--last-image` into `scripts/generate_ltx2.py`, but Video Gen only resized the start image when both anchors were present. The end frame could therefore reach `KeyframeInterpolationPipeline.generate_and_save()` at its original dimensions. `videoGen/local.js` now treats ltx2 true-FFLF as a real last-image consumer and runs the same ffmpeg resize/crop pass used for multi-keyframes, with a regression test asserting the helper receives resized start and end paths.
- **[ref-watch-phosphene-teacache-extend-a2v-denoise] TeaCache now accelerates ltx2 Extend and A2V Stage 1.** `scripts/generate_ltx2.py` now patches the LTX-2 `extend` and `a2vid_two_stage` denoise-loop import sites (the modules where `ExtendPipeline`/`AudioToVideoPipeline` actually call `guided_denoise_loop`), activating the existing Stage-1 TeaCache controller only around Extend and A2V calls and clearing the gate afterward. TeaCache stays default-on for those slow paths, with `--no-teacache` to opt out and `--teacache-thresh` to trade fidelity for speed (~1.2× at the 0.5 default, up to ~3× at 1.5).
- **[dup-detect-collection-count-undercount] Duplicate-universe suggestions now count collection items correctly.** On the Sharing → Duplicates tab, a universe that owned more than one media collection had its collection-item total reported as just the last collection's size rather than the sum, which could understate which duplicate held the most content; the count now adds up every linked collection.
