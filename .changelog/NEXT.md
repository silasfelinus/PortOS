# Unreleased Changes

## Added

- **Refine Prompt action in the Media Lightbox.** Open any past image or video render and click *Refine Prompt* to describe what you'd like to change in plain English ("more painterly", "switch to watercolor", "drop the cyberpunk neon"). PortOS hands the original prompt + your feedback to any enabled AI provider — API (OpenAI, Anthropic, LM Studio) or CLI (Codex, Claude Code, Gemini CLI) — and shows the rewritten positive and negative prompts as editable text plus a one-line rationale. Tweak the new prompt if you want, then *Queue Render* to make a new render that inherits the original's model, resolution, LoRAs, seed, and other settings.
- **Pipeline — Arc canvas UI.** PipelineSeries's right-side canvas is now a structural Arc → Season → Episode tree. Top of the canvas shows the series arc card (logline, themes, collapsible summary + protagonist arc, edit-in-place fields). One-click `Generate arc` runs the Phase 3 LLM pass and persists `series.arc` + `series.seasons[]`; `Verify arc` surfaces continuity findings inline with severity badges. Each season expands/collapses, shows `N / target episodes` count, and exposes inline edit (number / title / logline / synopsis / ending hook / status / episode-count target). Episodes render as compact rows with delete, hover-revealed season picker (move between seasons), and direct deep-link into the issue editor. `Generate episodes (LLM)` on an empty season runs the per-season-episodes pass and creates one issue per beat with `seasonId` + `arcPosition` pre-filled; un-grouped issues stay visible in a dedicated bottom panel. Replaces the flat issue-card grid from Phase 1; backed by the schema + endpoints from Phases 2 + 3. Two-click-arm replace guard on arc regeneration and per-season delete. Phase 4 of the Story Arc Planning initiative.
- **Pipeline — LLM-driven series-arc planning.** Three new staged-LLM passes that plan and verify a multi-season series: (1) `POST /api/pipeline/series/:id/arc/generate` proposes the top-level arc — series-level logline, summary, themes, protagonist arc — plus a 2–5 season breakdown with per-season loglines, ending hooks, and episode-count targets, default `model: 'heavy'`. (2) `POST /api/pipeline/series/:id/seasons/:seasonId/episodes/generate` proposes the episode-by-episode breakdown for one season, taking the whole-arc protagonist trajectory and prior seasons' synopses as continuity context. (3) `POST /api/pipeline/series/:id/arc/verify` runs a cross-season continuity pass that surfaces character contradictions, dropped subplots, episode-count/arc-weight mismatch, unresolved finale hooks, arc-role imbalance, and theme drift as actionable `{ severity, location, problem, suggestion }` issues. All three calls accept `{ providerOverride, modelOverride }` and routes 1 + 2 also accept `commit: true` to persist the proposal in one shot (default returns a preview the UI can confirm before writing). Backed by `server/services/pipeline/arcPlanner.js` (~280 LOC) on top of the shared `stageRunner.js` + new stage prompts in `data.sample/prompts/stages/pipeline-arc-{overview,verify}.md` + `pipeline-season-episodes.md`, all three reusing the `_partials/bible-deference.md` block for character/setting consistency. Phase 3 of the Story Arc Planning initiative — no UI yet (Phase 4 builds the arc canvas on top of these endpoints).
- **Pipeline — series-arc + season schema.** Series records gain two new optional fields: `series.arc` (logline, summary, themes, protagonistArc, status) for the overall multi-season story spine, and `series.seasons[]` for the ordered season/volume list. Issues gain optional `seasonId` + `arcPosition` pointers so an issue can be slotted into a specific season at a specific ordinal. Backed by a new `server/lib/storyArc.js` (canonical shapes + sanitizers, sibling to `storyBible.js`) and a new `server/services/pipeline/seasons.js` service that handles delete-with-reassign so removing a season cleanly re-points or un-groups child issues. Routes: `GET / POST /series/:id/seasons`, `PATCH / DELETE /series/:id/seasons/:seasonId`. Pure additive — existing series files round-trip unchanged. Phase 2 of the Story Arc Planning initiative; no UI yet (Phase 4 builds the arc canvas on top of this).

- **Prompt template partials** — PortOS prompts now support Mustache-style `{{> partial-name }}` includes. Files in `data/prompts/_partials/` get inlined into stage templates at render time (recursive, with cycle detection + loud missing-partial errors). First two partials cover the duplicated bible-deference preamble and the scenes-JSON output contract that were copy-pasted across `writers-room-script.md` and `pipeline-extract-scenes.md`. Both prompts now reference the shared partials — future tweaks to "defer to canonical character names" or "this is the scenes[] JSON shape" land in one place. Step 7 of the Writers Room ↔ Pipeline DRY unification.
- **Writers Room ↔ Pipeline bridge.** A "Promote to pipeline" action on a Writers Room work creates a Pipeline series + first issue in one call, carrying over the active draft prose, all three bibles (characters/settings/objects), and the latest script-analysis scenes as fully-populated storyboards. The two systems record each other's ids so the menu flips to "Open in pipeline" on subsequent visits, and the Pipeline series page surfaces a "Writers Room" badge linking back. Idempotent — re-promoting a linked work returns the existing pair instead of duplicating. Backed by `server/services/writersRoom/promoteToPipeline.js` and `POST /api/writers-room/works/:id/promote-to-pipeline`. Step 6 of the Writers Room ↔ Pipeline DRY unification.
- **Pipeline page** under Create — a series-production workflow that chains a rough idea seed through prose → comic-book script + TV teleplay (parallel) → comic pages + storyboards (image gen) → episode video. Each Series carries a shared bible (logline, premise, characters, optional World Builder ref, style notes) that every issue/episode inherits into its stage prompts so identity stays consistent across the run. The "Auto-run text" button kicks off the LLM chain end-to-end with SSE progress; the visual stages (comic pages, storyboards) hand off to the existing image-gen queue.
- **Pipeline episode video** now wired end-to-end to Creative Director. The Episode Video stage on a Pipeline issue creates a CD project from the storyboards scenes (series style notes auto-prepended, i2v continuation from scene 2 onward, audio disabled, scenes auto-accepted — no LLM evaluator round-trip since the human already vetted storyboards), persists the `cdProjectId` on the stage, polls render progress, and renders the final stitched mp4 inline when complete. A second "Run everything (incl. video)" button on the issue page extends auto-run past the text stages — text chain → episode-video kickoff in one click.
- Codex image gen: hi-res resolution presets (1024×1536, 1536×1024, 1536×1536) — gpt-image-2's native hi-res tier. Picking any ≥1536 size automatically appends a `(high quality)` hint to the codex prompt so the model renders at full fidelity instead of its 1024 default.
- Image Gen batch mode: a `× N` count input next to the Generate button (async modes only — local + codex) queues N renders of the same prompt+resolution in a single submit. The first job streams its progress live; the remaining N-1 land in the server queue and surface as `+N queued`. Each job gets a fresh random seed by default so the batch produces variations, not duplicates.
- Media Lightbox full-screen mode (`F` key or top-right toggle): drops the modal's rounding and padding and lets the image fill the viewport. Swipe left/right (or arrow keys) navigates between gallery items; tap reveals the settings as a slide-in drawer. Layered Escape closes drawer → exits full-screen → closes the lightbox. The Seed row in the settings pane is now copyable for easy reuse in a remix.
- Media Lightbox surfaces a `Codex session` row for gpt-image-2 renders (the session-id from the codex CLI banner, written to the image's sidecar metadata). Closest analogue to a seed for codex output — copyable for traceability. The Seed row now reads `n/a (gpt-image-2)` for codex images instead of hiding, so it's clear why no seed is present.

## Changed

- **MediaCard thumbnails.** Removed the "IMAGE"/"VIDEO" badge overlay from the bottom-right of media cards — the icon and aspect already convey the kind, and the badge added visual noise on grid views.
- **Pipeline Series page — two-pane layout.** PipelineSeries (`/pipeline/series/:id`) drops its `max-w-5xl` cap and splits into a sticky left bible sidebar + right card-grid canvas at `lg+`. The bible inputs (name / format / logline / target issue count / premise / style notes / world / characters) live in the sidebar; issues/episodes render as a responsive card grid (1/2/3/4 cols across breakpoints) showing number, status badge, title (line-clamped), and `updated <timeAgo>`. Sidebar collapses to a 48px rail via a panel toggle (state persists in `localStorage` under `portos-pipeline-series-sidebar-collapsed`) so the canvas can use the full viewport when you're deep in episode work. Mobile (< lg) stays single-column with the sidebar reflowing above the canvas. Phase 1 of the Pipeline Story-Arc Planning initiative — pure UI refactor, no schema or API change; subsequent phases evolve the card grid into an Arc → Season → Episode tree.

- LTX-2.3 video model dropdown now groups the legacy notapalindrome runtimes (`ltx2_unified`, `ltx23_unified`, `ltx23_distilled_q4`) under a `Legacy` `<optgroup>`, with the dgrauet `ltx23_dgrauet_*` runtimes surfaced as the active picks at the top of the list. `defaultMacos` deliberately stays on `ltx23_distilled_q4` because the dgrauet runtime requires an opt-in venv install — fresh installs need a model that works out of the box. Applies to both Video Gen and Creative Director dropdowns. New Video Gen FFLF mode also gets a one-line keyframe-pair guidance hint ("use keyframes that share scene geometry — same camera, same subject; unrelated images produce a visual cut") to prevent the "looks like two stills" complaint.
- Writers Room bible CRUD (characters / settings / objects, one file per work) now collapses onto a single `createBibleStore(...)` factory in `server/lib/storyBible.js` — each domain file shrinks to ~30 LOC of factory configuration. -281 LOC across the three domain files. Pipeline bible routes share the writers-room zod schemas under generic names (`characterBibleCreateSchema`, etc.) instead of the prior `z.record(z.string(), z.any())` loose-record validation; the canonical `sanitizeBibleList` in `storyBible.js` remains the single source-of-truth for shape enforcement on both sides. Steps 2a + 2b of the Writers Room ↔ Pipeline DRY unification.
- Pipeline stage components (`ProseStage`, `TextStagePanel`, `EpisodeVideoStage`) now share a `useAsyncAction(fn)` hook for the common "setLoading(true) → await → catch+toast → setLoading(false)" pattern. `StoryboardsStage` deliberately stays on its existing pattern because its `savingIdx`/`extractingFrom` flags are keyed loading states, not booleans. Step 4c of the Writers Room ↔ Pipeline DRY unification.

- Pipeline `POST /api/pipeline/series/:id/extract-bible` accepts a `parallel: true` flag that fans out the three bible kinds concurrently (~3× wall-clock speedup on HTTP-API providers like OpenAI / Anthropic / LM Studio HTTP). Default stays sequential so CLI-only providers (codex, claude-code, gemini-cli) — which serialize at the provider session anyway — aren't penalized.
- Creative Director `GET /api/creative-director/:id` accepts a `?slim=1` query param that drops the unbounded `runs[]` history and the heavy treatment fields, returning only what a polling UI needs (`status`, `updatedAt`, per-scene `{ id, order, status }`, `finalVideoId`, `failureReason`). Pipeline EpisodeVideoStage's 4s poll now uses slim mode — keeps the UI responsive without pulling the entire render history on every tick.
- Pipeline Storyboards stage gains "From TV script" + "From prose" auto-fill buttons. Once the upstream stage has output, one click runs the new shared `extractScenes()` LLM pass (sluglines parsed from the teleplay, or paragraph-grain beats lifted from the prose) and replaces `stages.storyboards.scenes[]` with the result. A two-click-arm replace guard surfaces a warning toast before clobbering hand-curated scenes. Backed by `POST /api/pipeline/issues/:id/stages/storyboards/extract-scenes` and the new `server/lib/sceneExtractor.js`, which both Pipeline AND the Writers Room evaluator's `script` analysis now route through — deleting ~25 LOC of duplicated SHAPERS.script logic and giving the Pipeline a one-click path from "TV script ready" to "all storyboards primed for image gen." Step 5 of the Writers Room ↔ Pipeline DRY unification. ## Fixed

- **DRY pointer cursor on interactive elements.** Tailwind v4 dropped the default `cursor: pointer` on `<button>`, which meant every clickable in PortOS had to spell out `className="cursor-pointer"` — and most didn't, so buttons hover-rendered with the text I-beam. A single global rule in `client/src/index.css` now restores pointer for `button`, `a[href]`, `summary`, `select`, clickable input types (checkbox/radio/submit/button/reset/file/range/color), `<label>` wrapping a checkbox/radio, and ARIA roles (`button`, `link`, `tab`, `menuitem`, `option`, `checkbox`, `radio`, `switch`). Placed before the existing `[disabled] { cursor: not-allowed }` block so disabled elements still get the correct override. Existing `cursor-pointer` utility classes are now redundant but harmless.
- Pipeline issues no longer get stuck in `status: 'running'` after a server restart mid auto-run. Boot now walks every issue and demotes any `running` to `needs-review` (the same terminal state a normal completion lands on) — the UI shows the issue as needing attention instead of spinning forever waiting for SSE that will never arrive.
- Pipeline EpisodeVideoStage final-video render no longer breaks when the underlying mp4 is missing — both surfaces (Creative Director SegmentsTab + Pipeline EpisodeVideoStage) now route through a shared `ScenePreview` component with the onError-missing-media fallback + Retry + cache-bust pattern.
- Render Queue UI now lists jobs in execution order (running first, then queue position ascending) instead of mixing a long-running job into the middle of newer queued batches. The route was sorting all jobs by `startedAt || queuedAt` DESC, which slotted an active job below a fresher batch.
- CoS agent dedup race in `spawnAgentForTask` — under sustained scheduler tick load, two parallel `task:ready` emissions for the same task id could both pass the dedup-has-check because `spawningTasks.delete(task.id)` ran BEFORE the agent was queued to the runner. A second spawn would slip through the window and produce a duplicate agent. The delete now lives inside a `try { spawnViaRunner|spawnDirectly } finally { spawningTasks.delete }` block so the guard outlives the spawn call. New behavioral race-repro test in `agentLifecycle.test.js` proves the contract.
