# Unreleased

## Apps

- **The per-app Jira Kanban board now shows your full workflow lifecycle, not just To Do / In Progress / Done.** The board used to collapse every ticket into Jira's three built-in *status categories*, so distinct workflow stages like **Blocked** and **In Review** (both "In Progress" to Jira) were invisible — they all piled into one column. The board now resolves the project's real workflow stages and renders one column per stage. When the app's Jira config has a `boardId`, PortOS reads that agile board's actual column layout from Jira (`/rest/agile/1.0/board/{id}/configuration`), so the columns and their order mirror exactly what you see on the board in Jira — including custom stages. Without a board id it falls back to the project's distinct statuses ordered by category, and if neither can be read it falls back to the original three-category board. Columns are color-cued (Blocked → red, Review/QA → purple, otherwise by category), and drag-to-transition now targets the exact status of the column you drop into (any ticket whose status maps to no column gets its own column so nothing disappears). The board now also spans the **full width** of the app Overview tab (the surrounding details stay capped for readability) so all lifecycle columns are visible across the desktop instead of being clipped at ~1024px, falling back to horizontal scroll only when there are more stages than fit. (`server/services/jira.js`, `server/routes/jira.js`, `client/src/components/KanbanBoard.jsx`, `client/src/components/apps/tabs/OverviewTab.jsx`)

## Chief of Staff

- **Feature Agents moved from Dev Tools to the Chief of Staff section.** The Feature Agents page (`/feature-agents`) now lives under the Chief of Staff sidebar group alongside the other agent-management pages, rather than buried in Dev Tools. The route is unchanged — only the sidebar grouping and the ⌘K / voice nav section label moved. (`client/src/components/Layout.jsx`, `server/lib/navManifest.js`)

- **The CoS → Schedule page is now a scannable card grid instead of a flat expandable list.** Each task type renders as a status card you can read at a glance — a colored status dot (active / on-demand / waiting-on-deps / disabled), a **next-run countdown** ("in 3h", "Mon 9:00") that was previously computed server-side but never shown, an app-coverage bar with the enabled/total count kept front-and-center, and last-run/run-count. Cards sort by status then soonest next run (so what's about to run rises to the top), the All/Enabled toggle is replaced by per-status filters (Active / On-Demand / Waiting / Disabled), and a name filter box jumps to a task quickly. A **Run** button sits on each card — for app-targeting tasks it opens an app picker so the on-demand run carries app context (just like the old per-task controls), and falls back to a plain global run for tasks with no managed apps; **Configure** opens a slide-over drawer with the full per-task settings (interval, provider/model, prompt, agent options, dependencies, pipeline stages, and per-app overrides) — the same controls as before, now in a focused panel that's deep-linkable via `?task=` and closes on Esc/backdrop. Mobile collapses the grid to a single column.

- **[issue-1563] Every Chief-of-Staff agent now records which machine produced it.** As a first step toward federating CoS agent history across peers, each spawned agent stamps the producing instance's federation identity (`instanceId`) into its metadata at registration. The stamp flows through to the agent's archived `metadata.json` automatically, so once agent history syncs between a node pair, each agent — and the worktree branch it created — can be attributed to the instance that ran it. This is the provenance prerequisite for the claim/lease ownership work that lets two federated peers share a backlog without ever duplicating each other's agent runs. (`server/services/agentLifecycle.js`)

- **[issue-1563] Two federated machines can now share a Chief-of-Staff backlog without ever running the same task twice.** Each task a CoS agent starts is now stamped with a time-bounded ownership *claim* — which machine took it, when, and a lease that stays alive while the agent works. Before starting any task, a machine checks the claim: if its peer holds a live lease, it backs off instead of spawning a second agent (which previously would have created conflicting worktrees and branches on the same repo and raced the recovery sweep). The lease auto-renews while the work runs and is released the moment the task finishes or fails back to the queue, and orphan recovery leaves a peer's in-flight task alone rather than resetting it. If a machine crashes mid-task its lease simply expires and the peer is free to pick the task up. The worktree each agent creates is also tagged with its origin machine. This is the safety groundwork that lets CoS be included in full peer mirroring (#1561). (`server/services/cosTaskClaim.js`, `server/services/agentLifecycle.js`, `server/services/cos.js`, `server/services/cosTaskStore.js`, `server/services/worktreeManager.js`)

## Local LLMs

- **The Hugging Face model search now lets you pick a quantization, and defaults to the highest-fidelity build that fits your machine's memory.** Previously every GGUF repo installed a single auto-chosen Q4-ish quant, hiding larger variants (like a `…-BF16.gguf`) that a big-memory box can comfortably run. Each Hugging Face result now carries a **Quant** menu listing every GGUF variant in the repo (multi-part shards summed into one entry), with a per-quant fit hint — *fits comfortably* / *tight fit* / *exceeds RAM* — measured against your usable memory. The default selection is RAM-aware: PortOS reserves headroom for the OS and KV cache, then picks the largest variant that still fits (so a 128 GB unified-memory Mac defaults to BF16/Q8 where a 16 GB laptop stays on Q4), and you can override it per model. This is purely a file-picker improvement — these are all still GGUF files the existing Ollama/LM Studio runtimes already support; nothing about the GGUF-only requirement changed (raw safetensors/full-precision HF models need a different runtime and remain out of scope). Ollama installs via `hf.co/<repo>:<quant>`; LM Studio via `<repo>@<quant>`.
- **[issue-1537] The Hugging Face search now surfaces Apple MLX models on Apple Silicon, not just GGUF.** On an Apple-Silicon Mac with LM Studio, search results now include MLX builds (Apple's native format — often the fastest path on unified-memory Macs) alongside GGUF, each tagged with an **MLX**/**GGUF** badge so you know what you're installing. MLX cards carry the same size + RAM-fit hint as GGUF cards and install through LM Studio (`lms get <repo>`). MLX is shown only where it actually runs: it's hidden on non-Apple hosts, and for the Ollama backend (Ollama's MLX acceleration applies to its own registry's GGUF models automatically and can't pull arbitrary Hugging Face safetensors repos).
- **The built-in "Recommended" model cards now have the same RAM-aware quant picker as the Hugging Face search.** Recommended models used to hard-code one quant each; cards backed by a Hugging Face repo now show the **Quant** menu and default to the highest-fidelity build that fits this machine's memory, so the recommendation adapts to your hardware instead of being fixed.
- **[issue-1539] Recommended models that pull from Ollama's own registry by bare name now get the RAM-aware quant picker too.** Previously only Hugging Face-backed recommended cards offered a **Quant** menu; cards for Ollama registry models (Llama 3.2, Qwen2.5, GPT-OSS, Llama 3.3, …) showed a single hard-coded build. PortOS now discovers each of those models' available quantizations from the Ollama registry and presents them in the same picker, with per-quant fit hints and the highest-fidelity build that fits your memory selected by default — so every recommended card adapts to your hardware, regardless of which registry the model comes from. When a model can't be reached on the registry, its card keeps its single curated build as before.
- **Hugging Face search results now rank by downloads and recency more strongly.** A model's download count and how recently it was updated now dominate the result ordering (recency is graduated — a model updated this week outranks an equivalent one from months or years ago, and stale models are demoted), so popular, actively-maintained models surface first instead of being outweighed by publisher/format bonuses.
- **[issue-1488] Pinning a small-context local model to a heavy analysis stage no longer overflows its window.** The reverse-outline, continuity-bible, comic-script verification, alternate-POV rewrite, and per-issue editorial-analysis stages used to size the manuscript they send to a fixed 48–60K-character floor regardless of the model — fine on a big-context provider, but on a genuinely small-window local model (e.g. an 8K-context model) that floor plus the prompt could exceed the window. These stages now scale the manuscript to the model's actual window, trimming it to fit a small one (never down to empty — a usable minimum is always kept) while still feeding the whole manuscript to a large-context model exactly as before.
- **Installing Ollama (or LM Studio) via Homebrew no longer reports a false "install failed."** `brew install` routinely exits non-zero *after* a successful pour — a failed `brew cleanup`, a dependency's post-install caveats, or env-hint noise (notably Ollama's new `mlx`/`mlx-c` dependencies) all bubble up as exit 1 even though the formula/cask is fully installed on disk. PortOS treated that exit code as a hard failure and told you the install didn't work, when in fact it did. The install path now verifies presence-on-disk via `brew list` after a non-zero exit and treats the package being installed as the source of truth — so a successful-but-noisy Homebrew install is reported as success and proceeds to start the backend, instead of surfacing a misleading error. (`server/services/localLlm.js`)
- **Clicking "Run at Startup" for Ollama no longer fails with `Bootstrap failed: 5: Input/output error`.** Registering Ollama as a login service runs `brew services start ollama`, which shells out to `launchctl bootstrap` — and that fails with launchd error 5 (EIO) when a stale service registration is already loaded (left over from a prior `ollama serve`, an interrupted start, or a Homebrew upgrade). PortOS now detects that specific bootstrap conflict, boots out the stale registration (`brew services stop`), and retries the start once — so the toggle succeeds instead of surfacing the raw launchctl error. Unrelated start failures (e.g. permission errors) are not retried. (`server/services/ollamaManager.js`)

## Comic pages

- **Comic pages now carry scene markers, and page renders chain within a scene but start fresh across a scene cut.** Scene boundaries are established once at the beat-sheet stage as a canonical `## Scenes` list (numbered `INT./EXT. LOCATION — TIME` sluglines) and flow down through the prose draft into the comic script, which now tags each page header `## Page N — Scene M: SLUGLINE`. The Comic tab shows the scene on every page, and the parser records `sceneNumber`/`sceneHeading` per page. The payoff is in rendering: the consistency-reference picker now defaults to **auto**, which uses the *previous* page as an image reference only when that page is in the **same scene** and already rendered — so incidental, un-described characters and the environment stay consistent through a continuing scene, while the first page of a new scene renders clean instead of dragging the prior scene's look across the cut. You can still override per page: **none** forces a fresh render mid-scene, or pick **prev/next page** to force a specific reference (e.g. to fix a character-consistency drift across a scene boundary). Pages with no scene markers (legacy comics) behave exactly as before — no auto-chaining. (Series Autopilot's page renders inherit the same within-scene continuity automatically.)
- **[issue-1534] Refine a rendered comic page with a small fix instead of re-rendering it from scratch.** Each rendered page now has a **Refine** box: type a free-text adjustment ("warm the lighting", "remove the extra signage text", "make the two background figures match across panels") and the page re-renders image-to-image from its *existing* image, applying only that change while preserving everything that was already right. The AI tweaks the page's current render prompt to reflect your instruction (it does not regenerate the prompt from the comic script), then runs a low-denoise re-render off the page's own image — so the panel layout, composition, and lettering survive. It refines the page's final render when one exists, otherwise its proof, and shows a short "what changed" summary after each refine. Works with both the local and Codex image backends.

## Storyboards

- **[issue-1552] Storyboard scenes now line up exactly with comic-page scenes.** The beat sheet's canonical `## Scenes` list is now the single source of truth for storyboard scene numbers and sluglines too. When you click "Generate scenes from prose/teleplay" on an issue whose beat sheet carries a `## Scenes` list, PortOS still extracts the rich per-scene content (dialogue, visual prompts, shots) from the prose/teleplay — so downstream audio voice-over and image/video generation keep everything they need — but overlays the canonical scene numbers and `INT./EXT.` sluglines on top, so "Scene 2" is the same Scene 2 across the beat sheet, prose, teleplay, comic script, comic pages, and storyboards. (For an issue with a scene list but no prose/teleplay yet, the scenes come straight from the list.) Issues with no canonical scene list fall back to the existing model-driven extraction exactly as before. The TV teleplay stage also now reuses the canonical scene numbers and sluglines verbatim when the source material carries them, so teleplays stay aligned with every other adaptation.

## ChatGPT import

- **[issue-1527] A failed ChatGPT export import no longer leaves a stray temp file behind under heavy disk load.** When an import was aborted mid-stream (e.g. a corrupt asset-name map or an oversized member), the cleanup that removes the half-written `.part` temp file could run *before* the asset still streaming to that file finished closing — so under heavy concurrent disk I/O the write could re-create the `.part` file just after it was deleted and orphan it. Cleanup now waits for every in-flight asset write to close before removing temp files, so a rejected import reliably leaves nothing behind.

## Apple Health import

- **[issue-1532] Apple Health exports that include clinical records now import instead of crashing.** Importing a Health export ZIP that contained a `clinical_records/` folder (the FHIR records you get after linking a healthcare provider) failed outright — the import aborted while reading those JSON files. Such exports now import correctly, with their clinical records loaded alongside the rest of your health data; exports without clinical records are unaffected.

## Per-issue story generation

- **[issue-1511] Per-issue prose and comic-script generation now sees only the characters that issue involves.** Generating an issue used to load the entire series character bible into every prose and comic-script prompt — on a large-cast series that's tens of thousands of tokens of character detail (including one-off bit players from unrelated issues) re-sent on every issue and every stage. Generation now sends full character records only for the cast named in that issue's beats, synopsis, and source material, with a compact one-line roster of everyone else kept for continuity. This sharply cuts token cost on big-cast series and, on context-bounded providers, leaves more room for the actual draft. The series' lead/recurring characters stay in context for every issue, so the core cast is never dropped — generation just stops carrying the full record of every bit player from unrelated issues.
- **[issue-1529] A character whose name is a common word (Will, May, Grace) is no longer pulled into a prompt by an incidental word.** When scoping an issue's character bible, an ordinary phrase like "the team **will** regroup" could be mistaken for naming a cast member called "Will" — incidentally scoping the prompt down to that one character. A single-word character name now only counts as named when it appears capitalized as a proper noun ("Will entered"), so common-word names match the same way every other name does. Multi-word names and nicknames are unaffected.

## Editorial checks

- **[issue-1581] A new editorial check catches timeline and canon contradictions — a dead character who reappears alive, an age that contradicts the bible, an impossible chronology.** PortOS had no editorial check for internal contradictions against canon and the story's own timeline, so a character killed in issue 3 who walks back in alive in issue 6, an age the prose contradicts (the bible says 16, the page says "in her 30s"), or a journey that takes eight days but is dated to day two could slip through to the reader — the kind of mistake that's nearly impossible to catch by eye once an arc spans months of drafting. The new **Timeline / canon contradiction** check (`continuity.timeline-contradiction`) reconciles the stitched manuscript against the **continuity-bible facts ledger** (the ground-truth ages, dates & elapsed time, locations, and world rules already extracted by the continuity bible), the canon character facts (age, role, status, description), the reverse-outline scene ordering (for chronology), and the authored per-character arcs — flagging each contradiction with the two facts that conflict and a verbatim anchor. It runs across the whole manuscript — carrying each character's last-known state (alive/dead, age, location) forward between chunks so a death-then-resurrection that spans chapters is actually visible — and degrades to a prose-only scan when no canon or outline exists. It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking plot-breaking contradictions high-severity per finding. Its findings go stale (and re-run) when any source it reads drifts — including the continuity-bible ledger — via the new `continuityBible` editorial source. (`server/lib/editorial/checkRegistry.js`, `server/services/pipeline/editorial/checkRunner.js`, `data.reference/prompts/stages/pipeline-editorial-timeline-contradiction.md`, `scripts/migrations/129-editorial-timeline-contradiction-stage.js`)

- **[issue-1582] A new editorial check catches unearned personality shifts — a reserved character who suddenly cracks jokes, a stated fear silently ignored, knowledge that appears with no on-page learning.** PortOS's `arc.transitions` check finds *change moments* but nothing flagged a change the story never *earns*: a guarded character trading easy banter a chapter later with no beat to motivate it, an established trait (a fear, an allergy, a skill) quietly contradicted, or a POV character who acts on information they were never shown learning. These read as author slips rather than growth and are hard to catch by eye once an arc spans months of drafting. The new **Character consistency** check (`character.consistency`) reconciles the stitched manuscript against the established canon character *traits* (personality, fixed traits, mannerisms, likes/dislikes, speech), the reverse-outline scene ordering (to spot a knowledge jump within a scene), and — crucially — the **authored per-character arcs**, so an intentional, earned transition the author recorded is *not* flagged. It runs across the whole manuscript, carrying each character's established temperament forward between chunks so a baseline set in an early chapter and an unearned shift in a later one are visible together, and degrades to a prose-only scan when no canon or outline exists. It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking trait contradictions that break a plot beat high-severity per finding. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-character-consistency.md`, `scripts/migrations/130-editorial-character-consistency-stage.js`)

- **[issue-1583] A new editorial check catches a weak climax — a protagonist who is rescued at the payoff scene, or a climax that wins the plot but never lands the story's emotional/thematic core.** Nothing checked that the climax is the protagonist's *hardest, most active* moment: a finale where an ally swoops in to resolve the conflict, the antagonist conveniently self-destructs, or a coincidence settles things — or a climax that lands the action but leaves the theme the story has been building unanswered — could ship unflagged, even though a passive climax is story-breaking for the reader who invested in the whole arc. The new **Climax / resolution power** check (`arc.climax-agency`) identifies the climax scene and judges two things: *agency* (does the protagonist drive the resolution through a hard, costly choice, vs. being a spectator at their own climax) and *resolution power* (does the climax resolve the core problem and land the emotional/thematic argument, not just the surface plot). It reconciles the prose against the authored reader-map payoffs (what the reader was promised) and the declared themes, using the reverse-outline scene map to locate the climax, and gates its verdict on the final manuscript part — the climax can only be judged once the whole story is in view — degrading to a whole-manuscript scan when no reader-map, themes, or outline exists. It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking a fully passive climax high-severity; it complements `plot.structure-momentum` (which flags a passive protagonist arc-wide) by focusing the lens on the single payoff scene. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-climax-agency.md`, `scripts/migrations/131-editorial-climax-agency-stage.js`)

- **[issue-1584] A new editorial check catches emotional beats that don't match the stakes — a trauma the story never processes, or a minor setback that triggers wildly outsized grief.** Nothing checked that reactions track event magnitude: a character could lose their brother in issue 2, the prose moves on, and the wound is never touched again — or a small disappointment could trigger a chapter of despair the story hasn't earned — and either way the emotional authenticity collapses without being flagged. The new **Emotional beat proportionality** check (`emotion.reaction-proportionality`) reads the stitched manuscript with the reverse-outline scene map to weigh each event's magnitude and flags two mismatches: an *under-reaction* (a high-magnitude event — death, trauma, betrayal, major loss or win — that draws no on-page reaction and is never processed in later issues) and an *over-reaction* (a minor setback triggering grief, rage, or despair out of all proportion). Because an unprocessed event in an early issue can stay unaddressed many issues later, the check carries every high-magnitude event still awaiting a proportionate reaction forward between manuscript chunks, so a trauma introduced early and then ignored is flagged against the later issue that should have paid it off; it degrades to a whole-manuscript scan when no outline exists, and respects characterization the story has established (a deliberately stoic or numb character isn't flagged for measured restraint). It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking a major trauma left wholly unprocessed high-severity. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-reaction-proportionality.md`, `scripts/migrations/132-editorial-reaction-proportionality-stage.js`)

- **[issue-1585] A new editorial check catches flat secondary characters — a recurring side character who appears scene after scene but is exactly the same at the end as at the start.** The existing `pov.justified` check asks whether each *POV* character earns their viewpoint, but nothing judged the *non-POV* cast — so a recurring side character (a partner, a foil, a co-worker who shows up across half the book) could stay completely static, draining the story's texture, without ever being flagged. The new **Secondary-character arcs** check (`character.secondary-arc`) tallies the recurring non-POV cast from the reverse-outline scene map — characters present across multiple scenes who never hold the viewpoint — and judges whether each shows meaningful change, flagging a *flat arc* (the same situation, attitude, and standing at the end as the start) or a *purposeless regression* (an arc that quietly undoes itself with nothing in the story making it meaningful). It reads the stitched manuscript with the recurring-cast roster and the canon roster, deliberately ignoring genuine walk-ons (a one- or two-scene bit player owes the reader no arc) and a deliberately-static foil whose constancy is the point. Because a flat arc is a whole-story claim, it carries each recurring secondary's established state forward between manuscript chunks and gates its verdict on the final part — so a character established early whose change lands in a chapter no longer in view isn't falsely flagged — and degrades to a whole-manuscript scan when no outline exists. A configurable **recurring threshold** (default 2 scenes) tunes how prominent a character must be before it's held to an arc. It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking a prominent recurring character left wholly flat medium-severity. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-secondary-arc.md`, `scripts/migrations/133-editorial-secondary-arc-stage.js`)

- **[issue-1587] A new editorial check catches dialogue-tag tics — "she said" eight times in one scene, or a different fancy verb on nearly every line.** The existing `dialogue.said-bookisms` check flags ornate tags ("expostulated", "opined"), but nothing caught the opposite problems: within-scene **monotony** (one tag verb hammered over and over until it starts to call attention to itself) or **over-variation** (a fresh tag verb on almost every line — "said/asked/replied/murmured/whispered" thesaurus churn that's just as distracting). The new **Dialogue tag variety** check (`dialogue.tag-variety`) inventories the speech tags (plain and ornate) adjacent to quoted lines, scene by scene — splitting a manuscript on its scene dividers (`***`, centered rules, markdown scene headings) so the count isn't diluted across a whole issue — and collapses inflections onto a base verb so "said"/"says" count as one. It flags a scene where a single verb dominates a tag-dense exchange, or where nearly every tagged line uses a different verb, and leaves a healthily varied, mostly-invisible-"said" scene alone. It's a default-on, deterministic series check (no LLM call) with per-series thresholds for how many tags a scene needs before it's judged, how dominant a verb must be to read as monotonous, and how varied to read as churn — and a house-style allowlist to mute a tag verb. Findings surface in the existing editorial-review UI and Series Autopilot like every other check, at low severity. (`server/lib/editorial/dialogue.js`, `server/lib/editorial/checkRegistry.js`)

- **[issue-1586] A new editorial check catches narrator-voice whiplash — narration that's witty in issue 1, grim in issue 3, then witty again in issue 5.** The existing `dialogue.voice-distinctiveness` check guards each *character's* dialogue voice, but nothing watched the *narrator's* voice across a series — so the storytelling tone could lurch from issue to issue (a wry, ironic narrator turning flat and clinical, a spare hard-boiled voice going lush and lyrical) with no one flagging the inconsistency, even though voice consistency is part of the promise to the reader. The new **Narrative voice / tone consistency** check (`style.voice-consistency`) fingerprints each issue's narrative tone — diction, register, humor, emotional temperature, sentence rhythm — and flags two problems: a *cross-issue tonal shift* the story never earns, and *drift from the series style guide's intended voice* (the declared `tone` words) when one is set. It reads the stitched manuscript with the style guide's intended tone, deliberately ignoring a purposeful modulation the events justify (a grim chapter because the plot took a grim turn) and judging the narration only, not character dialogue. Because the comparison spans issues, it carries each issue's tone fingerprint forward between manuscript chunks, so a later issue's narration is judged against the voice the series established earlier; it degrades to a pure cross-issue consistency scan when no style guide tone is declared. It's a default-on series check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking a sharp, unexplained whiplash medium-severity. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-voice-consistency.md`, `scripts/migrations/134-editorial-voice-consistency-stage.js`)

- **[issue-1588] A new opt-in editorial check catches contradictions to real-world facts — Paris placed in Germany, an anachronistic technology, a physiologically impossible feat.** Distinct from the internal timeline/canon check (#1581), nothing reconciled grounded fiction against *external* truth, so a checkable howler in a historical or hard-SF series — a city in the wrong country, an event dated before the war it belongs to, a survivable injury that isn't — could ship and quietly torch the story's credibility. The new **Research / fact accuracy** check (`research.fact-accuracy`) scans the stitched manuscript against an **author-supplied fact reference** and flags geography, history/chronology, and science/physiology claims that contradict it. It is deliberately **opt-in and gated**: it runs only when a series is marked **fact-critical** *and* a non-empty fact reference is supplied, so it never second-guesses deliberate invention in pure fantasy. A new **Fact accuracy** section in the series bible (Pipeline → series) carries the toggle + reference textarea; the reference is a fingerprinted editorial source (`series.factReference`), so editing it re-runs the check, and findings surface in the existing editorial-review UI and Series Autopilot under a new **Fact accuracy** category. Both new series fields (`factCritical`, `factReference`) are wire-gated (pipelineSeries schema v7) so an older federated peer can't strip-then-LWW the loss. (`server/lib/editorial/checkRegistry.js`, `server/services/pipeline/series.js`, `client/src/pages/PipelineSeries.jsx`, `data.reference/prompts/stages/pipeline-editorial-fact-accuracy.md`, `scripts/migrations/135-editorial-fact-accuracy-stage.js`)

- **[issue-1589] A new editorial check catches comic↔prose drift in hybrid issues — a plot beat the prose narrates that no panel shows, panel dialogue that contradicts the prose, or events ordered differently across the two media.** The existing `comic.*` checks validate comic content *in isolation* (lettering load, balloon attribution, panel rhythm), but for an issue told as **both prose and comic** nothing checked that the two media tell the *same* story — so a character the prose stabs could stand unharmed in the panels, a line could be put in a different character's mouth on the page, or the comic could reorder events the prose places in a fixed sequence, all unflagged. The new **Comic ↔ prose synchronization** check (`comic.prose-sync`) pairs each hybrid issue's prose narration with its authoritative comic pages (each panel's visual description, dialogue, captions, and SFX) and flags three substantive divergences: an **unshown beat** (a consequential event the prose narrates that no panel depicts), **contradicted dialogue** (a line said with materially different meaning or attributed to a different speaker), and a **chronology disagreement** (the two media ordering the same events incompatibly). It is deliberately tuned to *ignore* ordinary medium translation — comics legitimately compress, cut, reorder for page rhythm, and externalize interior narration into image — so a faithful adaptation that simply differs in density isn't flagged. It runs one model call per issue that has both prose and comic content (capped per run), reading each issue's **`prose`-stage** text (specifically, not the stitched manuscript — which prefers the comic script for a hybrid issue) plus the `comicScript.pacing` comic content, and anchors every finding to its issue; its findings go stale and re-run when either the prose or the comic for that issue drifts. It's a default-on, issue-scope check that surfaces in the existing editorial-review UI and Series Autopilot like every other check, marking a flat contradiction or a dropped consequential beat medium-severity. (`server/lib/editorial/checkRegistry.js`, `data.reference/prompts/stages/pipeline-editorial-comic-prose-sync.md`, `scripts/migrations/136-editorial-comic-prose-sync-stage.js`)

- **[issue-1590] The telling-not-showing emotion check now catches the whole show-vs-tell family, not just named emotions.** `prose.telling-emotion` previously flagged only outright emotion labels ("she was sad", "he felt nervous"), missing three sibling forms of telling that weaken prose the same way: **agency-removing constructions** where the situation narrates itself with no one acting ("the story unfolded", "tension filled the room", "the argument escalated"), **reported-sensation filtering** where a feeling is routed through a perceiving verb instead of dramatized ("she saw the anger in his face" → "his jaw clenched"), and **asserted / unearned knowledge** handed to the reader with no grounding ("somehow she knew", "he just knew it was over"). The LLM stage prompt now covers all four, and explicitly defers purely mechanical filter words ("she saw the bus pull up") to the deterministic `prose.filter-words` check so the two don't double-flag the same line. Existing installs auto-upgrade the prompt via a hash-driven migration (customized copies are left intact and warned about). (`data.reference/prompts/stages/pipeline-editorial-telling-emotion.md`, `scripts/migrations/137-editorial-telling-emotion-broaden.js`)

- **[issue-1592] The adverb check no longer flags invisible reporting dialogue tags like "said quietly" — only emotion-telling ones like "said angrily".** `prose.adverbs` treated *every* adverb on a dialogue tag as a weak attribution, but "said quietly / softly / sharply" is a near-invisible stage direction that tells the reader *how* a line was voiced, while "said angrily / happily / sorrowfully" *names the feeling* the dialogue itself should carry. Lumping them together produced false positives on legitimate reporting adverbs — which trains writers to ignore the check. The check now classifies each tag adverb as **reporting** (a curated manner/volume/pace allow-list) or **emotion-telling** (everything else, so a novel emotion adverb still gets caught) and flags only the emotion-telling bucket by default; the bulk per-1000-word adverb-density signal (the prose-body flagging) is unchanged. A new per-series **"Also flag reporting tags"** toggle restores the old flag-every-tag behavior for anyone who wants the strict pass. (`server/lib/editorial/proseTics.js`, `server/lib/editorial/checkRegistry.js`)

## Series Autopilot

- **[issue-1510] Series Autopilot now catches whole-book continuity problems at the cheap beat stage, before it generates full scripts.** The autopilot reviewed continuity at two altitudes — the cross-volume synopsis pass (before beats exist) and the full-text editorial pass (after every issue's script is written, the most expensive stage) — leaving a gap in between. So structural defects spanning the whole book (a cliffhanger raised in one issue and never resolved, a finale that drifts onto a new subplot instead of landing the arc's intended ending, a promised through-line that never pays off, an event staged as the "first" in two different issues, a broken volume-to-volume hand-off) were only caught *after* all the scripts already existed — or not until a human read it. The autopilot now runs a **whole-manuscript beat-continuity pass over every issue's beat sheet, between beat generation and script generation**. Blocking findings are auto-resolved by rewriting just the affected issues' beats and re-checking (bounded, then it pauses for review with the residual findings), so the fix lands before any script is generated. The full-text editorial pass is unchanged and still catches what only the verbatim script can show (pacing, repeated/looping passages, dialogue craft). A new **Beat continuity rounds** control in the autopilot Options tunes how many auto-resolve rounds it attempts (0 skips the pass), saved as a per-install default like the existing Arc-verify and Editorial round knobs.
- **[issue-1349] Series Autopilot now refreshes its scene segmentation before running the scene-based editorial checks, so those checks read the current draft.** The autopilot's scene-aware editorial checks (scene balance, POV justification, cliffhangers, arc transitions) read a cached "reverse outline" of the manuscript's scenes — but the autopilot's own editorial-completeness pass can edit the manuscript earlier in the same run, leaving that segmentation stale. The autopilot now regenerates the reverse outline between the completeness pass and the scene-based checks, so the checks always see the freshly edited draft instead of a pre-edit snapshot. It only spends effort when there's something to do: the refresh is skipped entirely when no scene-based check is enabled, and when the stored segmentation is already current — so a run that doesn't need it costs nothing extra.
- **[issue-1514] A per-stage provider pin now wins over Series Autopilot's run provider on *every* stage, not just script verification.** Pinning a single pipeline stage to a specific provider on the Prompts page already stuck for the script-verification gate, but the autopilot still threaded its run-wide provider as a hard *override* into every other stage it drives — arc generation and verification, episode seeding, prose/comic-script text stages, volume beats, the reverse-outline refresh, the editorial checks and their auto-fixes, and manuscript completeness — so a pin on any of those was silently ignored mid-run. The autopilot now passes its run provider as a soft *default* across all of them: a pinned stage keeps running on its chosen provider, while unpinned stages still follow the run's provider. As a deliberate consequence, when the run's provider is unavailable an unpinned stage now falls through to the active provider instead of failing the run with a "provider unavailable" error. Manual "regenerate with provider X" buttons are unchanged — an explicit per-call pick still wins everywhere.
- **[issue-1558] A per-stage *model* pin now also wins over Series Autopilot's run model — completing the #1514 work.** #1514 made a pinned stage keep its chosen *provider* during an autopilot run, but the run's *model* was still threaded as a hard override, so if you launched a run with both a provider and a model and a stage had its own provider+model pin, that stage ran on the pinned provider but with the *run's* model id — which could be wrong (or invalid) for the pinned provider, and ignored the model you deliberately set on the stage. The autopilot now passes its run model as a soft *default* too, with the same precedence as the provider: an explicit per-call model pick wins, then the stage's own model pin, then the run's model, then the provider's default — so a pinned stage now keeps both its provider *and* its model, while unpinned stages still follow the run's model.
- **[issue-1571] Series Autopilot now stops a verify/editorial loop early when it stops making progress, instead of burning the whole daily budget.** The arc-verification, beat-continuity, and editorial convergence loops auto-resolve their blocking findings and re-check, bounded by a round cap. Previously a loop that wasn't actually improving — a fix that re-introduced a different break, or that just oscillated between the same findings — kept spending AI budget round after round until it hit the cap and paused anyway. The autopilot now watches the blocking-finding count across rounds and, if it stops reaching a new low for two rounds running, pauses immediately with a distinct **"not converging"** state (visible as a badge on the paused panel) so you know it needs a human edit rather than more rounds. Loops that are genuinely making progress, and runs on the default round caps, are unaffected — the early stop only kicks in once a raised cap lets a stalled or oscillating loop keep going.
- **[issue-1573] Series Autopilot now flags editorial checks that errored instead of reporting a silent "clean" run.** Each editorial check runs inside a boundary guard so one bad check (a malformed prompt, a provider timeout, a missing context) can't abort the whole pass — but a check that *threw* produced no findings, and the autopilot's per-pass broadcast always reported "0 blocking" and completed as if everything had run. A check that errored every run looked clean until someone inspected the raw per-check results. The editorial-checks round frame now carries an **errored count + the failing check ids**, and a run that finished with any errored check is qualified on the terminal summary and the persisted status banner (a caution, not "production-ready") so the partial failure is visible — the dimension that errored was never actually evaluated. The standalone "run editorial checks" panel surfaces the same errored aggregate on its completion frame. (`server/services/pipeline/seriesAutopilot.js`, `server/services/pipeline/editorial/checkRunner.js`, `server/services/pipeline/series.js`, `client/src/components/pipeline/AutopilotPanel.jsx`)
- **[issue-1574] Series Autopilot now retries a failed beats/text child run before giving up, instead of marking it attempted and silently skipping it.** When the autopilot delegates to a child runner (volume beats or per-issue text/scripts), the child can finish with its target stage still empty because the underlying LLM call failed. Previously the work was marked *attempted* before the result was inspected, so the resolver skipped it on the next pass — a failed beats run wasn't caught until a downstream stage tripped over the empty `idea`, and a failed text run only paused after the fact. The autopilot now **verifies the child actually produced its output** (every issue in the volume has a beat sheet / the issue's required script stages landed) and, on a miss, **retries the child once** (skip-existing, so the retry only fills the gap) before marking the work attempted. Each attempt is budget-gated and bills one action like any other step; if the output is still missing after the retry the run emits an escalation frame and **pauses for review with the missing stages as residual findings** rather than marching on toward a "done" with no script. A per-run **`maxChildRetries`** option tunes the retry budget (0 restores the legacy single-attempt behavior). (`server/services/pipeline/seriesAutopilot.js`)
- **[issue-1575] Series Autopilot can now run a subset of editorial checks for a single run, without touching the global enabled set.** The editorial-checks pass always ran *every* enabled check, so piloting one new check — or skipping an expensive one for a single run — meant toggling checks globally and remembering to flip them back. The autopilot start payload now accepts an optional **`editorialCheckIds`** array; when present, that run's editorial-checks pass evaluates only those check ids (unknown or disabled ids are ignored), and the same subset drives the budget gate so billing and execution agree on the set. Absent or empty falls back to the full enabled set (the unchanged default). The dry-run plan also names the subset (e.g. *"per-run subset of 2 editorial check(s)"*) instead of implying the whole enabled set will run. (`server/services/pipeline/seriesAutopilot.js`, `server/routes/pipeline/autopilot.js`)
- **[issue-1576] The Series Autopilot dry-run plan now estimates its budget cost, so a large series on a small daily cap can be spotted before it burns through text generation and never reaches editorial.** The dry-run plan enumerated the steps it *would* run but gave no sense of how much of the daily CoS action budget they'd consume. Each planned step now carries an estimated **cos-action count** — the same unit the daily cap gates on (`recordDomainUsage('cos', …)`) — leaning toward the high end: convergence loops (arc verify, beat continuity, editorial review) counted at their max rounds (verify + resolve per round), per-item steps at one action per issue, and the pure gates that bill nothing (editorial health gate, canon verify) shown as zero-cost. (The editorial review's per-comment auto-fixes bill extra and scale with the findings, which isn't knowable at plan time, so a heavy editorial pass can exceed its estimate.) The editorial-checks pass — which bills a single action but fans out to many LLM calls whose count depends on manuscript chunking — additionally carries a rough **LLM-call proxy** (*issues × enabled LLM checks*, e.g. *"~18 LLM call(s)"* for a 9-issue series with 2 LLM checks). The plan panel shows each step's `≈N act` and an **Est. budget** footer totalling the run's actions and editorial-check LLM calls, on both the start and the terminal dry-run frame. (`server/services/pipeline/seriesAutopilot.js`, `client/src/components/pipeline/AutopilotPanel.jsx`)
- **[issue-1578] Series Autopilot now streams per-check editorial progress, not just a single end-of-pass total.** The editorial-checks pass runs every enabled check across the manuscript and previously broadcast only one terminal frame — a total finding count with `0 blocking` — so during a long (issues × checks) pass the UI could show only "running" until the whole pass finished, with no signal about which check was executing or what it found. The autopilot now forwards the runner's **per-check `check:start` / `check:complete` frames** up its live progress stream, each carrying the check's label and a **high/medium/low severity breakdown** of that check's findings, so the panel shows real-time progress like *"Editorial check: Info dumping — 3 finding(s) (1H/2M/0L)"* as each check completes (errored and gated-skip checks are labeled distinctly). The same severity breakdown is now attached to each entry in the run's `perCheck` summary. (`server/services/pipeline/editorial/checkRunner.js`, `server/services/pipeline/seriesAutopilot.js`, `client/src/components/pipeline/AutopilotPanel.jsx`)

- **[issue-1562] New "Full mirror" mode lets a peer mirror *everything* with one toggle.** Federation used to require enabling each sync category by hand, and records that existed *before* a peer was added were never back-subscribed — so you could never be sure a node pair was a true replica. - **Claude Code TUI agents now reliably wait for the input prompt before sending their task (and no longer wedge the shell on startup hiccups).** TUI-provider CoS agents (e.g. `claude-code-tui`) run their CLI as a *child* of a persistent PTY shell, and the old prompt-delivery gate fired on a crude "saw output, then went idle" heuristic. During Claude Code's startup that lull happens *before* the input box exists — and on a folder it hasn't seen, claude shows a **"trust this folder?" gate** that `--dangerously-skip-permissions` does not bypass (common because agents start in fresh worktrees). The nonce is split in the probe source (`'PORTOSRDY' '<nonce>'`) so the command echo never contains the assembled string — only the executed output matches — and instant-prompt keystroke buffering replays the probe into the real shell, so the gate is theme- and shell-agnostic with a bounded fallback if the probe never round-trips, (2) gates the paste on claude **re-enabling bracketed-paste mode** (`ESC[?2004h`, after the shell turned it off to run the command) as the input-ready signal — this is what makes the paste's leading `ESC` read as a paste rather than as Escape (which cancels claude's input — the intermittent "something other than Enter canceled it" failure); the visible footer text (`bypass permissions on (shift+tab to cycle)`) is deliberately NOT used because claude renders it with inter-glyph cursor moves that strip its spaces after ANSI-stripping, making it unmatchable, (3) **auto-confirms the folder-trust prompt** (Enter on the default "Yes, I trust") so claims run in fresh worktrees, (4) probes that the CLI is still a live child of the shell before pasting, and (5) **never blind-pastes** — if the input prompt never appears within 45 s (or the CLI exited), it finalizes the run as a failure carrying the CLI's captured output (`tui-not-ready` / `tui-exited-early`) instead of dumping into the shell.
