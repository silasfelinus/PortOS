- **Federated media collections + per-category sync integrity.** Media collections are now a first-class peer-sync record kind (`mediaCollection` in `PEER_SUBSCRIBABLE_KINDS`, schema-version-gated at `mediaCollections: 1`): they auto-subscribe on create, push per-record on edit, and propagate deletes via tombstones ŌĆö collections now reach parity between federated machines the way universes/series already did, instead of only riding along as `linkedCollection` cargo or the opt-in 60s snapshot. Required adding soft-delete to collections (`deleted`/`deletedAt`; `deleteCollection` marks instead of splicing; `listCollections({ includeDeleted })`; tombstone-aware LWW merge; tombstone GC), so a delete on one node can't be resurrected by a reverse-subscribed peer. Synced images now carry their **generation prompts**: the peer asset manifest advertises a canonical gen-params-only `sidecarSha256` (machine-local hash-cache stripped so it converges across peers), and the asset-pull worker fetches each image's `.metadata.json` sidecar alongside the bytes ŌĆö fixing images that previously landed in "Unsorted" with no prompts. A new per-category **sync integrity** surface (`GET /api/peer-sync/manifest`, `GET /api/peer-sync/integrity`, pure `computeRecordIntegrity` diff) drives a `SyncBadge` on every Universe / Series / Media-Collection row (in-sync / diverged / assets-missing / local-only / on-peer-only / "not syncing ŌĆö enable?") plus a deep-linkable `SyncDetailDrawer` (`/ŌĆ”/:id/sync`) showing the per-peer breakdown, previews, and manual actions: **Sync to peer** (`POST /sync-record`, force-push bypassing the unchanged-hash short-circuit), **Sync now** (`POST /sync-now`, backfill-subscribe + retry a peer's pending pushes), and **Pull missing prompts** (`POST /pull-metadata` + a button on the Unsorted view that backfills sidecars from online peers). `mediaCollections` sync stays **opt-in** per peer (default off); the badge surfaces a clear "enable?" state rather than flipping the default.
- **Manual tombstone GC trigger.** The 60s `syncOrchestrator` sweep buffers tombstones for 24h to give every peer time to ack a deletion before the bytes leave disk ŌĆö fine for normal use, painful right after a mass-delete. New `POST /api/sync/tombstones/sweep` (Zod-validated optional `{ graceMs }` clamped to `[0, 24h]`) calls `sweepTombstones({ graceMs })` so callers can shrink the buffer; the per-kind null-cutoff refusal (snapshot-mode peer with no per-record subscription) still fires independently of `graceMs` to preserve resurrection safety. Companion `GET /api/sync/tombstones/status` returns the dry-run refusal status without invoking the prune helpers. A new "GC tombstones now" section on the Instances page (between SelfCard/AddPeer and the peers list) POSTs `graceMs: 0` and surfaces per-kind prune counts + any refusals in a toast; the button is disabled only when every kind is refused. `scripts/gc-tombstones-now.js` is the equivalent CLI path (dry-run by default; `--apply` to prune) for headless cleanups ŌĆö uses the same "stop the server first, single-process write-tail isn't cross-process safe" wording as `cleanup-test-data.js`. `sweepTombstones` now returns `{ universes, series, issues, refused }` so the manual trigger can explain why a kind didn't shrink. The orchestrator path keeps the 24h default behavior unchanged.
- `server/lib/collectionStore.js` ŌĆö per-type, per-record JSON storage helper with explicit type-level `schemaVersion` stamping. `createCollectionStore({ dir, type, schemaVersion, sanitizeRecord })` returns `loadOne` / `saveOne` / `listIds` / `loadAll` / `deleteOne` / `loadTypeIndex` / `saveTypeIndex` / `verifySchemaVersion`. - **Cross-instance sync now gates on schema version.** New `server/lib/schemaVersions.js` is the single source of truth for the per-category storage-layout contract (`PORTOS_SCHEMA_VERSIONS = { universes: 5 }`); future category bumps add entries. Every outbound sync payload (federated peer push, 60s snapshot sync, share-bucket manifest) carries a `portosMeta { portosVersion, schemaVersions }` envelope. Receivers run `compareSchemaVersions(sender, local)`; when the sender is **ahead** on any category the receiver REJECTS ŌĆö federated peer push returns 409 + `{ code: PEER_SYNC_SCHEMA_VERSION_AHEAD, context: { details: { ahead, behind, senderPortosVersion, receiverSchemaVersions } } }`, snapshot sync surfaces `blockedBySchema` on the peer record, share-bucket import skips the manifest with reason `portos-schema-ahead` and emits a `sharing:portos-schema-ahead` socket event. Senders persist the gap (`sub.blockedBySchema` on peer subscriptions, `peer.schemaGaps[category]` on peer records via `updatePeer`) and pause retries (5-minute cooldown; `peer:online` reconnect bypasses to re-probe in case the peer upgraded). Legacy senders without `portosMeta` pass through unchanged. The Instances page renders a new `<SchemaGapBadge>` per peer card explaining the gap in both directions: "Peer X is on an older PortOS ŌĆö they need to update before we can sync universes" / "Peer X is on a newer PortOS ŌĆö update PortOS to receive their universe updates." A "sender behind" mismatch DOES NOT gate ŌĆö the existing sanitizer chain backfills older inputs.

## Fixed

- **Background agent cleanup no longer removes an in-progress manual claim's workspace.** A long-running hands-on task could have its isolated workspace deleted out from under it by the periodic agent-cleanup sweep, which mistook the human-owned workspace for an abandoned agent one. The sweep now leaves manual-claim workspaces alone and only reaps truly abandoned agent workspaces, and it runs on the daily maintenance schedule instead of every 15 minutes.

- **Federated media-collection sync reconciliation + resolution UX.** Follow-up to the federated-media-collections feature, fixing issues found running it across two machines. (1) **Universe/series-linked collections now reconcile across peers.** Each machine used to mint a random UUID for its "Universe: X" / "Series: Y" collection, so per-record sync (keyed on `id`) treated each machine's copy as distinct ŌĆö integrity reported both sides "local only" and the merge created a *duplicate* on every peer. Linked collections now derive a deterministic id (`uc-<universeId>` / `sc-<seriesId>`); create paths reclaim the id when a tombstone holds it (re-creating a deleted collection). Migration 038 canonicalizes existing ids on each machine, merging (union of items) any same-owner duplicates and rewriting `mediaCollection` subscription recordIds ŌĆö idempotent; both machines converge independently. (2) **New "Pull from peer" action** (`GET /api/peer-sync/record` + `POST /api/peer-sync/pull-record`) lets a machine that is *behind* on a record fetch it and background-pull its asset bytes via `applyIncomingPush`, instead of "Sync to peer" being push-only; the Sync Details drawer is now direction-aware (Pull for peer-only, Sync for local-only, both for diverged/assets-missing). (3) **The Sync Details drawer can no longer hang on "LoadingŌĆ”"** ŌĆö the record fetch gets a 12s timeout + explicit error state. (4) **Removed the unbounded "Live-pushed records" per-record list** from the Instances page (the Sync Details drawer covers per-record status). `mediaCollections` sync remains opt-in per peer ŌĆö enable the "Media Collections" category on the Instances page for collections to sync.
- **[upcoming-tasks-widget-drop-catch-to-empty] Upcoming Tasks widget no longer blanks on a transient hiccup** ŌĆö a momentary failure of the dashboard's Upcoming Tasks background refresh now keeps the last-loaded tasks on screen instead of clearing the list.
- **[propagate-runner-errorAnalysis-into-onRunFailed-context] Propagate `metadata.errorAnalysis` into the `onRunFailed` event context.** Runner-provided error analysis now rides the failed run result into `promptRunner`, so fallback provider marking reuses the category and wait-time already computed by the runner instead of analyzing the wrapper error again. The emitted `AI_PROVIDER_EXECUTION_FAILED` context also includes `errorAnalysis` for downstream auto-fix and diagnostics consumers.
- **`plan-task` agents now claim their own PLAN.md item instead of one the scheduler pre-picks.** The scheduled "plan-task" job used to reserve a PLAN.md slug at dispatch time and hand it to the agent as a hard constraint, but the actual lock (the `claim/<slug>` branch) isn't created until the agent runs ŌĆö so two runs firing close together could both be handed the same first-available item, and one would waste its turn (the failure mode behind the 2026-05-21 duplicate-PR incident). The agent now picks the next unclaimed item itself at execution time and immediately creates the claim branch, exactly like the `/claim` command, so concurrent runs naturally land on different items. The dispatch-time scan is kept only as a gate that skips the run when nothing is claimable. Existing scheduled-task prompts auto-upgrade on next load (`plan-task` prompt v6 ŌåÆ v7); `feature-ideas` is unaffected.
- **Codex agents can now open PRs.** Codex CLI agents were spawned as `codex exec` with no sandbox/approval bypass, while Claude (`--dangerously-skip-permissions`) and Gemini (`--yolo`) got one ŌĆö so Codex ran under the default sandbox that blocks network (its `gh pr create` couldn't resolve `api.github.com`) and, in non-interactive mode, silently cancelled any command needing approval. A Codex agent could push a branch over SSH yet fail to open the PR over HTTPS. Codex now gets `--dangerously-bypass-approvals-and-sandbox`, matching the other CLIs' unrestricted posture on PortOS's private single-user machine.
- **Non-Claude agents get a working "simplify" step.** `/simplify` is a Claude Code built-in slash command, so Codex/Gemini agents couldn't run the literal `/simplify` instruction baked into task prompts. The reuse/quality/efficiency self-review step is now phrased so any CLI can do it ŌĆö Claude Code still runs `/simplify`, while other CLIs are told to do the equivalent diff review by hand. Applies to the scheduled-task templates and the per-agent completion sections.
- Providers (`/ai`) and Prompts (`/prompts`) pages now render the same Settings sub-nav tabbed header as the other settings pages, so users can hop between Backup / Database / General / MortalLoom / Prompts / Providers / Sharing / Telegram / Voice without going back to the sidebar. Extracted a shared `SettingsTabsHeader` component used by `Settings.jsx`, `AIProviders.jsx`, and `PromptManager.jsx`.
- **[test-autosubscribe-fanout-getpeers-guard-remaining] Apply the `getPeers ŌåÆ []` test stub to the remaining record-creating test files (and consider a shared guard).** Added `mockNoPeers()` to the shared test helper module and applied it across the remaining universe/series-creating suites, preserving the sharing tests' explicit identity stubs. Documented the convention in `CLAUDE.md` so new record-creating tests do not leak fixtures into live peer sync data.
- **Test fixtures no longer leak into the real `data/universes/` via peer-sync fan-out.** `createUniverse` fires a non-awaited `autoSubscribeRecordToAllPeers`, which reads the live peer registry through `instances.js#getPeers` ŌĆö a `dataPath` closure that resolves to the REAL `PATHS.data` once the post-return microtask escapes a test's `fileUtils` mock window. Three universe-creating suites (`universeBuilder.test.js`, `universeBuilderPromote.test.js`, `universeCanon.test.js`) created non-ephemeral fixtures without stubbing `getPeers`, so their "Moebius SciFi" universes were pushed across the federation and persisted into real `data/` by receiving instances (and re-leaked intermittently on every run). Each suite now mocks `getPeers ŌåÆ []`, matching the existing guard in `importer.test.js` / `promoteToPipeline.test.js`.
- **[dashboard-active-layout-revert-tracks-server-confirmed-id] Failed dashboard layout switches snap back to the real active layout.** If two dashboard layout switches fail to save back-to-back, the dashboard now reverts to the layout the server last confirmed instead of getting stuck displaying one that was never actually applied.
