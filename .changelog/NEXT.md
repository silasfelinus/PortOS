## Added

- **Image gallery: remove the Gemini / Nano-Banana ✦ sparkle** — gallery images now get a **Remove ✦** action in the lightbox that erases the visible sparkle logo Google stamps into the bottom-right corner of its image-gen output. It localizes just that corner and reconstructs it from the surrounding pixels (a dependency-free harmonic inpaint — no GPU, runs on every install), leaving the rest of the image untouched, and saves the result as a new variant alongside the original. This is distinct from the existing **Regenerate** (which overwrites the *invisible* SynthID watermark by re-rolling the whole image) — the sparkle is a visible, fixed-position logo, so it gets a surgical corner fix instead. Available everywhere the media lightbox appears (Image Gen, History, Collections, Universe Builder).
- **[issue-900] Autobiography: deeper follow-ups and a woven life story** — the autobiography's AI follow-up questions now get more reflective as a story thread grows, moving from "what happened" to "what it meant" so each round builds a richer narrative instead of circling the same details. Once a story is part of a follow-up chain, a new **Weave** button stitches the whole thread — every prompt and answer — into a single flowing first-person passage you can read and copy.
- **Local LLM catalog: Nex-N2-mini** — the local-model install picker now lists Nex AGI's Nex-N2-mini, a 35B/3B-active agentic MoE (Apache-2.0, built on Qwen3.5) strong at coding, tool calling, long-horizon agent tasks, and vision. The Q4 GGUF is ~22 GB — comfortable on a 32GB machine and trivial on a 128GB unified-memory Mac. Its 397B sibling Nex-N2-Pro is intentionally left out: it won't fit a 128GB Mac even at Q4.
- **[issue-1028] Songs: a practice mode to learn and memorize a song** — the song editor and reader gain a Training panel that turns the sheet music into a drill. Pick the whole song or a single section, get a one-bar count-in, sing it, and watch each note light up green/yellow/red with an accuracy score for the take — then repeat, or turn on Loop to keep cycling the same part. PortOS tracks your progress over time: each section earns a rolling average and flips to "learned" once you hit it accurately a few takes in a row, a memorization bar shows how much of the song you've got down, and it nudges you toward your weakest sections to practice next. As your accuracy on a part climbs, the lyrics fade and finally hide so you're singing it from memory. It runs entirely on the built-in pitch detection — no extra audio library — and completes the song system's vocal pitch-detection and training toolkit.
- **[issue-1025] Songs: color-match playback that grades you against the score** — sing along while recording and the sheet music grades you note by note in real time. A one-bar metronome counts you in, then the song walks through the written melody in tempo and lights each notehead green when you're in tune, yellow when you're close, and red when you miss or stray off pitch. When you stop, you get a per-take score — the percentage of notes you nailed — so you can track how well you're learning the song. It listens to the same take you're recording (no second microphone), works on mobile, and shuts the mic and analysis down cleanly when you stop.
- **Songs: a live vocal tuner while you record** — the song recorder now shows the note you're singing and how flat or sharp it is, in real time, right next to the metronome. Hit a take and the tuner reads the same microphone you're recording into (it never opens a second mic), displaying the detected note and a green/yellow/red cents needle that swings flat ↔ sharp and centers when you're in tune. When you're not recording, a **Tune** button opens a standalone tuning mode that listens to the mic without recording or saving anything — handy for warming up or finding a starting pitch. Mobile-friendly, and the mic and analysis shut down cleanly the moment you stop or leave the page. Built on the in-house pitch-detection core — no third-party audio library. (#1023)
- **[issue-1026] Songs: sing a melody and turn it into sheet music** — the score editor has a new **Sing to score** panel. Press Sing, get a one-bar count-in so you know where the downbeat is, sing your phrase, then press Stop — PortOS listens to the melody, picks out the notes, snaps their timing to the song's tempo, and writes them out as notation. The detected notes preview on a real staff before you commit, and you can append them to the end of your score or drop them in place of whatever you've selected in the editor. Notes are spelled to match the song's key (flats in a flat key), and it all runs on the built-in pitch detection — no extra audio library.

## Fixed

- **Database page works on Windows** — the Database page's Start/Fix/Status actions invoke the `db.sh` manager through bash; on Windows the script's native path (`H:\…\db.sh`) was passed with backslashes, which bash collapsed into a bogus path (exit 127, "No such file or directory"), so the page reported the Docker database as stopped even while it was running. The path is now normalized to forward slashes, which Git Bash accepts on Windows. A second Windows-only failure in the per-backend export path (spawning `mkdir -p`, which isn't a real executable on Windows) was also fixed by using Node's filesystem API instead. Mac/Linux behavior is unchanged.
- **Database page works on Windows (follow-up: use Git Bash, not WSL)** — even after the path was forward-slashed, Start/Fix/Status still failed with exit 127 on machines that have **both** Git Bash and WSL installed: the server (launched by PM2) resolved a bare `bash` to WSL's `C:\Windows\System32\bash.exe`, which mounts drives at `/mnt/h/…` and genuinely cannot see a `H:/…` drive path — and would run `db.sh` against Linux tools even if it could. PortOS now resolves the interpreter explicitly, preferring **Git Bash** (the correct Windows interpreter), with a `PORTOS_BASH` override for non-standard installs. Mac/Linux behavior is unchanged (bare `bash`).

## Fixed

- **Federated brain sync no longer loops forever** — on installs with two or more federated machines, deleting a Brain item (a person, project, idea, link, etc.) could quietly resurrect it: a peer would echo back the old "create," the item would reappear, the delete would re-run, and the two halves would ping-pong between machines indefinitely — flooding the logs with constant sync activity and ballooning the sync counters into the tens of thousands. Brain deletes now leave a durable "tombstone" so a stale create from a peer is rejected instead of reviving the record, matching how the creative catalog already handles deletes. A one-time cleanup neutralizes any already-resurrected items and compacts the runaway sync log (one install collapsed from 74,000 log entries to 7) without disturbing peer sync positions, and old tombstones are swept automatically after a grace period. If you disabled federation sync to stop the churn, it's safe to re-enable once every machine is updated.
