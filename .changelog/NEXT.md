## Fixed

- **`update.sh` / `update.ps1` now start PortOS when it wasn't already running.** Both scripts `pm2 kill` the daemon early to clear stale fork paths, then previously called `npm run pm2:restart` to bring the app back. `pm2 restart <ecosystem>` against a torn-down daemon doesn't reliably (re)start processes that aren't currently managed, so users who ran the updater while PortOS was stopped ended an "update complete" cycle with no server running and the auto-open script timing out against a dead port. The restart step now mirrors the `npm start` pattern — `pm2 delete --silent` → `pm2 start ecosystem.config.cjs` → `pm2 save` — which boots a fresh daemon, starts the apps, and persists the process list so they come back on reboot. The `STEP:restart:…` markers are preserved so the UI progress panel (`UpdateTab.jsx`) continues to render correctly.

- **[med-pipeline-server-services-pipeline-arcplanner] `sanitizeSeasonList` now warns when LLM-emitted seasons collide on id.** `server/lib/storyArc.js#sanitizeSeasonList` dedupes by id with last-write-wins (Map.set overwrite) on every series persist. Two LLM-emitted seasons sharing an id were silently dropped before persist while child issues kept their `seasonId` pointers — leaving the issues attached to the surviving season under the colliding id, which is rarely what an arc-resolve LLM rewrite intended. The helper now tracks colliding ids during the loop and emits a single `console.warn` listing the offending ids (counting both total overwrites and distinct colliding ids) so the responsible run is grep-able in logs. Behavior is preserved (LWW still wins, no throw); the warning is the new signal. Four tests added to `storyArc.test.js` cover the warn, the unique no-warn baseline, the multi-id collision listing, and the same-id-3+-times listing (deduped via Set).

- **[med-migration-scripts-migrations-014-attribute] Migration 014 no longer writes `'unknown'` as a phantom annotation author.** When the migration runner fires before `ensureSelf()` on a fresh install (the order in `server/index.js` — migrations at startup, identity later), `instances.json` is absent and the old fallback `instances?.self?.instanceId || 'unknown'` stamped the literal string `'unknown'` into every legacy annotation entry forever (`setAnnotation` refuses to write under `'unknown'` and `annotationsSync.flushAll` refuses to export it, so those notes became unrecoverable / unexportable). `scripts/migrations/014-attribute-existing-annotations.js` now lazy-creates the local identity inline (same shape as `ensureSelf()` — `randomUUID()` + `os.hostname()`) and writes it to `data/instances.json` before reading. Read-side belt-and-suspenders: `mediaAnnotations.readAll()` heals any pre-existing `authors.unknown` bucket by re-keying it to the real `localInstanceId` (in-memory only; the first subsequent `setAnnotation` persists the clean shape via the normal write path). When a real local entry already exists alongside an unknown phantom, the real entry wins and the phantom is dropped. Tests in `scripts/migrations/014-attribute-existing-annotations.test.js` and `server/services/mediaAnnotations.test.js`.

- **HF token banner no longer surfaces in Codex / External mode.** The `needsHfTokenGate` predicate in `client/src/pages/ImageGen.jsx` checked `currentModel?.requiresHfToken && !isFlux2Model` but forgot to also require `isLocalMode`. A user who'd selected a gated mflux model (e.g. FLUX.1-dev) and later switched the backend chip to Codex saw the "Flux 1 Dev is a gated model — paste your HF token" banner over the Codex form, even though Codex uses gpt-image-2 and never touches HuggingFace. The gate now requires `isLocalMode` first, matching the FLUX.2 install/status block right below it.
