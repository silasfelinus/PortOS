# Next Release

## Added

- **Describe a canon character / place / object from reference image(s)** — Each canon card in the Universe editor (Cast, Places, Objects) gains a **Describe from image(s)** button (the scan-text icon next to Render). Upload one reference image to turn it into an image-gen-ready prose description, or several images of the *same* subject (a character from multiple angles, a location across lighting conditions) and a vision model returns the description **common to all of them** — ignoring incidental differences like pose, background, and framing. You pick the **provider + model on demand** in the modal (only API/vision-capable providers are offered), edit the generated prose before applying, and it writes straight into the entry's descriptor field (`physicalDescription` for characters, `description` for places/objects) — the same field the render prompt reads. Vision runs through the existing API runner's `image_url` content-block path; the new `runPromptThroughProvider({ screenshots })` arg threads images through the shared runner, and the stateless `POST /api/universe-builder/describe-from-images` endpoint resolves an API provider (clear `NO_API_PROVIDER` error if none is configured) and builds the per-kind vision prompt. - **Character LoRA training — train consistent character adapters from your universe canon** — Create → Media Gen gains a **Training** tab (`/media/training`) that closes the loop from "character exists in a universe" to "every render of that character looks the same." Per character, a **dataset workbench** builds reference material three ways: batch-generate single-subject training renders from the character's canon descriptors (deterministic view/pose/expression/outfit variation matrix, queued through the normal image lane), upload your own images (normalized to PNG via sharp), or slice the character's existing reference-sheet turnaround into grid crops. A **vision-LLM captioner** (same provider pathway as the voice agent's visual describe) auto-captions every image prefixed with the character's trigger word, with per-image caption editing and re-captioning. Character cards in the universe editor and catalog detail pages show a trained-LoRA chip + dataset entry point; trained LoRAs in the library link back to their dataset. Datasets are machine-local (like the LoRA files themselves — never federated) and fully backed up; training checkpoints/caches are excluded as regenerable. Run records live in PostgreSQL (`lora_training_runs`). - **LoRA training picks the best checkpoint instead of blindly deploying the final step — and you can promote any checkpoint by eye** — A FLUX.2 character LoRA could finish "successfully" yet render garbage: the trainer always handed back the FINAL step's adapter, but late training **divergence** (a recurring FLUX.2 failure — the loss keeps dropping while the sample image collapses to a near-black frame) made that final step the worst one. Loss is no guide here; on the run that motivated this it was *anti-correlated* with quality (the lowest-loss checkpoint was visually broken). Two changes close the gap: (1) a **collapse guard** at finalize analyzes each checkpoint's preview (sharp entropy + per-channel stdev) and, when the final preview is a degenerate near-black/near-uniform collapse, deploys the latest *healthy* checkpoint instead and flags the run for review — calibrated with a wide margin (a real frame measured entropy 5.5–7.1 vs the collapse's 2.8) so it never vetoes a legitimate image; (2) a new **checkpoint picker** under a completed run (`/media/training`) shows every saved checkpoint as a preview thumbnail with its step + loss, and a **"Use this"** button re-extracts that step's adapter and overwrites the deployed LoRA in place — the human override for subtler degradation the guard leaves alone. Per-checkpoint loss is now captured at checkpoint time, and `lib/zipStream` gains `extractZipEntryToBuffer` to crack one adapter out of an mflux checkpoint zip long after the trainer exited. The final debounced checkpoint/sample now flush to the run record before finalize reads it, so the guard always sees the last artifacts. (`server/services/loraTraining/checkpoints.js`, `server/services/loraTraining/index.js`, `server/lib/zipStream.js`, `server/routes/loraTraining.js`, `client/src/components/loraTraining/CheckpointPicker.jsx`)

- **LoRA training defaults tuned down to avoid divergence** — The default training recipe drove the divergence above: at 1000 steps / learning-rate 1e-4 the sample collapsed by ~step 500 while the loss kept falling. Defaults are now **600 steps, lr 5e-5, 768px** (up from 512 — a 4K source crushed to 512 loses the face), with checkpoint/sample cadence tightened to every 150 steps so the picker has more healthy steps to choose from. Existing installs that pinned their own `settings.loraTraining.defaults` keep them. (`server/services/loraTraining/runtimes.js`)

Leaving the picker on its defaults preserves the prior behavior (active provider, per-task model selection). Only AI-agent jobs show the picker; shell/script jobs never reach the AI runner. A pinned provider is now resolved *before* the active-provider availability gate, so a job pinned to a healthy provider runs even when the globally active provider is down or unset. ## Fixed

- **Tests can no longer wipe your real universes, series, and writing — and every deletion is now audited** — The DB-backed `*.db.test.js` suites (universe, pipeline series/issues, story builder, writers room, catalog, creative director, media index) connect to the single real `portos` Postgres whenever it's reachable and run `DELETE FROM …` in their setup, restoring a snapshot afterward. Because git worktrees share that one database, a CoS agent running the suite in its worktree wiped all 12 universes + 4 series + the writers-room works when its process was torn down before the restore finished (the snapshot/restore swallowed every error). Two new guards in `server/lib/db.js` close this off: (1) `checkHealth()` reports "disconnected" under `NODE_ENV=test` against any non-test database, so every db-backed suite **skips cleanly** instead of deleting — a database is "safe for destructive tests" only when its name ends in `_test` (e.g. `portos_test`) or `TEST_DB_OK=1` is set (`isTestDatabase()`); (2) `query()` is a hard backstop that **throws** on `DELETE FROM` / `TRUNCATE` under the test runner on a non-test DB even if a suite reaches it without gating first. Separately, a new **`record_audit` table + generic DB trigger** now logs every tombstone, un-tombstone, and hard-delete of user-authored content (universes, pipeline series/issues, story-builder sessions, writers-room works/folders/drafts, catalog ingredients/scraps/types, creative-director projects, LoRA runs) — captured at the database layer so it records deletions from *any* source (the app, a stray test, a manual `psql`), with the old row snapshot kept for recovery and the originating SQL for forensics. Inspect it with `node scripts/show-deletions.mjs`. (`server/lib/db.js`, `server/scripts/init-db.sql`, `server/lib/db.guards.test.js`, `server/lib/db.catalogDdlParity.test.js`, `scripts/show-deletions.mjs`)

- **Opening a LoRA dataset whose universe was deleted no longer spams the server log with `❌ Route error`** — The dataset workbench (`/media/training/:id`) speculatively fetches the dataset's `character.universeId` ({ silent: true }) to pre-load the canon character's variation axes. When that universe has since been deleted, the lookup 404s — harmless, and the client already swallows it — but the server logged every 404 as `❌ Route error [GET /api/universe-builder/<id>]: Universe not found`, and since the page reconnects every ~20s it logged the same line on a loop. `GET /api/universe-builder/:id` now tags a not-found result `severity: 'warning'`, so the error middleware skips the log line (and the client's error-toast hook already drops warnings) — matching the existing benign-404 precedent for speculative media-job archive lookups. (`server/routes/universeBuilder.js`)

- **A gated-model LoRA training failure now shows a one-click "Request access" link, like image rendering already does** — Training a FLUX.2 Klein LoRA pulls a *different, separately-gated* HuggingFace repo than rendering does (the mflux trainer resolves `flux2-klein-9b` → `black-forest-labs/FLUX.2-klein-base-9B`, whereas inference uses `FLUX.2-klein-9B`). So a user who'd accepted the inference model's license — and whose stored token is otherwise valid, with `canReadGatedRepos` — still hit a `GatedRepoError` the instant training started, with no obvious next step. Image rendering already turns this into an actionable banner with a deep link to the license page; training now does the same. The failure classifier extracts the gated `owner/repo` from the trainer's stderr (both the `Access to model … is restricted` and `Cannot access gated repo for url …` forms), persists it on the run record as `errorRepo`, and the training panel renders a **"Request access to `<repo>` ↗"** button that opens the exact license page — accept it there with the token's account and retry, no token change needed. The repo-extraction the image runner already did inline is now a shared `extractGatedRepo()` helper (`server/lib/hfErrors.js`) both subsystems consume. (`server/lib/hfErrors.js`, `server/services/loraTraining/failure.js`, `server/services/loraTraining/index.js`, `server/services/imageGen/local.js`, `client/src/components/loraTraining/TrainingPanel.jsx`)

- **mflux-trained FLUX.2 LoRAs now load into diffusers (double-block `to_out` key mismatch)** — Even on the bf16 base, an mflux-trained adapter failed to load with `Target module ModuleList(...) is not supported`, so the whole LoRA was dropped and the render fell back to base. Cause: mflux targets a double transformer block's `attn.to_out` as a bare Linear and exports the key `transformer_blocks.N.attn.to_out.lora_*`, but diffusers' `Flux2Attention` wraps that projection in `ModuleList([Linear, Dropout])` — PEFT needs the inner `.0`. The shared LoRA loader (`scripts/lora_utils.py`) now normalizes mflux adapters on load: it inserts the `.0` for double-block `to_out` only (single-block `to_out` is a bare Linear in diffusers and is left alone), then hands diffusers the remapped state dict. Verified against the real diffusers Flux2 structure — all 100 of the adapter's target modules resolve to `nn.Linear` after the fix. Self-healing: existing trained LoRAs are fixed at load time with no re-training or migration. Every other mflux target already matched diffusers naming. (`scripts/lora_utils.py`)

- **Trained character LoRAs now actually apply at render time (were silently ignored on quantized FLUX.2)** — A trained LoRA attached to a FLUX.2 Klein render did *nothing*: the default inference model is the **SDNQ 4-bit quantized** Klein, and PEFT can't inject an adapter into SDNQ-quantized `Linear` layers (`Target module SDNQLinear … is not supported`). The runner's LoRA loader is best-effort, so it logged the failure and rendered the **base model anyway** — every "LoRA" render was the plain base, which looked like a broken LoRA. Since a trained adapter is trained against the bf16 base, the image runner now **auto-routes any LoRA render off the quantized repo onto the gated bf16 base** (`black-forest-labs/FLUX.2-klein-{4B,9B}`, resolved from the model's size variant) with `quantization=none`; non-LoRA renders stay on the fast SDNQ/Int8 path. The bf16 base repos are now a single source of truth in `lib/runners.js` that LoRA training reuses, so the train-against and render-against repos can't drift. (`server/lib/runners.js`, `server/services/imageGen/local.js`, `server/services/loraTraining/runtimes.js`)

- **LoRA training that died with a bare "exited with code 2" now says what actually broke — and a stale mflux fails loudly instead** — A FLUX.2 LoRA training run could fail instantly with the useless message `Training exited with code 2 — ❌ mflux trainer exited with code 2`. The real cause was a stale **mflux 0.12.x** in the local image-gen venv: that version predates FLUX.2 LoRA training, so its `mflux-train` CLI rejected the wrapper's `--config` flag (it wants `--train-config`) and had no `flux2-klein-*` base models — argparse exited 2 before any work began. The actual argparse error was lost because the wrapper merges mflux's stderr into stdout, leaving the Node failure classifier's stderr window empty. Two fixes: (1) on a non-zero exit the wrapper now replays the trainer's last ~40 output lines to stderr (so tracebacks, argparse errors, and version-mismatch complaints reach the classifier instead of vanishing), and a new `CLI_MISMATCH` classification turns the argparse rejection into an actionable *"your mflux is too old for FLUX.2 LoRA training — `pip install -U 'mflux>=0.17'`"*; (2) `scripts/setup-image-video.sh` now pins **`mflux>=0.17`** (the first release with FLUX.2 training) and lets pip resolve mflux's declared runtime deps instead of a hand-kept list — mflux 0.17 imports packages the old list omitted (pillow, matplotlib, platformdirs, sentencepiece, …) and constrains `transformers>=5,<6` and `mlx<0.32`, so the stale list (which pinned `transformers<5` and an unbounded `mlx`) would have left `mflux-train` failing at import. (`scripts/train_mflux_lora.py`, `server/services/loraTraining/failure.js`, `scripts/setup-image-video.sh`)

- **Concurrent LoRA caption runs no longer fan out parallel requests at a single-GPU backend** — Within one run the captioner already processes images one at a time, but nothing stopped several runs from overlapping (captioning two datasets at once, re-captioning multiple single images, or a double-click). Each run is a detached loop, so concurrent runs fired N parallel `/chat/completions` requests at the same local Ollama/LM Studio — and on one GPU that's pure downside: every Ollama parallel slot reserves its own 32K-token KV cache, so concurrent vision requests multiply VRAM and risk an OOM or model reload, while parallel decoding gives no throughput win (it just time-slices the same compute and makes each request slower). A module-level mutex now serializes every caption vision call across all runs, holding backend load at concurrency 1 regardless of how many runs are in flight. One-shot vision calls (voice `ui_describe_visually`, the vision-test page) stay off the lock so a long batch run can't FIFO-starve an interactive describe. (`server/services/loraDatasetCaption.js`)

- **LoRA caption failures now tell you they happened instead of failing silently** — The captioner already reported per-image failures and a final failure tally over its progress stream (a vision model refusing an image or returning an empty description), but the dataset workbench threw that away: the "Caption all" button just stopped spinning and the run looked done, even when half the images were never captioned. The terminal frame is now surfaced as a toast — an actionable error when the whole run failed up front, a "Captioned 8/10 — 2 images failed (model refused…). Re-caption individually or pick another vision model." summary when some failed, and a success confirmation when all landed. This also covers the per-image "Re-caption" button, whose failures were previously invisible. (`client/src/pages/LoraDatasetDetail.jsx`)

- **LoRA auto-captions no longer silently come back as just the trigger word** — When a vision model returned an empty reply (common when it refuses a realistic close-up face, or when a thinking model spends its whole token budget reasoning), the captioner prefixed that empty string with the trigger word and saved the result — so the image was bound but undescribed, yet the run still logged it as captioned. A blank model reply is now treated as a per-image failure: it's surfaced in the caption run's progress (with the model name and a hint to try a different vision model or caption manually) and never persisted, so the image stays un-captioned and a re-run re-attempts it instead of leaving a degenerate trigger-word-only caption. (`server/services/loraDatasetCaption.js`)

- **LoRA caption failures now name the real cause — reasoning model vs. refusal — instead of always blaming a refusal** — A blank vision reply used to fail with one fixed message ("it may have refused this image"), which was usually wrong: capability-tagged reasoning models (a `qwen3.6:35b`-style MoE, thinking Gemma builds) emit a hidden `<think>` block FIRST and, at the old 300-token caption budget, routinely hit the budget mid-reasoning and return *empty* content — a token-exhaustion problem, not a refusal. The captioner now reads the response's `finish_reason`, token usage, and any out-of-band reasoning (`describeImageDataUrlDetailed`) and writes the diagnosis into the failure: "it spent its whole token budget on hidden reasoning (spent N completion tokens, the model-reported count) and ran out before writing a caption — this is a reasoning model, not a vision model. Pick a dedicated VLM (Qwen2.5-VL, LLaVA, MiniCPM-V, Llama 3.2 Vision)" for a thinking model, a plain truncation message when cut off with no reasoning trace, and the original refusal wording only when there's genuinely no diagnostic signal. The per-image reason now also reaches the dataset-workbench toast (not just the generic tally), so a single-image run tells you exactly what to do. The caption token budget was also raised 300→600 to give a borderline reasoning model enough headroom to finish; dedicated VLMs stop well before it, so it costs them nothing. (`server/services/visionTest.js`, `server/services/loraDatasetCaption.js`, `client/src/pages/LoraDatasetDetail.jsx`)

- **LoRA caption-model picker now sees vision-capable Ollama models whose name lacks a `vl`/`vision` token** — The picker showed "none installed" even with a vision model like the `qwen3.6:35b` MoE installed, because Ollama's `/api/tags` carries no vision flag and the id heuristic only matches names containing a `vl`/`vision`/known-family token. Ollama *does* report `capabilities: ["…","vision",…]` via `/api/show`, but PortOS only consulted that for LM Studio. `listVisionModels` now enriches each Ollama model the id heuristic doesn't already catch with its `/api/show` capabilities (cached per model, busted on delete), so capability-tagged vision models surface in the picker — and, through the same path, in the captioner's auto-pick and explicit-model validation. (`server/services/ollamaManager.js`, `server/services/localLlm.js`)

- **TUI agents no longer leave the prompt sitting unsent in the input box** — CoS agents that run inside a terminal-UI provider (Claude Code, Codex) bracketed-paste their task prompt and then send a single Enter to submit it. On a large multi-line prompt that one `\r` could arrive while the TUI was still reflowing the paste and get swallowed — so the prompt sat typed-but-unsubmitted, the agent did no work, idled out after 3 minutes, and was (wrongly) recorded as a successful run. You'd see it only by opening the Shell tab and hitting Enter yourself. The submit Enter now fires a few times spaced ~700ms apart so one lands after the paste settles; re-sending is safe because once the prompt submits the input box is empty and a bare Enter is a no-op. Applies to both the long-running agent path and the one-shot TUI runner. (`server/lib/tuiHandshake.js`, `server/services/agentTuiSpawning.js`, `server/lib/tuiPromptRunner.js`)

- **`calendarSync` test no longer depends on the developer's machine state** — the "throws 401 when Google OAuth is not configured" test read the REAL Google credentials/tokens off disk, so on a machine with OAuth configured it failed with a `GaxiosError` (400 invalid_grant) instead of the pinned 401. The auth client is now mocked to the not-configured state. (`server/services/calendarSync.test.js`)
