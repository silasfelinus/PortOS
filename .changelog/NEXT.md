# Unreleased

## Creative onramp

- **[issue-1633] A new "Start a Story" page gives you one clear front door for beginning a story.** Instead of guessing whether to open the Story Builder, the Importer, or the Writers Room, you now pick how you want to begin — from an idea, from an existing work, or by writing prose — and the page routes you into the right tool. You can also choose up front whether to start a fresh universe or attach the story to one you already have; that choice carries through to the Story Builder and Importer automatically. All the existing pages stay exactly where they were — this just adds a signposted entry point (reachable from the sidebar, ⌘K, and voice). (First phase of the larger creative-onramp effort; prose-mode universe linking comes later.)
- **[issue-1633] Brain notes can now flow into the creative Catalog — making it the one place your ideas, characters, and scenes collect as reusable ingredients.** Two new paths: (1) every Brain record (idea, journal memory, project, admin item, person) gets a **Send to Catalog** button that runs it through the same extract→review→commit pipeline as paste/URL/voice, so a captured thought becomes typed ingredients (characters/scenes/ideas/…) you confirm before keeping. (2) A sticky **Creative** toggle on the Brain capture box (Inbox + Quick Capture) flags a note as a creative idea at capture time — todos and links stay out — and a **Send N creative notes → Catalog** action in the inbox batches the flagged notes into one ingest review when you're ready. Reuses the existing catalog ingest flow end-to-end; no new data model. (`server/services/catalogIngestSources.js`, `server/services/brain.js`, `client/src/components/brain/tabs/InboxTab.jsx`, `client/src/components/QuickBrainCapture.jsx`, `client/src/pages/CatalogIngest.jsx`)
- **[issue-1722] Batch-sent creative notes now drop out of the inbox once their Catalog ingest is committed — so you can't accidentally send the same notes twice.** Previously, after sending your flagged creative notes to the Catalog, the "N creative notes ready" banner kept showing them and a return trip would re-submit the same text. Now the notes are marked consumed the moment you commit ingredients from that review (not on mere navigation — abandoning the review leaves them re-sendable), so the banner clears itself for the notes you've already turned into ingredients.

## Chief of Staff

- **[issue-1683] CoS daily action-budget can no longer overshoot by one at the boundary.** `completeAgent()` emitted `agent:completed` — whose handler schedules `dequeueNextTask()` and its daily action-budget gate — *before* recording the just-finished action via `recordDomainUsage('cos', …)`. At the exact `maxActionsPerDay` boundary the dequeue read stale usage that didn't yet include the completing action (which was already `status: 'completed'`, so it wasn't counted as in-flight either), admitting one autonomous spawn past the cap. The usage ledger is now written before the `agent:completed` emit so the gate always counts the completing action. (`server/services/cosAgents.js`)

## Federation

- **[issue-1650] Completed Chief-of-Staff agent history now mirrors across full-sync peers.** Each paired machine's CoS history tab becomes a complete replica: a full-sync peer advertises its date-bucketed completed-agent archives (the metadata, transcript, and prompt for every finished agent), and the other machine receiver-pulls any it's missing on the ~60s sweep, verifies each file against its advertised hash before writing, then updates its history index so the arrivals appear. Archives are immutable once an agent finishes, so this is safe append-only replication — nothing is overwritten or merged. It runs only for peers explicitly flagged full-sync; a non-full-sync peer is unaffected. Live/running agents, their terminals, and worktrees stay machine-local as before. (Task-list and claim-metadata federation is tracked separately.) (`server/services/sharing/peerSync.js`, `server/routes/peerSync.js`, `server/services/cosAgents.js`)
- **[issue-1712] The live Chief-of-Staff task list now mirrors across full-sync peers — so two machines never both spawn the same task.** Completing the CoS federation started in #1650, each paired machine now sees the other's live backlog and, crucially, its claims: when one machine claims a task to work it, the other learns of that claim on the ~60s sweep and yields instead of spawning a duplicate agent. Because both machines edit the task files live, the backlogs are merged per task rather than copied over each other — a task you've just claimed is never clobbered by a peer's stale copy, completions converge so a finished task stops showing as in-progress on the other machine, and an expired claim (from a crashed machine) becomes freely re-claimable. It runs only for peers explicitly flagged full-sync; a non-full-sync peer is unaffected. Live terminals, in-flight spawns, and worktrees stay machine-local. (`server/services/sharing/peerSync.js`, `server/services/cosTaskMerge.js`, `server/services/cosTaskStore.js`, `server/routes/peerSync.js`)
- **[issue-1714] Chief-of-Staff task edits now converge to the most recently edited version across full-sync peers.** Building on the task-list federation in #1712: when the same task is edited on two paired machines while it stays in the same state (you re-prioritize it on one, tweak its description on the other), the machines now keep the *newest* edit instead of an arbitrary-but-stable winner. Each task carries an edit timestamp stamped whenever its content changes — but deliberately *not* on a background claim/lease heartbeat, so merely working a task can't make a stale edit win. Older tasks from a peer that predates this still converge safely (a missing timestamp counts as oldest), and task completion still always wins over an in-progress edit. (`server/services/cosTaskMerge.js`, `server/services/cosTaskStore.js`)
- **[issue-1645] Writers Room folders and writing-sprint exercises now federate across peers.** Following works (#1644/#1565), the library's folders and exercise sessions replicate over the per-record peer-sync push pipeline with full last-writer-wins + tombstone semantics. Both tables gained soft-delete columns (`deleted`/`deleted_at`) so a deletion propagates as a tombstone the merge keeps until garbage-collected — the LWW model never propagates a hard delete, so omitting a record would let an out-of-date peer resurrect it. New record kinds `writersRoomFolder` / `writersRoomExercise` are wired through the schema-version gate, wire sanitizer, conflict journal (with restore), peer-sync subscribe/push/merge, tombstone GC, auto-subscribe-on-create, the Instances sync-category toggles, and both storage backends. Exercises predate federation and carry no `updatedAt`, so the wire sanitizer derives a stable LWW key from their existing `startedAt`/`finishedAt`. (`server/services/writersRoom/`, `server/services/sharing/peerSync.js`, `scripts/migrations/139-…`)

## Editorial checks

- **[issue-1697] Health-panel deep-links now honor the triage's session-local check mute.** Muting a check from the Editorial Findings triage (#1602) hid its findings group but the health panel kept advertising that check/category row as clickable — clicking it scrolled to an empty filtered view. The session mute state (`hiddenCheckIds`) was lifted from `EditorialFindingsTriage` up to `PipelineEditorialChecks`, which now subtracts muted checks from the filterable sets the health panel receives; a category stays clickable only while a non-muted check still contributes an open finding to it. (`client/src/pages/PipelineEditorialChecks.jsx`, `client/src/components/pipeline/editorial/EditorialFindingsTriage.jsx`)
- **[issue-1667] Final-part editorial verdicts no longer lose their carried setup context on a packed last chunk.** The cross-chunk "setup so far" digest normally yields to manuscript coverage — but two whole-story checks (`arc.climax-agency`, `emotion.reaction-proportionality`) gate their verdict to the final part and anchor it on a snippet carried in that digest (the climax candidate / the unprocessed event). When the final chunk packed to within a few hundred chars of the provider window, the digest was silently dropped and the final-only finding was missed. These checks now opt into a `reserveSetupDigest` guarantee that trims the final chunk's manuscript tail to make room for the digest (the inverse of the usual yield), scoped to the final chunk and opt-in checks only — every other chunk and every non-reserving check keeps full manuscript coverage. (`server/lib/editorial/checkRegistry.js`)
