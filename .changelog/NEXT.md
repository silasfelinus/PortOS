## Fixed

- **Pipeline arc regenerate ŌĆö preserve child issues when the LLM mints fresh season uuids.** The follow-up to the `arc/resolve` remap shipped earlier today: the "Regenerate arc" button (`POST /api/pipeline/series/:id/arc/generate` with `commit:true`) was writing brand-new season ids on every run without remapping child issues ŌĆö every regenerate stranded every issue behind a defunct `seasonId`. Hoisted the resolve path's remap+reassign body into a shared `commitSeasonsWithRemap(currentSeries, { arc, seasons })` helper on `arcPlanner`; the `/arc/generate` commit branch now routes through it, so a regenerate now reattaches existing issues to the new seasons by normalized title ŌåÆ unique `number` ŌåÆ positional fallback (exact mirror of the resolve path). Also added `scripts/repair-orphaned-issues.js --series=<id>` as a one-shot data tool for installs already affected by the gap ŌĆö runs dry-run by default, requires `--apply` to write, refuses non-1:1 orphan/season counts.

- **Pipeline arc resolve ŌĆö preserve child issues when the LLM rewrites the seasons array.** Auto-resolve (`POST /api/pipeline/series/:id/arc/resolve`) replaces `series.seasons[]` with whatever the LLM returns. The prompt instructs the LLM to "preserve volume `id` for every existing volume you keep", but when it ignored that and minted fresh UUIDs (or returned seasons without ids), every child issue still pointed at the now-defunct seasonId and vanished from the Arc canvas ŌĆö the UI groups issues by `seasonId` and never iterated the orphan buckets. `resolveVerifyIssues()` now tags each merged season with its provenance, derives the set of dropped old seasons, and builds an oldŌåÆnew remap (normalized title ŌåÆ unique `number` ŌåÆ positional fallback when counts line up). Affected child issues are migrated to the matched new seasonId (or `null` = ungrouped, which is visible) in the same write, mirroring `deleteSeason`'s bulk-reassign idiom: `updateIssue(ŌĆ”, { skipRenumber: true })` per call + one `recomputeIssueNumbersForSeries` after, wrapped in `withReexportSuppressed('series', seriesId, ŌĆ”)` so a 96-issue series doesn't fan out N socket emits + N debounced re-exports. As a backstop, `ArcCanvas.jsx` now buckets any issue whose `seasonId` isn't in the current `seasons[]` under the existing "Unassigned issues" group instead of a key nothing renders ŌĆö so non-resolve-path orphans (stale data, future bugs) stay visible too.

- **Sharing exporter ŌĆö fall back to per-image metadata sidecar when the media-job archive has aged out.** The `mediaJobQueue` archive trims completed jobs after 24h, so subscription re-exports of any universe with older images were spamming `ŌÜĀ’ĖÅ sharing.exporter: imageJobId <id> not found in live queue or archive` for every aged-out job, and the recipient's bundle lost the per-job metadata record (prompt, model, dimensions) used for full-fidelity re-renders. `exportMediaJobAndAsset` now falls back to reading `data/images/<jobId>.metadata.json` (written at gen time by both the codex and local image-gen paths, lifetime tied to the asset file) and synthesizes a job record from it when the in-memory queue/archive miss. The warning now only fires when both lookups miss ŌĆö meaning the image file or its sidecar is actually gone ŌĆö and the message text now includes `or sidecar` to reflect the new lookup path.

- **Universe Builder ŌĆö coalesce per-image share re-exports during a batch render.** When a subscribed universe was rendering a large batch (e.g. 160 prompts), every completed image fired a per-item `emitRecordUpdated('universe', ŌĆ”)` ŌåÆ debounced re-export, walking every historical `imageJobId` in the linked collection and warning for any that aged out of the 24h in-memory media-job archive. The render route now registers each run's job count with `universeBuilderCollectionHook`, which wraps every per-item `addItem` in `withReexportSuppressed` while the run is active and emits a single `emitRecordUpdated` once the last terminal event (completed / failed / canceled) lands. A 160-image batch now schedules one debounced re-export at the end instead of ~one per completion, and the "imageJobId not found in live queue or archive" warnings collapse to a single sweep. Server restart mid-run drops the in-memory tracker entry ŌĆö the hook falls back to the legacy per-image emits for the rest of that run.

- **Media annotations now sync across PortOS instances via shared Drive buckets, with per-author attribution.** Previously, a note or star added on one machine was invisible to peers sharing the same Drive bucket ŌĆö `data/media-annotations.json` lived purely client-local. Storage is now per-author: each instance owns its own `instanceId` sub-entry under each `<kind>:<ref>` key (`{ authors: { [instanceId]: { authorName, starred, note, updatedAt } } }`), so my notes never clobber yours and vice versa. A new manifest kind (`media-annotations`) carries each instance's annotation snapshot into every auto-merge bucket via a stable filename (`annotations-<instanceId>.json`), debounced 2s after the last local edit. The export is per-bucket scoped to assets actually present in that bucket, so a private annotation can't leak into an unrelated share. On import, peer records merge per-author by `updatedAt` LWW and re-broadcast `media:annotation:updated` so open clients refresh without a reload. Stars are also per-author ŌĆö "my favorites" remains my view, not a tug-of-war. The lightbox now shows my note in the editable textarea and "Notes from others" as labeled read-only blocks below (with the peer's display name + their star). Migration `012-attribute-existing-annotations.js` rewraps existing notes under the local instance's id on first run.
