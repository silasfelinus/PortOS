# Unreleased

## Added

- **Custom scheduled tasks per app** — Each managed app's Automation tab gains a "Custom Tasks" section where you write your own prompt and pick a schedule (interval or cron), and a Chief-of-Staff agent runs it against that app's repo on cadence — just like the built-in task types, but fully your own. Per task you choose priority, autonomy level, and whether the agent works in an isolated worktree and opens a PR (or commits/auto-merges) in the target app. Run-now triggers an immediate pass. Custom tasks also appear (with an app badge) in the Chief of Staff → System Tasks list, where the create/edit form now has an optional app-scope picker so any agent job can be pointed at a specific app or left global.
- **Password-protected federated peers** — If one of your PortOS instances sits behind an HTTP Basic auth proxy (Tailscale `serve`, Caddy, nginx), its peers can now reach it: probes were failing with `HTTP 401`. Each peer in the Instances page gains an optional credential — set a username (optional) and password when adding a peer, or click "set credential" on an existing peer. The credential is attached to every outbound hop to that peer (health probes, snapshot sync, per-record push/pull, asset pulls, and the live agent socket relay), so a reachable-but-401 peer now connects. A peer that returns 401/403 is flagged "auth required" in the UI instead of just reading as offline. Credentials are stored locally per peer and are never synced or announced to the peer itself.
- **[issue-708] Digital Twin "Voice" tab — spoken vs. written style** — Paste a transcript of yourself speaking and the Digital Twin compares it against how you write (a pasted sample, or your existing twin documents), surfacing where your spoken voice differs from your writing — formality, verbosity, sentence length, directness, filler words — and suggesting a communication profile for voice contexts that you can apply in one click.
- **[issue-711] "While You Were Away" dashboard card** — A new dashboard widget recaps what your agents did since your last visit: a count of completed runs, success/failure tally, total time worked, and the most recent wins plus anything that failed and needs a look. Each item opens the agents view, and a "Mark as seen" button resets the window to now. It appears on the Everything and Agent Watch layouts and can be added to any layout via Arrange.
- **[issue-708] Digital Twin "Appearance" tab — image identity source** — Upload a photo of yourself and a vision-capable provider extracts your visible appearance and self-presentation (apparent age range, build, hair, style, grooming, vibe, setting, expression). Review and edit the generated write-up, then save it as your "Appearance & Presentation" core identity document so the twin has a grounded sense of how you look. Use a local vision model (LM Studio / Ollama) to keep the photo on-device; the image itself is never persisted, only the derived text.
- **[issue-711] Per-domain autonomy guardrails** — The Chief of Staff config gains a "Domain Guardrails" panel that lets you set how much each domain acts on its own, independently: Brain auto-classify, Memory auto-extract, CoS auto-run, and Messages auto-send. Each is Off (never act automatically — leave it for you), Dry-run (do the thinking and surface what it would do, but don't commit the side effect), or Execute (act automatically — the default, so nothing changes unless you dial a domain back). Dry-run captures and classifies thoughts but holds them for your confirmation, routes extracted memories to pending approval instead of storing them live, lists the autonomous tasks it would have run without spawning them, and skips outbound message forwarding while logging what it would have sent.
- **[issue-713] Pinned & Recent in the sidebar** — The top of the left navigation now shows the pages you've pinned and the ones you visited most recently, so your working set is one click away. Hover (or keyboard-focus) a page row inside any sidebar section to reveal a pin button; pinned pages stay at the top until you unpin them, and Recent tracks your last few pages automatically. Both sections appear only when they have something to show.
- **[issue-715] "Why am I seeing this?" chips on Insights** — Your taste-identity theme cards now carry a tappable provenance chip alongside the confidence badge, the same source-chip affordance the health views use. Tap it to see that the theme was inferred by AI from your taste profile (not something you stated outright) and what would change it — answering more of your profile or regenerating. Confidence (how strong the pattern is) and provenance (how it was derived) are now two distinct, honest signals on every card.
- **[issue-842] Turn a draft into a Creative Director project from the editor** — With live mode enabled in Writers Room, a new "CD Bridge" panel can read the prose around your cursor and propose a short film treatment — a logline, synopsis, an overall visual style, and a handful of filmable scenes (each with its own shot description and duration). Review it in the panel and send it into a brand-new Creative Director project in one click, then jump straight to it via an "Open in Creative Director" link on the work menu. It never overwrites an existing project, and it shares the same opt-in and daily suggestion budget as the inline continuation suggestions.
- **[issue-715] "Why am I seeing this?" chips across Chief of Staff, Goals, and the dashboard** — The same provenance chip now appears wherever PortOS shows you something it derived rather than something you set. The Chief of Staff's actionable-insights banner explains how its top item was derived — a live count of your records (pending approvals, blocked tasks) reads as data-backed, while a success-rate-driven suggestion reads as inferred. A goal's detail panel marks its urgency and activity-budget readings as modeled from your health timeline and linked activities — not numbers you entered — so a derived reading never reads as a fact you stated. On the dashboard, the Proactive Alerts card declares its alerts are inferred from trends in your goals, costs, and system metrics, while the Decision Log card declares its entries are read straight from the ledger of choices the Chief of Staff actually made. Tap any chip for how it was derived and what would change it.
- **[issue-912] Regenerate gallery images to defeat SynthID watermarks** — Generated images gain a "Regenerate" action in the lightbox, next to "Clean", whenever a local FLUX image generator is installed. The existing Clean only strips embedded provenance metadata and can't touch SynthID — Google's invisible per-pixel watermark — which survives by design. Regenerate is the honest defeat path: it round-trips the picture through a local FLUX model at low denoise so composition holds but the watermark signal is overwritten by fresh sampling. It runs entirely on-device, is never applied automatically, and saves the result as a new variant beside the original (which is kept); the lightbox's variant switch lets you toggle between the original and the regenerated copy.

## Changed

- **[issue-878] City photo mode freezes the scene for a clean shot** — When you enter photo mode, the city now holds still once the camera settles into its framing — particles, data streams, weather, and landmark pulses pause — so your postcard captures a deliberate, motion-free frame instead of whatever the animation happened to be doing. The camera still flies smoothly to each preset (and re-animates when you switch shots); the scene only freezes once it arrives. Leaving photo mode resumes the live, always-moving city.

- **City night sky, horizon, and flight controls** — Night is now a moonlit cyber-night rather than a near-black dusk: a galaxy starfield wraps the dome, distant mountains and themed terrain fill the horizon, and the sky dome cross-dissolves with daylight so the stars fade out as day comes up. Buildings gain lit floor bands and brighter facades so dark towers stay readable, and the AI Core spire gets light rings, an additive apex glow, and a point light. First-person exploration is now free-look flight: WASD moves, Q/E descend/ascend, Shift sprints, and the interact key moved to F (E is now "up"); you can fly up and over the city, with building collision disabled above rooftop height. The galaxy texture only loads at night. No change to the daytime look.

- **City navigation is now map-style** — in orbital view, left-drag pans the camera across the city (so you can reach off-center districts without dropping into first-person), scroll zooms, right-drag (or ctrl-drag on a Mac trackpad) rotates, and arrow keys pan. Previously the only way to look away from the center was first-person mode.

- **City time-of-day follows your theme (day/night), with a real daytime look** — the old four-way sunrise/noon/sunset/midnight picker is now Auto/Day/Night. Auto (the default) renders a bright daytime city under light themes and a dark neon city under dark themes; you can still force Day or Night. Daytime is now a genuine bright, sunlit style rather than dusk: the night-tuned color grade, bloom, film grain, and chromatic aberration fade out in daylight; the sky opens to a luminous blue with high-key sun + sky fill; the ground sheds its neon grid and wet-street glow; and buildings shed their neon (edge strips, halos, base glow) to read as light, sunlit solids. The dark neon night look is unchanged. Existing installs pick up the coupling automatically (any previously-saved time-of-day reads as Auto).
- **City labels stay readable in daytime** — in the daytime view, the in-world text labels (app and process names, plus every district readout — goals, health, task queue, memory, Jira, AI core, productivity, backup, federation mesh, voice, review/instance beacons, and the archive-district sign) no longer wash out against the bright sky. At night they keep their neon glow; as day comes up they switch to dark ink with a soft light halo so they read against sunlit buildings and the open sky. Decorative neon signage still dims in daylight, as it should. Works across all eight themes (four day, four night). No change to the night look.

Building bodies, the district/monument plinths and footings, the building window-grid bases, and the upper sky gradient all pick up a tint from the active theme accent (with their original darkness/brightness preserved, so the scene's depth and shading are unchanged — just recolored). ## Fixed

- **[issue-737] Keyframe-interpolation video is smoother on high-memory Macs** — When you interpolate a video between two keyframes, the result no longer stutters on machines with plenty of memory. The clip-length cap that governs motion quality used to be pinned to a conservative 48 GB-safe value, which on any machine clamped interpolated clips so short that motion advanced in juddery near-duplicate pairs. That cap now scales with your Mac's unified memory — a 128 GB machine renders the longer clips needed for smooth motion out of the box, while machines with 64 GB or less behave exactly as before. You can still override it with the `FFLF_LTX2_PIXEL_BUDGET` setting.
- **[issue-915] Chief of Staff dry-run preview now matches what would really run** — In Dry-run autonomy mode, the "would spawn" preview for auto-run system tasks now lists only the tasks the Chief of Staff would actually start — accounting for free agent slots, retry limits, app cooldowns, and per-app caps — instead of every queued task regardless of whether it could run. The preview is now a faithful plan of the next pass.
- **Local image generation no longer thrashes VRAM on large diffusers models (Windows/CUDA)** — The shared diffusers runner behind Z-Image-Turbo, ERNIE, HiDream, and Qwen loaded a model's full weights onto the GPU at once. The big bf16 pipelines are ~19–20 GB, which on a 24 GB card leaves no room for the forward pass's activations — and on Windows the NVIDIA driver answers the overflow with silent "system memory fallback" (spilling to host RAM over PCIe) instead of an out-of-memory error, so every denoising step ran ~50× slower than the GPU is capable of. A single 1024px Z-Image-Turbo image took **35 minutes**. The runner now measures each pipeline's weight footprint and, when it would fill most of the card, switches to diffusers' model CPU offload — keeping only the active component resident (the transformer stays on the GPU across the whole denoising loop while the text encoder and VAE are paged in just for their one-shot pass). The same image now renders in **~1.5 minutes** with live step previews intact, and peak VRAM drops from ~21 GB to ~13 GB. Small models (SD-1.5, SDXL) keep the faster full-residency path. The stepwise live-preview decode is fixed to target the VAE's execution device so previews still render under offload. Override the heuristic with `PORTOS_IMAGE_OFFLOAD=1` (force offload) or `PORTOS_IMAGE_OFFLOAD=0` (force full residency).
- **[issue-934] City archive district no longer overlaps the AI Core on sparse installs** — On installs with few or no active apps, the archived-apps "warehouse" district (and its ARCHIVE DISTRICT sign) sat right on top of the central AI Core landmark — most visibly on an all-archived install with no active apps, where the sign landed on the monument. The district is now floored to always sit clear of the core. Installs with several active apps are unaffected — their layout is unchanged.
- **[issue-894] Numeric edge-case hardening** — Task-learning duration stats can no longer go negative when resetting a task type against an older or hand-edited store, goal velocity reports no trend (instead of a garbage value) when a progress entry has a malformed date, and a stray non-numeric music gain falls back to the default instead of breaking an audio render.
- **Federated peers with PortOS password auth now connect** — If a remote PortOS instance has its built-in password enabled (Settings → Security), peer probes were always returning 401 even with the correct credential set in the Instances UI. The root cause: PortOS's auth gate only accepted session cookies and `Authorization: Bearer` tokens; peer probes send `Authorization: Basic <base64>`, which was silently ignored. The auth gate now also validates HTTP Basic credentials against the stored password hash, so federated probes, syncs, and the live socket relay all authenticate correctly. The backoff-reset fix is bundled: if a peer was deep in exponential backoff (up to 24 h) from prior 401 failures, re-saving the credential in the Instances UI immediately clears the backoff and re-probes without waiting.
- **Autofixer dashboard now loads over Tailscale HTTPS** — On installs with a Tailscale TLS cert, the main app is served over HTTPS, so the sidebar's "Autofixer" link points at `https://<host>:5560`. The standalone Autofixer UI server was hard-coded to plain HTTP, so that HTTPS request hit an HTTP-only listener and failed to connect (e.g. `https://void.taile8179.ts.net:5560/` wouldn't load). The Autofixer UI now serves HTTPS with the same shared `data/certs/` cert when one is present (and hot-reloads it on renewal), falling back to plain HTTP when no cert exists — so its scheme always matches the link the sidebar generates.
- **City daytime sky no longer washes out to white; night reads as a galaxy backdrop** — The bright-day sky and pavement were tuned so light, so the dome and ground turned into a near-white fog sheet over the city; daytime now keeps real blue in the horizon bands and a muted blue-gray street so the city stays legible in full sun. At night the galaxy map is the backdrop, the shader sky-dome gets out of the way, and the moon is a smaller, softer disc instead of an over-bright bloom. The experimental "Dreamworld" sky-theme picker (and its drifting clouds) has been removed — the City now has one canonical look that follows your theme's day/night mode; any previously-saved Dreamworld setting falls back to it automatically.
- **City night sky is a real environment map (the galaxy now also lights the scene)** — The night galaxy was a pair of hand-placed meshes that frequently didn't show and contributed no light. It's now wired through three.js's environment system the way an HDRI is: the equirectangular panorama is assigned to `scene.background` (a true 360° spheremap, visible in every camera direction) and, via `PMREMGenerator`, to `scene.environment` — so the galaxy also tints reflections and lighting on the metallic building/ground materials. The low-resolution (1774×887) AI-generated panorama that read as pixelated was replaced with a crisp 8192×4096 Milky Way (Solar System Scope, CC BY 4.0), brightness/saturation graded for the stylized night. Day is unchanged.
- **City no longer flashes a blank white scene when the GPU drops the WebGL context** — Heavy post-processing plus an always-on frame-readback buffer made Chromium lose the City's WebGL context easily, leaving a white canvas with no recovery. The readback buffer is now only enabled while taking a postcard, the render resolution is capped to keep the context stable, and a lost context is detected and either recovered on restore or surfaced over the page background instead of a white void. Known THREE.js deprecation and expected context-loss console messages are also filtered out so the browser console stays usable. With the post-processing pass gone, the City settings panel drops the now-inert Bloom, Film Grain, Chromatic Aberration, Color Grading, and Exposure controls so nothing toggles an effect that no longer renders.
- **LoRA downloads no longer fail with a spurious `mkdir` error on Windows** — Installing a Civitai LoRA could fail with `UNKNOWN: unknown error, mkdir '…\data\loras'` even though the directory already existed. On Windows, `fs.mkdir(..., { recursive: true })` can intermittently throw `UNKNOWN`/`EPERM`/`EEXIST` — antivirus locks, OneDrive sync, and network/mapped drives all surface spurious failures that the POSIX "recursive is a no-op when the dir exists" contract doesn't anticipate. The shared `ensureDir` helper (and the LoRA installer, now routed through it) swallows the error whenever the path is in fact a directory afterward, so installs succeed on existing directories.
- **Browser download keep-alive no longer spams JSON-parse errors** — The browser service's CDP keep-alive treated any `200 OK` on its DevTools port as a live Chrome endpoint and called `res.json()` on it unconditionally. When another process answers on that port with HTML (e.g. the Vite dev server auto-incrementing onto it), the parse threw `Unexpected token '<', "<!DOCTYPE "...`, which the reconnect loop logged every 2 seconds. The CDP check now validates the response is actually Chrome's JSON (`webSocketDebuggerUrl` present) and treats anything else as "not reachable." Separately, the Vite dev server now uses `strictPort` so it fails loudly instead of silently walking up onto a reserved PortOS port (5555 API, 5556 browser CDP).
