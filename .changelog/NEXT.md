# Unreleased

## Fixed

- **Codex TUI CoS agents no longer get stuck in the sandbox.** The shipped Codex TUI provider pinned `--ask-for-approval never`, which silences approval prompts but leaves Codex's *sandbox* at its default (`workspace-write`) — so headless agents couldn't reach the network (`gh`/`npm`/`git fetch` failed) and could still hit surprise escalation prompts mid-run. Because an explicit `--ask-for-approval` flag also suppressed the auto-injected full-bypass, the shipped default actively defeated the headless posture the CLI codex path already uses. The default now ships `--dangerously-bypass-approvals-and-sandbox` (approvals off + sandbox off + network on), and migration 121 rewrites existing installs whose args still match the old default — custom Codex flags are left untouched.
- AI provider context-window planning is now model-aware: known Claude model windows are used after explicit provider overrides and before local `num_ctx`, and the provider settings UI shows/edits the planning window so mixed-model providers no longer collapse to one blunt 128K budget.
- **AI fixers can no longer take down the whole server with `pm2 kill`.** PortOS runs every managed app under one shared pm2 daemon, so `pm2 kill` / `pm2 delete all` stops *everything* — including PortOS itself. A "Fix with AI" / autofixer agent (which runs its CLI fully unrestricted) could be talked into running one while trying to "restart the app." Three layers now block this: (1) the shell-command allowlist (`validateCommand`) rejects `pm2 kill`/`startup`/`unstartup` and any `pm2 <verb> all` for the manual command runner and autonomous jobs; (2) spawned AI agents (CoS tasks, the autofixer app) get a guarded `pm2` shim prepended to their PATH that intercepts those same destructive subcommands before they reach the real pm2 — a hard guard that holds even under `--dangerously-skip-permissions`; (3) every agent prompt now states the shared-daemon rule and to only `pm2 restart <its-own-process>`. Scoped commands (`pm2 restart my-app`, `pm2 logs my-app`, `pm2 list`) still work.

## Changed

- The **Social Agents** page now lives under the **Chief of Staff** sidebar section instead of **Brain**, in both the sidebar and the ⌘K / voice nav manifest.
- The **Songs** workbench is now called **Rounds** (its real subject is musical rounds — Hey Ho Nobody Home, the quodlibet trio, the "500 Miles" worked example). This is a full-stack rename: the page moved from `/songs` to `/rounds`, the API from `/api/songs` to `/api/rounds`, and the on-disk store from `data/songs.json` to `data/rounds.json` (top-level `songs[]` → `rounds[]`, per-record `partnerSongIds` → `partnerRoundIds`). An auto-running migration (120) converts existing installs on the next boot/update, preserving every saved round, recording, and training-progress record — nothing to do by hand. The ⌘K palette and voice navigation still resolve "songs"/"song" to the page, so old muscle memory keeps working.

## Fixed

- Scheduled improvement tasks (documentation, security, code-quality, performance, test-coverage, accessibility, dependency-updates, ui-bugs, mobile-responsive, release-check) no longer keep an app-name-hardcoded "PortOS" prompt on existing installs. These prompts went generic (`{appName}`) a while ago, but were never versioned — so deployed instances kept the old PortOS-targeted text (and some had it mis-flagged as a user customization). They are now versioned with their prior defaults preserved, and `loadSchedule` self-heals the mis-flag, so every install auto-upgrades to the generic body on the next boot/update without clobbering genuine custom prompts.

## Fixed

- **Manuscript editorial review no longer reports a fully-drafted book as "empty."** The completeness pass injected the entire universe canon (characters + places + objects JSON, which for a large universe can be 800K+ chars / 200K+ tokens) as fixed context on every call. On any model whose context window the canon exceeded, the budgeter's usable-input budget collapsed to zero and the manuscript was sliced to an empty string — so the LLM "reviewed" a blank draft and returned a wall of identical "the drafted manuscript is empty" findings that could never be resolved, hard-pausing the autopilot at editorial review. The canon is now hard-capped to a reference-sized budget and window-aware-trimmed so the manuscript always keeps a budget floor (mirroring the editorial *check* runner's #1459 protection): a frontier-window model reviews the whole book in one pass, a smaller window chunks it — but the draft is never starved to empty.

## Added

- **Context-window size on available-model cards (Settings → Local LLM).** The model cards in the install catalog and the Hugging Face search results now show each model's native context window (e.g. "128K ctx") alongside its size and parameter count — previously this only appeared on already-installed models, so you couldn't compare context windows before deciding what to install. Live Hugging Face results read the true value from the repo's GGUF metadata (`gguf.context_length`, backfilled from the same per-repo fetch that already supplies file sizes); curated catalog entries carry it when it's a documented spec for that build.

- **Music studio (Create → Music).** A new home for generating and organizing music with local OSS tools, with three tabs:
  - **Artists** — reusable musical personas (the audio analogue of Authors): name, genre, bio, musical style, plus a physical description + portrait style used to **generate an artist portrait** with your existing image-gen options, **upload** one, or pick from the gallery.
  - **Albums** — ordered track collections under an artist, with **cover art** you generate (1024×1024 via your image-gen options), upload, or pick from the gallery; add tracks and reorder them.
  - **Tracks** — singles or album members with lyrics + a generation prompt and an audio file you **upload**, **attach** from the shared music library, or **generate on-device** (see below), played back inline.

  Artists, Albums, and Tracks are db-primary PostgreSQL entities (federation-ready, mirroring Authors). Cross-peer sync for them is tracked separately (#1502); for now they're local-only.

- **On-device music generation in the Track editor.** Generate a track's audio locally from its prompt (+ lyrics) — no API key, no network at inference. Three OSS engines behind one picker: **MusicGen** (MLX) and **AudioLDM2** (diffusers) for instrumental/ambient beds, and **ACE-Step** (`INSTALL_ACESTEP=1 bash scripts/setup-image-video.sh`) for full songs *with vocals*, which uses the track's lyrics as conditioning. Engines whose opt-in venv isn't installed show their install hint and gate the Generate button (same posture as the FLUX.2 image venv). The generated WAV lands in the shared music library and attaches to the track with its engine/model/duration metadata.
- **Install additional audio models from HuggingFace.** The Track editor's generator panel can pull extra checkpoints (e.g. a larger MusicGen, an AudioLDM2 variant) straight from HuggingFace into the local cache — same streamed-download path as the image/video model installer — and they become selectable for generation immediately. User-installed models persist in `data/audio-models.json` and can be removed from the same panel.
- **"Audio & Music" category in the Local LLM model installer (Settings → Local LLM).** The Hugging Face model search gained an audio/music output category that surfaces generation models (MusicGen, AudioLDM2, ACE-Step, Stable Audio, Magenta…) — the GGUF filter is relaxed for this category since audio models aren't GGUF. Installing an audio model downloads it into the HF cache and registers it in the **same** shared audio-model registry the Music studio reads (`data/audio-models.json`), so there's one DRY place to manage audio models: install it once from the installer and it's immediately selectable in the Music studio. Models no on-device engine can run yet — and ACE-Step, which uses a fixed checkpoint — are discovery-only with a "Visit" link. Curated suggestions lead the list: **ACE-Step v1.5**, **`google/magenta-realtime-2`**, and **`stabilityai/stable-audio-3-medium`** (the last gated behind a Hugging Face data-sharing agreement).

- **Autopilot now heals episode-level arc contradictions instead of pausing on them.** The arc-verify convergence loop re-verifies and auto-resolves the arc each round, but auto-resolve could previously only rewrite the arc + volume/season synopses — it was explicitly forbidden from touching episodes. So a contradiction that *originated* inside one episode's planning synopsis (e.g. an episode stages an event a later volume reserves as its own "first" occurrence) could never be fixed: every round re-flagged it because the episode text never changed, and the run always hard-paused with "arc verification did not converge." The auto-resolve pass can now return targeted episode-synopsis corrections (keyed by season + episode number), applied at the arc-verify gate **before** any beats/scripts are generated — so the cheap structural layer is made consistent before expensive script generation. Locked episodes are left untouched; already-expanded beats on a corrected episode are cleared so they regenerate from the fixed synopsis. (`pipeline-arc-resolve` prompt + migration 120.)
- **Autopilot can target one format of a multi-format series.** A `comic+tv` series can now be driven to just its comic draft (or just TV) in a single run via a per-run format restriction, so you don't pay to generate, say, 24 teleplays when you only want the comic proof. Absent a restriction the run authors every format the series targets (unchanged default).
- **Autopilot convergence rounds are now configurable.** The series autopilot re-verifies the arc (and re-reviews the manuscript) in bounded auto-resolve loops, and previously hard-stopped after 3 arc rounds / 2 editorial rounds with "did not converge after 3 rounds" — leaving no way to give a stubborn arc more attempts. You can now set **Arc verify rounds** and **Editorial rounds** (0–20, where 0 skips that gate) in Autonomous mode → **Options**; the values persist as a per-install default and are reused on **Resume**. A per-run override still wins over the saved default, which wins over the built-in floor (3 / 2). When a gate still can't auto-resolve within the limit, the pause message now says how many blocking findings remain and points you to edit the arc/manuscript or raise the limit and resume.
- Editorial Checks page now has an AI provider + model selector that configures the editorial pass. Leave it on "Default provider" to use the active/stage provider, or pick a specific provider/model to override the LLM-backed checks for that run.
- App **Automation** tabs now show a banner when scheduled automation is globally paused, with a one-click **Resume**. Manually clicking **Run** on a task now bypasses the global pause — the pause only stops scheduled/autonomous spawning, not explicit user triggers.
- **Vite Dev-UI host guard.** Launching a managed app's Dev UI over Tailscale used to fail with Vite's "Blocked request. This host (…) is not allowed" because Vite ≥5 rejects unknown hosts. The app detail view now checks each online Vite app's config against the host PortOS is served under and shows a warning banner when it would be blocked, with two one-click fixes: **Allow all hosts (auto)** deterministically rewrites the app's `vite.config` to `server.allowedHosts: true` (safe on a private Tailscale network), and **Fix with AI** queues a CoS agent to edit (or create) the app's `vite.config` in its own repo for configs too unusual to auto-edit. Launching by IP/localhost already works since Vite never blocks those.

## Fixed

- **The GitHub Release workflow no longer crashes on a large changelog.** The v2.22.0 release (a ~175KB changelog) failed at the "Create Release" step with `Argument list too long`: the changelog was passed to `softprops/action-gh-release` as the `body:` input, which GitHub materializes as an `INPUT_BODY` environment variable, and a 175KB env var overflows `ARG_MAX` when the action's Node process is exec'd. The body also exceeds GitHub's hard 125,000-character release-note limit (a separate HTTP 422). The workflow now writes the (placeholder-substituted) changelog to a file and feeds it via `body_path:` instead of an env var, and caps it at 120,000 chars on a whole-line boundary — appending a pointer to the full changelog file at the tag when truncated. (`.github/workflows/release.yml`)
- Scheduled-task descriptions in the Automation UI no longer fall back to a raw dasherized label. `claim-work` (the single-source work-tracker router), `code-reviewer-a`/`code-reviewer-b`, and `do-replan` now show real one-line descriptions instead of "claim work" / "code reviewer a", which read as orphaned/legacy tasks. A parity guard test now keeps the description map in sync with the task-type list.
- PM2 standardization no longer assigns ports that are already in use. The analysis now feeds the AI the set of listening/reserved ports (plus the common Vite defaults 5173/5174) to avoid, and a deterministic post-pass reassigns any port that still collides into the 6000–6099 range — so standardizing an app never lands on a port another app already holds.
- Editing an app's ports now persists. Port fields are derived from `ecosystem.config.cjs`, so saving only updated PortOS's registry while PM2 kept the old ports and the next config refresh reverted the change. Saving a changed port now rewrites the matching value in the app's `ecosystem.config.cjs` (the source of truth). Restart the app for the new port to take effect.
- Editing a **shared or derived** port now persists too. When two port fields resolve to the same literal (e.g. the prod UI is served by the API, so `uiPort === apiPort`), the value-keyed rewrite couldn't tell them apart and the edit returned a 422. The PUT route now falls back to a **per-process-label-targeted** rewrite: it locates the specific process block in `ecosystem.config.cjs` and rewrites only the touched label's port (`api`/`ui`/`devUi`), leaving the sibling field — and any same-valued unrelated number — untouched. Edits to ports that live only under a process's `env`/`args` (no top-level field) persist as well, and an explicit `ports: { ui: N, devUi: N }` map with matching values is split correctly (only the edited key changes). A save that mixes a distinct port and a shared port is written **atomically** — both rewrites are computed in memory and the config file is written once, only when every edit is persistable — so a request that's rejected (422) never leaves the config half-changed. The 422 is reserved for the genuinely unpersistable case (the literal isn't in the touched process block — e.g. it lives in a shared `const PORTS = {…}` referenced via `ports: PORTS.server`, or is computed entirely at runtime), so a shared-value edit never falsely reports success while the displayed port silently reverts. For an app whose prod UI is **served by the API** (no separate UI port), the UI port can't be set on its own — it always follows the API port: the stored UI port is normalized to track the API port on every save (so it never freezes at a stale value), and you change it by changing the API port.
