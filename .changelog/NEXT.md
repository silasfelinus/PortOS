- **[issue-1321] Override a LoRA run's base-model precision and memory mode per run.** A training run can now opt into a heavier or lighter frozen base than the one PortOS picks from your free memory, without editing any code: start a run with a **`baseQuant`** (16 = full bf16, or 8 / 4-bit QLoRA) and/or a **`lowRam`** toggle, and that run uses exactly those settings on the mflux engine. - **[issue-1330] Phase-aware stall watchdog catches a wedged GPU mid-training and auto-resumes from the newest checkpoint.** The training job queue's idle watchdog only trips on *total* silence (no runner output for 30 minutes) — but a *soft* GPU hang during mflux/FLUX.2 training is invisible to it, because the trainer's heartbeat and telemetry threads keep emitting lines while the step loop is wedged, perpetually resetting the flat timer. PortOS now runs a **phase-aware** stall detector that keys off the trainer's `STAGE:`/`STEP:` line protocol: once steps are flowing it derives a per-step budget from the *observed* step rate (a few × the slowest recent interval, floored and capped below the flat watchdog) and enforces it **only during `STAGE:training`** — model-load, dataset-encode, mid-training sampling, and inter-segment cooldown stay on the generous flat watchdog so a legitimately slow phase is never false-killed. When training genuinely stalls, the wedged child is **SIGKILLed immediately** (SIGTERM relies on the GIL the hang holds) and the run **auto-resumes from its newest checkpoint** via the existing resume path — bounded to a couple of attempts per run so a config that wedges every segment can't resume-loop forever, after which the run stays failed for manual resume. This recovers *soft* hangs faster; it does **not** address the hard-reboot kernel panics (checkpoint-resume already covers those, since the machine is gone before any userspace watchdog runs). Off-switch: `settings.loraTraining.stallWatchdog = false`. (`server/services/loraTraining/stallDetector.js`, `server/services/loraTraining/index.js`, `server/lib/validation.js`) (#1330)

- **[issue-1335] Import can clean up paste/PDF formatting with AI as it brings a manuscript in.** Text imported from a PDF or paste arrives mangled — hard-wrapped lines, split drop-caps, hyphenated wraps, orphaned quotes — and until now every imported issue needed a manual Reformat pass in the editor afterward. The import Review panel gains an optional **Clean up formatting with AI** checkbox (off by default): when ticked, each issue's excerpt is run through the same word-preserving reformat pass the editor's Reformat button uses, so manuscripts (and comic scripts / screenplays) land already-clean. It's best-effort — a reformat that fails or would change any wording silently keeps the original text, so an import never breaks — and a live per-issue checklist shows progress while it runs. Left off by default because it's one AI pass per issue, so a large import opts in deliberately. (#1335)

- **[issue-1325] Video renders now record the exact runtime they ran on — and you can see it at a glance.** Garbled or "mosaic" video output is often a per-chip/per-runtime numerical quirk rather than an outright crash, and until now a render left no trace of the stack that produced it. Each local video render now stamps a one-line **runtime fingerprint** — the resolved ltx/mlx (or torch+CUDA on Windows) versions plus your chip, OS, and Python — into the render log and onto the saved history record, so a problem render self-documents the environment that made it. The Video Gen status header also shows your host chip/OS and the installed runtimes' versions on demand, without running a render — so a bug report can carry the version info that makes it actionable. (#1325)

- **[issue-1324] Catch corrupt model weights and repair them in one click.** A download that finished with the *right file size but wrong bytes* (an interrupted resumable fetch, bit-rot) silently decodes to garbled "mosaic" renders that a clean re-download fixes — a leading non-hardware cause of bad output. PortOS now checks downloaded model weights for integrity: a fast structural scan of each `.safetensors` header (catches truncation without loading any tensors) runs automatically and is surfaced on the image- and video-gen model status, plus an optional deep per-file checksum scan on demand. When the Video Gen model you've selected has damaged weight files, a **Repair model** banner appears that deletes the bad files and re-downloads clean copies with progress — no need to wipe and re-fetch the whole multi-GB model by hand. (#1324)

- **[issue-1304] Read aloud — proof a manuscript section by ear with synced karaoke highlighting.** Reading prose aloud is one of the most effective proofing techniques: clunky rhythm, tongue-twisters, repeated words, and unnatural sentences are obvious to the ear and invisible to the eye. The Manuscript Editor gets a **Read aloud** button (in the header, next to Impact preview) that opens a read-along panel for the active section. It synthesizes the section's prose with the same local TTS engines that already voice storyboard dialogue (Kokoro/Piper — no new dependency, no network), splitting the text into sentence segments so it can **highlight each sentence as it's spoken** (karaoke-style) and let you click any sentence to jump there. Sync needs no word-level timestamps: the server measures each sentence's WAV duration and the client plays the segments back-to-back, driving the highlight + a progress bar from those durations. A pick-a-narrator-voice control (remembered across sessions) reuses the shared VoicePicker. Each sentence also carries a lightweight **"hard to say" scan** — over-long sentences, hard consonant clusters, alliteration runs, and a word repeated in close proximity get a wavy underline + hover tooltip so likely trouble-spots are visible before and while you listen. Narration is strictly **non-destructive** — it only reads the section's current text and never mutates the manuscript. New endpoint: `POST /api/pipeline/tts/narrate` (text → per-segment audio + duration + readability). - **[issue-1320] Character-LoRA caption lint — warn when identity leaks into captions, and strip it in one click.** When every caption for a character repeats the same invariant features ("white hair, circlet, tooth necklace"), the LoRA binds the look to those *phrases* instead of the trigger token — so the bare trigger renders a generic subject and only the full identity description reproduces the character (the exact failure that surfaced training Freydis). The dataset workbench now analyzes the captioned images, and when a descriptive fragment repeats across ≥80% of captions it shows an **advisory banner** listing the shared fragments (with their hit counts) and a **Strip shared identity from all captions** button that removes those fragments from every caption while preserving the trigger word and the per-shot detail (pose, framing, setting). The strip recomputes the shared set server-side (`POST /api/lora-datasets/:id/strip-shared-fragments`) so it never trusts a stale client list, and it's idempotent. The "Tips for a strong training dataset" panel gains the caption recipe — *describe what changes shot-to-shot, not who the subject is* — so the guidance is on the page before a caption ever drifts. The threshold is high on purpose: a *varying* attribute (a single view or pose) only appears in a slice of the set, so it separates baked-in identity from legitimate variation without flagging the latter. - **[issue-1290] Perspective Lab — rewrite an issue in another character's POV, and see what you learn.** A revision technique that exposes what each POV character knows, wants, and withholds now has a home on the Prose stage. - **Series Autopilot — drive a whole series end-to-end autonomously.** A new `POST /api/pipeline/series/:id/autopilot/start` runs a *conductor* that inspects the series' current state and walks every missing step in order — generate arc → generate episodes per volume → verify arc → per-issue beat sheets → per-issue text stages (prose + scripts) → structural comic-script gate → series-level editorial review → **canon descriptive-integrity gate** → **draft cover + back cover + all interior comic pages** — by composing the existing pipeline service functions (it never re-implements generation). The "what's next" decision is a **pure resolver over the canonical records**, so it is non-destructive and resumable: anything already `ready`/`edited`/rendered is skipped (every pass runs `force:false`), and on restart you just start again and it picks up at the first still-missing step (a partially-built series is never clobbered). Progress streams over SSE (`/autopilot/progress`); `/autopilot/cancel` stops it (propagating to the active child runner); `/autopilot/status` exposes a thin persisted `series.autopilot` marker for resume/paused UI. Gated on the **cos** autonomy domain: `off` rejects the start (409), `dry-run` emits a no-side-effect `plan` of what it would do, `execute` runs and meters the daily cos action budget (pausing when exhausted — every draft render is one budget-gated action, since a comic is many GPU jobs). The arc-verify and editorial loops are **bounded** and **pause for human review** when they can't converge (arc verification can surface fresh findings every pass) — the residual findings are persisted on the marker. Draft renders are fire-and-forget proof-quality jobs whose in-flight job ids are persisted onto each cover/page slot (so a resumed run never re-renders an already-queued slot). Boot recovery demotes a marker stuck at `running` to `paused`. With opt-in `fileGaps`, capability/quality gaps the run can't resolve on its own — an unparseable script, a failing render, a stalled verify/editorial gate, a run-ending error — are filed as **deduped CoS tasks** (scoped to the pipeline app) so they're tracked rather than silently swallowed. The per-issue script gate runs two checks: a structural parse gate (does the script become pages/panels) and a craft-level LLM verify (`pipeline-script-verify` — un-renderable panels, panel-to-panel flow, dialogue/art balance, within-issue continuity, page-turn placement); the craft gate is advisory (surfaces + files blocking findings without hard-pausing, since the gating quality pass is the series-level editorial review). Before any visual production it also runs a **canon descriptive-integrity gate**: every canon noun (character/place/object) that appears where it would be *drawn* — the comic-script panel descriptions, not prose narration — must have a description, or the run **pauses for human review** (an artist can't render a name). The visual-source distinction is deliberate: an off-page character named only in narration is never drawn, so it's a Nouns-stage quality note, not a production blocker. The same check is exposed for the pipeline UI via `GET /api/pipeline/issues/:id/canon-readiness` and `GET /api/pipeline/series/:id/canon-readiness` (read-only, deterministic — reuses the committed canon name-matchers and `getSeriesCanon`). The whole feature is surfaced in the UI by an **Autonomous mode panel** on the series page (`AutopilotPanel`): a Run/Resume button with a run-options popover (draft visuals, file-gaps), live SSE step progress (with the dry-run plan when the cos domain is in dry-run), a Stop control, a paused/done/error banner driven by the persisted marker that lists the residual findings, and a one-click "production readiness" canon check that links each undescribed drawn noun to its issue's Nouns stage. Two pre-existing artifacts caused it: (1) mflux writes a `0000000_checkpoint.zip` from its `num_iterations==0` monitoring block — the *untrained* initial adapter — and `listRunCheckpoints` listed it as deployable even though promoting it is a no-op; (2) every **manual** resume is a fresh job/watcher with an empty `seen` set, so it re-emits checkpoints that already exist and `run.artifacts.checkpoints` accumulates duplicate entries for the same step. `listRunCheckpoints` now collapses to one entry per (filename-derived) step — keeping the record that captured a real loss, tie-breaking on the latest — and drops step 0 entirely, so `promoteCheckpoint(step:0)` correctly returns `CHECKPOINT_NOT_FOUND`. - **Deleting a brain memory now hard-prunes its mapped vector row instead of leaving a dead embedding in Postgres forever.** A user-initiated brain delete used to flow through the bridge's archive path (`UPDATE memories SET status='archived'`), which correctly hides the entry from search but left the row and its 768-dim embedding in the database indefinitely — so clearing a large import (e.g. a 1,482-record ChatGPT export) orphaned ~1,482 vectors that nothing ever reclaimed. The bridge now distinguishes a **genuine local delete** (a `{type}:deleted` / `journals:deleted` event, emitted only by the local `remove()`/`deleteJournal` paths) from a **sync-driven archive** (a peer delete arriving via the event-silent `applyRemoteRecord` → `sync:applied`): a real local delete hard-deletes the mapped memory row + embedding and drops its bridge-map entry, while a synced delete stays soft-archived so a peer un-delete can still resurrect it. A present-but-archived record is always kept soft (archived ≠ deleted). (`server/services/brainMemoryBridge.js`, #1318)

- **A TUI run whose model id is rejected by the backend now fails (and falls back) instead of returning a screen-scrape of the error.** When a Claude Code TUI run hit a non-recoverable model-id rejection — e.g. a bare Anthropic id like `claude-opus-4-8` passed to a Bedrock-backed Claude Code, which answers `API Error (…): 400 The provided model identifier is invalid` and suggests `global.anthropic.…` — the TUI just sat at an unanswered prompt. The one-shot runner saw output go idle, marked the run `success`, and (no `tui-response.txt` having been written) scraped the error *screen* and returned it as the model's "response." Downstream that surfaced as a baffling failure: AI manuscript Reformat reported `AI reformat changed the wording` (its integrity guard correctly rejecting the screen garbage), while other TUI-backed stages silently consumed the noise. The one-shot TUI runner now recognizes Claude Code's terminal model-id rejection (Bedrock 400 "model identifier is invalid" and Anthropic 404 `not_found_error`) and fails the run as `MODEL_NOT_FOUND`, so `runPromptThroughProvider` benches the provider and retries its configured fallback (e.g. `claude-code-tui` → `claude-code-bedrock`, whose Bedrock ids are valid) — turning a silent corruption into a transparent recovery, or, with no fallback, an actionable error. The check is line-anchored, requires the 400/404 to immediately follow the `API Error` prefix (so a retryable 429/500 Claude Code auto-retries is left alone), and is deliberately scoped to the one-shot runner — *not* the shared fallback detector the long-running agent spawn paths use, so an agent that merely echoes the error line in its own output (cat-ing a log, running the tests, investigating this very failure) can't have its run killed and misclassified.
