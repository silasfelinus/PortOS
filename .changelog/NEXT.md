# Unreleased Changes

## Added

- **World Builder for Media Gen.** A new `Media Gen ŌåÆ World Builder` tab (`/media/world-builder`) takes a one-line starter prompt ŌĆö e.g. "moebius and scavengers reign meets Prophet inspired sci fi universe" ŌĆö and asks the LLM of your choice to expand it into a structured prompt set: a positive `stylePrompt` prefix, a `negativePrompt`, and 6ŌĆō10 variations across five canonical buckets (landscapes, environments, characters, structures, vehicles). You can edit any field, save the world, then kick off a batch render that compiles `stylePrompt + variation.prompt` for every selected bucket and queues them through the existing image-gen pipeline (local mflux or Codex; external SD-API not supported for batches). Each batch lands in an auto-named Media Collection ("World: <name> ŌĆö <timestamp>") so the renders stay grouped with the template that produced them. Worlds support full CRUD, the LLM provider/model is per-template (falls back to active provider), and runs are logged with deep-links into the resulting collection. Backed by `server/services/worldBuilder.js` (storage + prompt compilation), `server/services/worldBuilderExpand.js` (LLM call), `server/routes/worldBuilder.js` (REST surface), and `server/services/worldBuilderCollectionHook.js` (hooks media-job completion ŌåÆ collection assignment). State persists in `data/world-builder.json`. 20 new tests cover sanitizers, compile-prompts logic, and the route surface.
- **Workflow tab in Chief of Staff with canonical project-maintenance ordering.** A new `/cos/workflow` page (and a `Workflow` tab in the CoS panel) renders the full set of scheduled tasks and autonomous jobs as a left-to-right pipeline grouped into seven canonical stages ŌĆö Hygiene ŌåÆ Review ŌåÆ Plan ŌåÆ Audit ŌåÆ Build ŌåÆ Report ŌåÆ Ambient ŌĆö so it's obvious which work happens before which. Each card surfaces its own schedule, last-run time, run count, enabled state, and live "due / waiting / disabled / waiting-on-dependencies" badge, plus inline `runAfter` arrows showing hard prerequisites. Backed by a new `GET /api/cos/workflow` endpoint and `server/services/workflow.js` with stage definitions, gate state for jobs, and dependency edges. Two default `runAfter` wirings ship with this: `do-replan` runs after `pr-reviewer` and `branch-cleanup` (so the plan reflects merged reality), and `feature-ideas` runs after `do-replan` (so new feature work lands on a fresh plan). Both can be overridden per task or per app from the existing Schedule tab.
- **Clean button on Image Gen lightbox.** Generated images now expose a Clean control with `Light` and `Aggressive` options that runs the dev-tool Image Cleaner (median denoise + C2PA strip + sharp re-encode) over the file in place and writes the result back to the gallery as `<base>_clean-<level>.png` with the source's prompt/seed/model carried forward in a sidecar. New endpoint `POST /api/image-gen/:filename/clean` (refactors the cleaning core out of `imageClean.js` so both the base64 and file paths share `cleanImageBuffer()`) and a `cleanGalleryImage(filename, level)` client API. Idempotent at a level (re-running Light overwrites the previous Light copy) and the new image is spliced to the top of the gallery without a refetch. 10 new tests cover happy path, level differences, idempotency, traversal/PNG-only/missing-source rejections, and sidecar lineage.
- **Dismissible AI Recommendations on the Learning tab.** Each recommendation in the Chief of Staff Learning Analytics panel now has an X to dismiss it. Dismissals persist to `data/cos/dismissed-recommendations.json` and are filtered out of future loads. For count-based alerts (e.g., "unknown errors occurred 74 times"), the dismissal records the count snapshot and only re-surfaces if the count grows materially worse (Ōēź1.5├Ś and at least +20). Rate-based recommendations stay dismissed until restored. A "Show dismissed" expander lets you restore individual entries or clear them all.

## Fixed

- **World Builder expansion no longer 0-variations on Codex CLI responses.** The expand-starter step had three failure modes against Codex's CLI provider that all manifested as "Expanded into 0 variations": (1) Codex echoes the user prompt back to stdout before the model's reply, and the prompt itself contains a JSON-shaped *schema example* (`{ "label": string (max 80 chars), "prompt": ŌĆ” }`) ŌĆö its braces balance but its contents are not valid JSON, so the previous first-balanced-block extractor crashed on it; (2) Codex occasionally emits an orphan `}` between a variation's close-brace and the array's `]` (`ŌĆ”blister"}}]}}}`), making `JSON.parse` throw on otherwise-recoverable output; (3) some prompt variants embed small literal `{ "label": ŌĆ”, "prompt": ŌĆ” }` example variations that parse cleanly but are not the response, so the extractor would happily return the wrong object. `extractJson` now (a) walks every top-level brace-balanced block and tries each in order, (b) prefers a block that has any of `stylePrompt` / `negativePrompt` / `categories` at the top level over an example-shaped one, and (c) attempts cheap repairs for trailing commas and the orphan-brace `}}]` ŌåÆ `}]` corruption before giving up. The expand path also logs the toolkit `runId` plus per-category variation counts so a 0-variation outcome is debuggable from `data/runs/<id>/output.txt`. Four new tests cover the Codex prompt-echo, orphan-brace, trailing-comma, and shape-preference cases.
- **CoS review-loop tasks now actually loop until clean and merge the PR.** Previously, scheduled or user tasks running with worktree + openPR + reviewLoop only fired one Copilot review request and stopped ŌĆö the worktree got cleaned up immediately, the PR sat with whatever feedback Copilot returned, and nothing addressed comments or merged. The fix adds a follow-up internal task that the original cleanup hook spawns once the PR is open and the initial Copilot review has been requested. The follow-up agent attaches a fresh worktree to the PR branch (via a new `existingBranch` option on `createWorktree`), runs the full `/do:rpr`-style poll/fix/push/re-request loop until Copilot returns zero unresolved comments (or hits a 10-iteration guardrail), then merges the PR with `gh pr merge --squash --auto --delete-branch`. The follow-up's own cleanup pass passes `skipMerge: true` so the worktree branch isn't re-merged locally on top of the squash-merge gh just executed. GitLab MRs and other non-GitHub forges still skip the loop entirely (Copilot reviewer is GitHub-only). See `server/services/agentLifecycle.js` (`spawnReviewLoopFollowUp`) and `server/services/agentPromptBuilder.js` (`reviewLoopFollowUpSection`).
- **Dashboard form input no longer wipes when rotating between portrait and landscape.** `DashboardGrid` was returning two structurally different render trees on either side of the 640px breakpoint ŌĆö a flat `<div className="space-y-4">` on mobile and a relative-positioned absolute-grid wrapper on desktop. Crossing the breakpoint (which happens any time you rotate an iPhone, since portrait ~390px is below it and landscape ~844px is above) put the widget at a different DOM depth, so React unmounted every widget and remounted a new one ŌĆö taking any in-progress `useState` value with it. The Quick Capture text field reset to empty mid-typing. The grid now uses a single render tree across both modes, toggling only `className`/`style` between flow and absolute layouts. Widget components keep their state across rotation.
- **Toast notifications and the eye-exam form no longer overflow on mobile.** The agent-feedback toast had a hard `w-[480px]` when expanded that overflowed past the right edge of a 375px viewport, and the global toast container's `max-w-[520px]` could similarly extend past the screen on narrow viewports. Both now cap at `calc(100vw - 2rem)` on mobile and switch to their full width at the `sm:` breakpoint (640px+). The eye-exam editor on MeatSpace ŌåÆ Body was also unusable on mobile ŌĆö its `grid-cols-7` layout squeezed seven number inputs into ~42px each at 375px wide; it now reflows to two columns on mobile (with the date field full-width), four on tablet, seven on desktop.
- **Image preview lightbox readable in glass-style day themes.** The MediaLightbox panel (used by Image Gen and Video Gen) inherited `--port-card-alpha: 0.6` from glass themes like Lumen Glass Day, rendering as muddy translucent grey over the modal's `bg-black/90` overlay. Combined with the existing tinted action buttons (`bg-port-accent/20 text-port-accent`), the Remix and Send to Video buttons became nearly invisible. The panel now pins `--port-card-alpha` and `--port-border-alpha` to 1 within the lightbox scope so it renders solid in any theme, and the action buttons switched from translucent-tint to solid-fill (`bg-port-accent text-white`, `bg-port-success text-white`) for high contrast across all day and night themes.
- **Action buttons no longer look disabled in day themes.** Tinted-fill action buttons across the app (e.g. World Builder's "Expand starter ŌåÆ variations", Voice Widget's mode toggles, OpenClaw's status banners) use the dark-mode-friendly `bg-{color}-600/30 text-{color}-200` pattern. In day themes (Lumen Glass Day, Black ICE Day, Blueprint Day) the existing CSS layer remapped `text-{color}-300/400/500` to dark variants for contrast on cream/white surfaces but skipped the lighter `text-{color}-100/200` tints ŌĆö so a "Light text on a light tinted background" combination ended up looking faded and disabled even when enabled. The day-mode text-color remap in `client/src/index.css` now also covers the 100/200 tones for every hue family (purple, violet, cyan, sky, blue, indigo, pink, rose, red, orange, amber, yellow, lime, green, emerald, teal, fuchsia), so the only remaining visual difference between an enabled and disabled action button is the explicit `disabled:opacity-50` (or similar) ŌĆö making the standard "tinted bg + saturated dark text in day mode, faded out when disabled" consistent across all day-mode themes.

## Changed

- **World Builder batch-render form gains the full Image Gen knob set.** The `/media/world-builder` Batch render panel previously exposed only Width / Height / Steps / CFG / Per-variation, and a backend dropdown limited to "Local (mflux)" / "Codex". It now reuses the same controls as the standalone Image Gen page via a new shared `client/src/components/imageGen/ImageGenControls` component: Model picker (auto-populated from `listImageModels`), resolution presets (extracted into `client/src/lib/imageGenResolutions.js` so both pages stay in lockstep), Steps, Guidance, CFG Scale (external mode), Quantize (mflux), plus a `BackendChipStrip` that mirrors Image Gen's chip-style selector and hides external SD-API (which the batch path can't drive). The `/api/world-builder/:id/render` route accepts the new `quantize` field. Successful expansion now auto-saves the world if it already has a name, so clicking "Render" right after "Expand starter ŌåÆ variations" no longer trips "No prompts to render" on a stale on-disk copy.
- **Browser downloads default to `~/Downloads`.** The CDP browser previously sent downloads to `data/browser-downloads/` inside the project; it now writes to the OS-native user Downloads directory (`~/Downloads` on macOS/Linux, `C:\Users\<u>\Downloads` on Windows) via `os.homedir()`. Both the keep-alive `Browser.setDownloadBehavior` call in `browser/server.js` and `PATHS.browserDownloads` in `server/lib/fileUtils.js` were updated so the Browser page UI's listing/delete operations stay in sync with where Chrome actually writes. Existing installs without an explicit `downloadDir` in `data/browser-config.json` pick up the new location automatically; users who want the old behavior can pin it by setting `downloadDir` in that config file.
