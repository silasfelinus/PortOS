## Added

- **Network Exposure dashboard widget.** New widget surfaces how PortOS is actually exposed on the network: HTTPS vs HTTP, bind address + audience (loopback-only / all-interfaces / specific-interface), cert mode (Tailscale-trusted hostname vs self-signed), the loopback HTTP mirror port when HTTPS is on, and a mic-availability hint computed from the browser origin (HTTPS or loopback ‚áí available; HTTP on a Tailscale IP ‚áí blocked by the browser's secure-context rule). Renders a warning banner when bound on all interfaces over plain HTTP. Seeded into the `default` (Everything) and `ops` built-in dashboard layouts so the security posture is visible product UX, not a docs-only invariant. Backed by `GET /api/network-exposure/status`.

- **CoS task form: pick the reviewer for the Review Loop (`--review-with`).** A new compact dropdown appears next to the Review Loop checkbox in `TaskAddForm.jsx` (copilot / claude / gemini / codex; default copilot). The choice is persisted on the task as `metadata.reviewer` and flows through the agent prompt, the follow-up agent's prompt (`/do:rpr --review-with <reviewer>`), and the worktree cleanup hook ‚Äî for non-copilot reviewers, PortOS skips the GitHub Copilot reviewer-API pre-request and instructs the follow-up to invoke the chosen CLI against the PR diff instead. AgentCard chips show the selected reviewer when it isn't the default.

- **Deep-linkable image preview modals via `?preview=<filename>`.** Every page hosting `<MediaPreview>` (Media History, Media Collections, Image Gen, Video Gen, Universe Builder variations + cast, Comic Script Stage, Nouns Stage) now drives the lightbox through a `?preview=<filename>` query param, so `/media/history?preview=foo.png` or `/universe-builder/:id?tab=cast&preview=foo.png` re-opens the same image on reload and is shareable. New `usePreviewRoute(items)` hook (`client/src/hooks/usePreviewRoute.js`) owns the URL round-trip ‚Äî first open pushes a history entry so the browser back button closes the modal; prev/next nav and close replace so the history stack stays clean.

- **Per-bucket exporter hash cache.** Subscription re-exports of large series no longer re-hash every referenced asset to confirm "blob already there." A new `<bucket>/assets/blobs/.index.json` sidecar maps `<sourcePath>:<mtimeMs>:<size> ‚Üí <hash>` so `server/services/sharing/exporter.js#copyAssetIfPresent` skips both `sha256File` (multi-GB stream read) and the redundant `copyFile` when the cache hit's blob is still on disk. mtime change invalidates automatically because it's part of the key. A 200MB re-export drops from ~200MB of disk reads to a single JSON load + stat per asset.

- **Auto-clean after generation ‚Äî per-provider opt-in (codex / local / external).** New `settings.imageGen.{codex,local,external}.autoClean` flag wires the existing image-clean pipeline into the post-generation success path for each backend independently. When on for a provider, every generated PNG is run through `cleanImageBuffer` (sharp `median(3).sharpen()` + C2PA strip) and the cleaned bytes replace the original in place via atomic temp+rename ‚Äî fires BEFORE the SSE `complete` event so subscribers see the cleaned image on first fetch. The sidecar gets `autoCleaned: true`, `cleanLevel: 'aggressive'`, `c2paStripped: <bool>`. A clean failure never fails the generation (logs + falls back to the un-cleaned PNG). Three new checkboxes in Settings ‚Üí Image Gen (one per mode section); the manual Clean button on existing gallery images still produces a separate `_clean-aggressive.png` sidecar copy. Reachable from the lightbox: the metadata panel now shows a "Cleaned" row when the sidecar carries either `autoCleaned` (in-place) or `cleanedFrom` (manual copy) lineage. The clean primitives moved from `server/routes/imageClean.js` to `server/lib/imageClean.js` so the new `autoCleanGeneratedImage` helper is callable from services without crossing the routes‚Üíservices dep direction; the HTTP route is now a thin wrapper.

- **Manual clean now auto-files into the source's collections.** Previously cleaning an image via the lightbox Clean button created `<filename>_clean-aggressive.png` in the gallery, but the cleaned copy didn't appear in any collection the source was part of ‚Äî users had to manually re-add it everywhere. `POST /api/image-gen/:filename/clean` now scans every collection (`listCollections()`), finds the ones containing the source `image:<filename>` key, and calls `addItem()` for the cleaned filename on each. Idempotent across re-cleans (`ERR_DUPLICATE` is swallowed); best-effort overall (collection-write failures log a warning but don't fail the clean).

- **Lightbox: original ‚Üî cleaned variant toggle.** When the previewed image has a manually-cleaned sibling in the same items list (or IS a cleaned sibling), a small segmented control appears at the top of the SettingsPane labeled "View: Original | Cleaned (aggressive)" ‚Äî click flips the lightbox to the other variant without closing the modal. Drives the swap through the host page's existing `setPreview` URL machinery, so the toggle is also a deep-linkable URL change. New `client/src/components/media/variants.js` houses the pure `computeImageVariantGroup(item, items)` helper (unit-tested in `variants.test.js`); `MediaPreview.jsx` computes the group from its `items` prop and threads `variantGroup` + `onSelectVariant` into `MediaLightbox`. Auto-cleaned images replace the original in place (only one file exists) so the toggle correctly does not appear for them.

- **Per-render "Auto-clean this render" checkbox on the Image Gen form** + a quiet fix to make saved auto-clean actually work for codex/local. The Image Gen page (`/media/image`) now exposes a checkbox under the Generate button that lets the user override the saved per-mode `imageGen.{mode}.autoClean` setting for a single render ‚Äî toggles label changes to "(overrides saved default)" when the value differs from the saved one. The checkbox is seeded on mount + every mode switch from the active mode's saved setting, so the default-off case stays unsurprising. Wired through a new `autoClean: z.boolean().optional()` field on the generate route's schema, with multipart string‚Üíbool coercion in `coerceFormFields`. Same fix uncovered a latent bug: codex + local enqueue through `mediaJobQueue` and bypass the dispatcher's autoClean resolution, so the saved settings flag previously had no effect for those modes (only external mode worked, and nobody had caught it yet). The route is now the single resolution point ‚Äî `data.autoClean = body.autoClean ?? settings.imageGen[mode].autoClean` runs once and the value flows through all three dispatch paths verbatim. The dispatcher still resolves autoClean as a safety net for direct callers (avatar route, voice tool). 6 new tests in `imageGen.test.js` cover the precedence matrix (body wins / saved inherited / default false / all three modes).

## Changed

- **Image Cleaner copy: honest about SynthID.** The lightbox Clean tooltip, both auto-clean settings descriptions (Settings ‚Üí Image Gen, and per-render Image Gen form checkbox), the `/devtools/image-clean` page subtitle, the success toast, the file-header comment in `server/lib/imageClean.js`, and the palette nav-manifest keywords (dropped `watermark`) now all explicitly say the cleaner removes the C2PA metadata chunk (verified: confirmed by openai.com/synthid showing "Content Credentials not detected" after a clean of a gpt-image render) but does NOT defeat SynthID. SynthID is embedded in pixel values and Google's published claims state it survives median+sharpen+re-encode by design; gpt-image / Imagen / Gemini renders stay detectable by their vendor watermark checkers after a clean. The prior copy ("strips C2PA provenance + reduces AI artifacts") was technically accurate per-step but implicitly framed the cleaner as a provenance-stripper, which over-promised against SynthID-bearing output. Devtools page also adds a yellow-tagged Note paragraph linking to openai.com/synthid.

- **`isPlainObject` promoted to a named export of `server/lib/objects.js`.** The `(v && typeof v === 'object' && !Array.isArray(v))` check was previously inlined at 16 sites across `server/lib/`, `server/services/`, and `server/routes/`. The helper is now a named export of `objects.js` (surfaces through the `server/lib/index.js` barrel automatically) and all 16 sites import it. Pure refactor ‚Äî no behavioral changes; the helper retains the "non-null, non-array `object`" semantics that match every existing call site (Date / Map / class instances still return true).

- **AI Toolkit "not initialized" errors now carry a typed code + 503 status across runner + promptService.** `server/services/runner.js` (13 sites) and `server/services/promptService.js` (12 sites) previously each inlined `if (!aiToolkitInstance) throw new Error(‚Ä¶)` checks that surfaced as untyped 500s with string-matched messages. Both now go through a private `requireToolkit()` helper that throws `ServerError('AI Toolkit not initialized', { status: 503, code: 'AI_TOOLKIT_NOT_INITIALIZED' })` ‚Äî the same shape `providers.js` already throws. Callers (e.g. `visionTest.js`) can gate on `err.code === 'AI_TOOLKIT_NOT_INITIALIZED'` uniformly regardless of which service raised, and routes that hit any of the three services during toolkit warm-up now return 503 (service-unavailable) rather than a generic 500. No behavior change for the warmed path.

- **Universe Builder cast preview now matches Media History exactly.** Clicking a cast/canon image used to open a stripped-down modal: only Refine Prompt / Add to Collection / Download, no negative prompt, no notes, and the "prompt" field showed the character description rather than the actual prompt that produced the image. The descriptor string is kept as a fallback label only when no sidecar exists (legacy canon renders). Character reference sheets still route through `/data/image-refs/` via a `canon-sheet:<filename>` key so a basename collision with a gallery image can't misroute the preview.

- **First-run no longer silently flips :5555 from HTTP to self-signed HTTPS.** `scripts/setup-cert.js` ran on every `npm start` and, when Tailscale was unavailable, auto-generated a self-signed cert ‚Äî breaking the documented `http://localhost:5555` URL and forcing a browser click-through. It now only auto-generates a self-signed cert when (a) the user explicitly passes `--self-signed`, or (b) a self-signed cert is already present (renewal path). Fresh installs without Tailscale stay HTTP-only on :5555 so the URL in the README and in `setup.sh`'s final banner just works. `setup.sh`'s "Access at:" banner is now cert-aware ‚Äî when HTTPS got provisioned it prints the loopback HTTP mirror (`http://localhost:5553`) and the Tailscale hostname instead of the broken `http://localhost:5555`.

## Fixed

- **[med-pipeline-server-services-pipeline-arcplanner] `sanitizeSeasonList` now warns when LLM-emitted seasons collide on id.** `server/lib/storyArc.js#sanitizeSeasonList` dedupes by id with last-write-wins (Map.set overwrite) on every series persist. Two LLM-emitted seasons sharing an id were silently dropped before persist while child issues kept their `seasonId` pointers ‚Äî leaving the issues attached to the surviving season under the colliding id, which is rarely what an arc-resolve LLM rewrite intended. The helper now tracks colliding ids during the loop and emits a single `console.warn` listing the offending ids (counting both total overwrites and distinct colliding ids) so the responsible run is grep-able in logs. Behavior is preserved (LWW still wins, no throw); the warning is the new signal. Four tests added to `storyArc.test.js` cover the warn, the unique no-warn baseline, the multi-id collision listing, and the same-id-3+-times listing (deduped via Set).

- **[med-migration-scripts-migrations-014-attribute] Migration 014 no longer writes `'unknown'` as a phantom annotation author.** When the migration runner fires before `ensureSelf()` on a fresh install (the order in `server/index.js` ‚Äî migrations at startup, identity later), `instances.json` is absent and the old fallback `instances?.self?.instanceId || 'unknown'` stamped the literal string `'unknown'` into every legacy annotation entry forever (`setAnnotation` refuses to write under `'unknown'` and `annotationsSync.flushAll` refuses to export it, so those notes became unrecoverable / unexportable). `scripts/migrations/014-attribute-existing-annotations.js` now lazy-creates the local identity inline (same shape as `ensureSelf()` ‚Äî `randomUUID()` + `os.hostname()`) and writes it to `data/instances.json` before reading. Read-side belt-and-suspenders: `mediaAnnotations.readAll()` heals any pre-existing `authors.unknown` bucket by re-keying it to the real `localInstanceId` (in-memory only; the first subsequent `setAnnotation` persists the clean shape via the normal write path). When a real local entry already exists alongside an unknown phantom, the real entry wins and the phantom is dropped. Tests in `scripts/migrations/014-attribute-existing-annotations.test.js` and `server/services/mediaAnnotations.test.js`.

- **HF token banner no longer surfaces in Codex / External mode.** The `needsHfTokenGate` predicate in `client/src/pages/ImageGen.jsx` checked `currentModel?.requiresHfToken && !isFlux2Model` but forgot to also require `isLocalMode`. A user who'd selected a gated mflux model (e.g. FLUX.1-dev) and later switched the backend chip to Codex saw the "Flux 1 Dev is a gated model ‚Äî paste your HF token" banner over the Codex form, even though Codex uses gpt-image-2 and never touches HuggingFace. The gate now requires `isLocalMode` first, matching the FLUX.2 install/status block right below it.
