# Unreleased

## Features

### CoS / Scheduled Tasks

- **Plan-item ID system for parallel-safe agent work.** Every `- [ ]` checkbox in a managed app's PLAN.md now carries a stable `[<slug>]` ID, and CoS scheduled `feature-ideas` / `plan-task` jobs encode the slug in their worktree branch name (`cos/<task>/<slug>/<agent>`). Before spawning, the scheduler reads PLAN.md, parses checkboxes, scans `git branch -a` + `gh pr list` for any slug already in flight, and pre-picks the first unclaimed item — concurrent agents now sit on distinct items instead of stomping on the same one. Brainstorm runs (no eligible item) generate a slug at insert time. New helper `server/lib/planIds.js` (slugify + parser + scanner + picker, 17 tests). `taskSchedule.js` prompts bump to feature-ideas v8 / plan-task v4 with a new `{planConstraint}` block; `worktreeManager.createWorktree` accepts `options.planId`; `agentLifecycle.js` plumbs it through both spawn paths. Backed by slashdo v2.15.0 which owns the ID-assignment pass via `do:replan` Phase 0 and preserves slugs across all `do:*` commands that edit PLAN.md.
- **slashdo submodule bumped to v2.15.0.** Brings drift detection in `do:replan` (flags pending items whose execution would regress newer features) and a persistent `Monitor` in `do:rpr` (one stream emits events for new Copilot reviews + CI bucket transitions, replacing the 5+ background subshells that previously accumulated per multi-round review).
- **`/claim` slash command.** Human-driven counterpart to the scheduled `plan-task` agent — at `.claude/commands/claim.md`. Reads PLAN.md, picks the first `- [ ]` whose `[<slug>]` isn't in flight (scans `git branch -a` + `gh pr list` for slug-as-path-segment, skips drift-flagged + NEEDS_INPUT items), creates a `claim/<slug>` worktree off `origin/main`, walks the user through verify → implement → archive (PLAN.md → DONE.md with slug preserved verbatim) → `/simplify` + `/do:pr` → merge → cleanup. Coexists with CoS sub-agents' `cos/<task>/<slug>/<agent>` claims because both place the slug as a `/`-segment visible to either side's in-flight scan. Optional argument lets you target a specific slug for out-of-order work.

### Data Sync

- **Pipeline + universe categories sync over Tailscale.** Sync orchestrator now covers the creative pipeline state — `series.json`, `issues.json`, and `universe-builder.json` — alongside the existing goals / character / digitalTwin / meatspace categories. Record-level only (no image/video blobs; those stay on the share-bucket path). LWW by `updatedAt`; local-only `runs[]` survives every universe merge. The merge runs through dedicated service-level entry points (`mergeUniversesFromSync` in `universeBuilder.js`, `mergeSeriesFromSync` in `pipeline/series.js`, `mergeIssuesFromSync` in `pipeline/issues.js`) so the read-modify-write window lives INSIDE the existing per-file write queue (`queueUniverseWrite` / `queueSeriesWrite` / `queueIssueWrite`) — a concurrent local edit, bible PATCH, season-cover render, or LLM auto-save can't clobber (or be clobbered by) the sync apply. Each entry point also passes incoming remote records through the service's sanitizer (`sanitizeTemplate` / `sanitizeSeries` / `sanitizeIssue`) so older-schema peers (pre-v4 universes missing the canon `kind` field, etc.) land already-migrated. `applyPipelineRemote` returns `{ applied, count, seriesChanged, issuesChanged }` where `count` is records actually changed/added this cycle (NOT total post-merge records — the prior shape over-reported when callers summed cycle deltas). Client `Instances` page adds Universe + Pipeline category meta with Sparkles + Film icons.

### Universe Builder

- **CanonCard provenance chip now names the source series.** The "from series" chip on `CanonCard` (rendered by both the Universe Canon section and the per-issue Nouns stage) now reads `from <Series Name>` when an entry's `sourceSeriesId` resolves through a new `seriesNameMap` prop; the tooltip carries both name and id (`Introduced by series "<name>" (<id>)`). Map is sourced from a new top-level `seriesNameMap` field on the `GET /api/universe-builder/:id/canon-usage` response, which now lists every linked series — not just ones with a prose-match row — so an entry whose `sourceSeriesId` points at a series with no matching prose still resolves to a name. Missing-map / unknown-id cases keep the legacy `from series` label + id-only tooltip. New `SourceSeriesChip` helper extracted in `client/src/components/pipeline/CanonCard.jsx`; 4 vitest cases in `CanonCard.test.jsx` + 2 vitest cases in `server/services/canonUsage.test.js`.
- **Other tab — Auto-sort with AI.** New `POST /api/universe-builder/:id/auto-sort` batches every `kind: 'other'` bucket into one LLM call that classifies each into characters / settings / objects (and optionally suggests a clearer key). Reassignments land atomically via one `updateUniverse` patch (defensive per-entry kind gate filters hallucinated kinds so the sanitizer can't coerce them back to `other`); `suggestedKey` renames surface in the response but stay opt-in. The Other-tab button replaces its old toast stub with a real spinner + result summary. New service `server/services/universeBuilderAutoSort.js` (16 unit tests, including separate prompt-injection pins for ASCII newlines on labels + styleNotes + logline, and for Unicode line/paragraph separators + NEL + form feed + vertical tab on the same three fields) + `server/routes/universeBuilder.test.js` (4 route tests) + `client/src/pages/UniverseBuilder.test.jsx` (2 client tests for the `OtherTab` Auto-sort button — clicks fire `onAutoSort` when enabled; disabled with "Sorting…" label while `autoSorting=true`); client helper `autoSortBuckets` in `apiUniverseBuilder.js`.
- **Other tab — manual trunk assignment.** Each bucket on the Other tab now has an **Assign to…** menu (Cast / Places / Objects) that retags `categories[bucket].kind` so the bucket moves out of Other into the matching trunk on next render. Variations stay in place; per-variation promote-to-canon remains a separate action.
- **Render tab — Other group + dedupe.** Removed the redundant "Prompt set" dropdown from the Batch render card — scope is now always chosen via the Render targets section below. Render targets gained an **Other** group with per-bucket sub-buttons and a `Bulk-render all (N)` button. Top-level action is now **Render everything (N)**.
- **Batch render — full Image Gen surface.** Batch Render now uses the shared `ImageGenSettingsForm`, so switching the backend chip (Local / External / Codex) reveals every per-mode option: model, resolution, steps, guidance/CFG, quantize, seed, **LoRAs**, **style preset**, negative prompt, and extra style. Server-side render schema accepts these as optional per-batch overrides and threads them through prompt composition + every enqueued job (`server/routes/universeBuilder.js` + `server/services/universeBuilder.js`).

### Media Gen

- **Removed duplicate Universe Builder tab.** The MediaGen tab strip no longer shows a Universe Builder entry — the Create sidebar already links straight to `/universe-builder`. Legacy `/media/universe-builder/*` URLs redirect to the top-level routes so bookmarks keep working.
- **Image Gen — multi-reference editing for FLUX.2 (UI + server contract).** A 4-slot reference-image picker appears under the Init Image block whenever a FLUX.2 model is selected. Each slot accepts a PNG/JPG/WebP upload with a 0..1 strength slider; populated slots are packed in submit order and posted as `referenceImage1`…`referenceImage4` multipart fields with a parallel `referenceStrengths` array. New `optionalUploadFields` helper in `server/lib/multipart.js`; the Image Gen route extends `generateSchema` with `referenceStrengths` (max 4 entries, 0..1 each), copies each upload into a sibling `data/image-refs/` directory (NOT the gallery) so refs never surface in the flat `.png` listing, and threads the paths + strengths through `local.js` → `buildArgs` → the FLUX.2 Python runner via `--reference-images`/`--reference-strengths`. The UI shows an "Experimental — no-op today" banner above the slot grid: the Python runner currently ignores these flags pending the `FLUX.2-klein-9B-kv` model swap (see PLAN.md `[flux2-multi-reference-python-runner]`), but accepts them as no-op argparse stubs so single-init renders still work. The route resolves the effective backend (`local` / `external` / `codex`) BEFORE staging any uploads and rejects refs with `REFERENCE_IMAGES_LOCAL_ONLY` for non-local backends + `REFERENCE_IMAGES_FLUX2_ONLY` for non-FLUX.2 models — both gates also unlink the multer-staged temp files before throwing, so a rejected request doesn't leak under `os.tmpdir()`.

### Sharing

- **"🔄 live" badge on inbox subscription rows.** Sharing → Inbox now flags subscription rows whose latest manifest was queued within the last 5 minutes with a small green pill (spinning RefreshCw icon + "live" label). Re-exports fire on every record edit with a 3s server-side debounce and propagate via chokidar in <1s, so a recently-received subscription is a real-time "sender is editing right now" signal. Pure `isLiveSubscription(item, now)` helper exported from `client/src/pages/Sharing.jsx` (handles null inputs + unparseable `receivedAt` + one-shot non-subscription shares without throwing); 5 new vitest cases in `Sharing.test.jsx`. Freshness is computed on render so the badge naturally decays as the page re-renders past the 5-min mark.
- **Per-sender subscription filenames.** `subscriptionFilename()` now includes the sender's instance id in the bucket-side filename (`sub-<kind>-<recordId>-<senderInstanceId>.json`) so two peers subscribed to the same series/universe don't clobber each other's deterministic share files. Inbox dedup keys on `(recordKind, recordId, senderInstanceId)` so a peer's update no longer wipes the other peer's inbox row. `unsubscribe` best-effort cleans up the pre-v2 sender-less filename when its `senderInstanceId` matches this instance, leaving any other peer's pre-upgrade file intact. New `legacySubscriptionFilename` export for the legacy name. Touched `server/services/sharing/{manifest,subscriptions,importer,exporter}.js` + tests.
- **Rotation-orphan inbox cull.** Per-sender inbox dedup means a peer that rotates its `instanceId` (factory reset / new device) would otherwise leave its old identity's row stranded forever. `applyInbox` now culls cross-sender orphans on every subscription-manifest arrival when (a) same `(recordKind, recordId, source)`, (b) different `senderInstanceId`, (c) existing row older than 30 days. The conservative window keeps two legitimately distinct peers with the same display name from being incorrectly culled. Cull removals persist even when the freshness-gate (`inbox-has-newer`) short-circuits the rest of `applyInbox`. 4 new integration tests cover the cull/preserve cases plus the freshness-gate early-return regression.

## Changed

- **Universe Builder — starter idea is no longer capped at 4000 chars.** The starter idea is whatever the user wants to type, from a one-line pitch to a full treatment. The Zod-backed `STARTER_PROMPT_MAX` is raised to 200,000 chars (a sanity ceiling, not an artificial brevity constraint) and the textarea's `maxLength={4000}` attribute is removed.
- **Sidebar + ⌘K label shortened to "Universe".** The Create sidebar link and palette manifest entry for `/universe-builder` now read "Universe" instead of "Universe Builder"; voice/palette aliases still include `universe-builder` so old phrasings keep resolving.

## Fixes

- **Pipeline — comic page lightbox now shows the real generation prompt.** The `Image settings` panel for a rendered series comic page previously displayed `Page 1` (the bare page-index label) instead of the prompt that was actually sent to the image-gen backend. The render slot (`page.proofImage` / `page.finalImage`) always carried `prompt`, but `ComicScriptStage.buildPageItem` was reading `page.prompt` from the page root, which is never populated. Now reads from the slot, and `openPreview` matches the clicked filename against the proof/final slots so the panel surfaces the prompt that produced *this* image (proof and final can diverge when re-rendered independently). `Refine Prompt` now has something real to refine. (`client/src/components/pipeline/stages/ComicScriptStage.jsx`)

## Refactors

- **`canonPrompt` shared helper (`server/lib/canonPrompt.js` + `client/src/lib/canonPrompt.js` mirror).** Single source of truth for per-kind canon descriptor field precedence. Exports `shortCanonDescriptorFragments` (UI-summary scope), `richCanonDescriptorFragments` (render-prompt scope), `descriptorForCanonEntry` (flattened short string), and `hasCanonDescriptorContent` (rich-field union for "has anything to render?" gates). Migrated four call sites: `synthesizeCanonPrompt` in `server/services/universeBuilder.js`, `KINDS[].descFor` in `client/src/components/universe/UniverseCanonSection.jsx`, `canonEntryHasContent` in `client/src/pages/UniverseBuilder.jsx`, and `settingFrags` in `server/lib/scenePrompt.js` + client mirror. Minor cosmetic uniformity: lowercase `palette:`/`era:`/`weather:`/`significance:` prefixes in canon render prompts are now capitalized to match the UI descriptor format — case-only change, no functional impact on diffusion models. 29 new tests in `server/lib/canonPrompt.test.js`.
- **`TabPills` UI primitive (`client/src/components/ui/TabPills.jsx`).** Consolidates ten duplicated tab-nav patterns — two local `TabNav` components (UniverseBuilder, StoryboardPanel) plus eight inline tab-button blocks (Brain, Calendar, ChiefOfStaff, DigitalTwin, MediaGen, Messages, Settings, Wiki). Two variants (`underline` default, `pills`) with knobs for the call-site quirks (`runningKind` per-tab spinner, `stretch` flex-1 equal-width, `mobileDropdown` `<select>` fallback, `hideLabelOnMobile` icon-only at < sm, `controlsIdPrefix` for `aria-controls` wiring, `listRef`+`onScroll` for ChiefOfStaff's scroll-arrow overlay). Covered by 7 vitest cases. Minor intentional visual unification across the surface area: StoryboardPanel active tab text-port-accent (was text-white), ChiefOfStaff and Settings gained the standard `bg-port-accent/5` active wash + `hover:bg-port-card` inactive.
- **CoS — explicit `init()` instead of module-level auto-init.** `server/services/cos.js` previously called `init()` at the bottom of the module, gated by `NODE_ENV !== 'test' && VITEST !== 'true'` so unit tests didn't spin up listeners/timers. The guard was a "test hack in the prod boot path" — `init` is now exported and called explicitly from `server/index.js` alongside `automationScheduler.init()` / `agentActionExecutor.init()` / etc. Test imports of `cos.js` no longer have side effects, and the prod boot path no longer branches on test-env vars.
- **`LoraPicker` extracted** (`client/src/components/imageGen/LoraPicker.jsx`). The LoRA multi-select previously inlined in `ImageGen.jsx` is now a shared component used by both the standalone Image Gen page and the Universe Builder batch-render form.
- **`ImageGenSettingsForm` extended** with `showLoras` + `showStylePreset` props (mounts `LoraPicker` / `StylePresetPicker`) and the previously-hidden CFG / quantize / model-defaults reset.
- **`getStylePresetById` helper** added to `server/lib/writersRoomStylePresets.js` — Map-backed O(1) lookup that replaces the hand-rolled `STYLE_PRESETS.find(...)` pattern.
- **`updateUniverse(id, patch, options)`** API client now passes options through to the underlying `request()` helper, enabling silent error mode.
- **`resolveProviderAndModel({providerId, model})`** added to `server/lib/promptRunner.js`. Collapses the 4 inlined `getProviderById/getActiveProvider/resolveEffectiveModel` chains in `universeBuilderExpand.js` (×2), `universeBuilderRefine.js`, and `universeBuilderPromote.js` onto one helper. The next caller is a one-import addition; the previous pattern was already drifting in error-handling specifics.
- **`findBibleEntryByName(list, name)`** added to `server/lib/storyBible.js` for the "match name OR aliases case-insensitively" predicate. Replaces the inlined `.find(...)` in `universeBuilderPromote.js`'s duplicate-collision check. (`foldRetiredCharactersBucket` keeps its O(1) `Set` index over normalized names+aliases — a per-iteration live-array lookup would be O(n*m) when folding a large retired bucket against a large canon.)
- **`updateUniverse(id, mutator)` callback overload** (`server/services/universeBuilder.js`). `updateUniverse` now also accepts an async `mutator(latest) => patch | null` callback that runs INSIDE `queueUniverseWrite` against the freshest persisted record. `autoSortOtherBuckets` and `promoteVariationToCanon` route their read-modify-write through it — closes the narrow race where a concurrent edit landing between the LLM call and the persist could be silently dropped. Promote's variation removal now re-locates by label inside the queue (the pre-LLM `sourceIdx` is stale by write time) and re-checks the duplicate-name 409 against the latest canon, surfacing `UNIVERSE_PROMOTE_CATEGORY_GONE` / `UNIVERSE_PROMOTE_VARIATION_GONE` when the bucket/variation disappears mid-flight. Duplicate-collision detection lives in a single `findCanonCollision(canon, label, targetKind, secondarySlug?)` helper used by both the pre-LLM short-circuit and the in-queue re-check. +6 server tests pinning the overload contract (happy path, null short-circuit, throw propagation, non-object validation, array rejection, queue-ordering against a parallel literal-patch write).
- **Universe Builder — flush dirty draft before LLM-driven mutators.** `handleAutoSort` / `handlePromoteVariation` / `handleGenerateInCategory` previously operated against the server-persisted universe; if the user had unsaved local edits when they clicked, the server's stale copy got merged back into the draft and silently discarded the pending edits. The page now tracks a `savedDraftSnapshotRef` baseline (via a `markDraftSaved` helper, computed by `draftSnapshotForDirty`) and each LLM handler awaits `handleSave()` first when the draft is dirty, aborting on save failure. `useUniverseAction` gained an optional `preflight` callback that runs AFTER the ref/savedId guards so double-clicks short-circuit before flushing. +4 client tests covering the preflight contract (returns-false abort, throw-as-false, returns-true continue, post-ref-guard ordering).
