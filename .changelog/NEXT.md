# Unreleased

## Changed

- The **Social Agents** page now lives under the **Chief of Staff** sidebar section instead of **Brain**, in both the sidebar and the ⌘K / voice nav manifest.
- The **Songs** workbench is now called **Rounds** (its real subject is musical rounds — Hey Ho Nobody Home, the quodlibet trio, the "500 Miles" worked example). This is a full-stack rename: the page moved from `/songs` to `/rounds`, the API from `/api/songs` to `/api/rounds`, and the on-disk store from `data/songs.json` to `data/rounds.json` (top-level `songs[]` → `rounds[]`, per-record `partnerSongIds` → `partnerRoundIds`). An auto-running migration (120) converts existing installs on the next boot/update, preserving every saved round, recording, and training-progress record — nothing to do by hand. The ⌘K palette and voice navigation still resolve "songs"/"song" to the page, so old muscle memory keeps working.

## Fixed

- Scheduled improvement tasks (documentation, security, code-quality, performance, test-coverage, accessibility, dependency-updates, ui-bugs, mobile-responsive, release-check) no longer keep an app-name-hardcoded "PortOS" prompt on existing installs. These prompts went generic (`{appName}`) a while ago, but were never versioned — so deployed instances kept the old PortOS-targeted text (and some had it mis-flagged as a user customization). They are now versioned with their prior defaults preserved, and `loadSchedule` self-heals the mis-flag, so every install auto-upgrades to the generic body on the next boot/update without clobbering genuine custom prompts.

## Added

- **Autopilot convergence rounds are now configurable.** The series autopilot re-verifies the arc (and re-reviews the manuscript) in bounded auto-resolve loops, and previously hard-stopped after 3 arc rounds / 2 editorial rounds with "did not converge after 3 rounds" — leaving no way to give a stubborn arc more attempts. You can now set **Arc verify rounds** and **Editorial rounds** (0–20, where 0 skips that gate) in Autonomous mode → **Options**; the values persist as a per-install default and are reused on **Resume**. A per-run override still wins over the saved default, which wins over the built-in floor (3 / 2). When a gate still can't auto-resolve within the limit, the pause message now says how many blocking findings remain and points you to edit the arc/manuscript or raise the limit and resume.
- Editorial Checks page now has an AI provider + model selector that configures the editorial pass. Leave it on "Default provider" to use the active/stage provider, or pick a specific provider/model to override the LLM-backed checks for that run.
- App **Automation** tabs now show a banner when scheduled automation is globally paused, with a one-click **Resume**. Manually clicking **Run** on a task now bypasses the global pause — the pause only stops scheduled/autonomous spawning, not explicit user triggers.
- **Vite Dev-UI host guard.** Launching a managed app's Dev UI over Tailscale used to fail with Vite's "Blocked request. This host (…) is not allowed" because Vite ≥5 rejects unknown hosts. The app detail view now checks each online Vite app's config against the host PortOS is served under and shows a warning banner when it would be blocked, with two one-click fixes: **Allow all hosts (auto)** deterministically rewrites the app's `vite.config` to `server.allowedHosts: true` (safe on a private Tailscale network), and **Fix with AI** queues a CoS agent to edit (or create) the app's `vite.config` in its own repo for configs too unusual to auto-edit. Launching by IP/localhost already works since Vite never blocks those.

## Fixed

- **The GitHub Release workflow no longer crashes on a large changelog.** The v2.22.0 release (a ~175KB changelog) failed at the "Create Release" step with `Argument list too long`: the changelog was passed to `softprops/action-gh-release` as the `body:` input, which GitHub materializes as an `INPUT_BODY` environment variable, and a 175KB env var overflows `ARG_MAX` when the action's Node process is exec'd. The body also exceeds GitHub's hard 125,000-character release-note limit (a separate HTTP 422). The workflow now writes the (placeholder-substituted) changelog to a file and feeds it via `body_path:` instead of an env var, and caps it at 120,000 chars on a whole-line boundary — appending a pointer to the full changelog file at the tag when truncated. (`.github/workflows/release.yml`)
- Scheduled-task descriptions in the Automation UI no longer fall back to a raw dasherized label. `claim-work` (the single-source work-tracker router), `code-reviewer-a`/`code-reviewer-b`, and `do-replan` now show real one-line descriptions instead of "claim work" / "code reviewer a", which read as orphaned/legacy tasks. A parity guard test now keeps the description map in sync with the task-type list.
- PM2 standardization no longer assigns ports that are already in use. The analysis now feeds the AI the set of listening/reserved ports (plus the common Vite defaults 5173/5174) to avoid, and a deterministic post-pass reassigns any port that still collides into the 6000–6099 range — so standardizing an app never lands on a port another app already holds.
- Editing an app's ports now persists. Port fields are derived from `ecosystem.config.cjs`, so saving only updated PortOS's registry while PM2 kept the old ports and the next config refresh reverted the change. Saving a changed port now rewrites the matching value in the app's `ecosystem.config.cjs` (the source of truth). Restart the app for the new port to take effect.
- Editing a **shared or derived** port now persists too. When two port fields resolve to the same literal (e.g. the prod UI is served by the API, so `uiPort === apiPort`), the value-keyed rewrite couldn't tell them apart and the edit returned a 422. The PUT route now falls back to a **per-process-label-targeted** rewrite: it locates the specific process block in `ecosystem.config.cjs` and rewrites only the touched label's port (`api`/`ui`/`devUi`), leaving the sibling field — and any same-valued unrelated number — untouched. Edits to ports that live only under a process's `env`/`args` (no top-level field) persist as well. A save that mixes a distinct port and a shared port is written **atomically** — both rewrites are computed in memory and the config file is written once, only when every edit is persistable — so a request that's rejected (422) never leaves the config half-changed. The 422 is now reserved for the genuinely unpersistable case (the literal isn't in the touched process block — e.g. it lives in a shared `const PORTS = {…}` referenced via `ports: PORTS.server`, or is computed entirely at runtime), so a shared-value edit never falsely reports success while the displayed port silently reverts.
