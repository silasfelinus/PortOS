# PortOS — Development Plan

- **Active / claimable work** — open issues labeled
  [`plan`](https://github.com/atomantic/PortOS/issues?q=is%3Aissue+is%3Aopen+label%3Aplan).
- **Parked ideas / not-yet-ready** — open issues labeled
  [`future`](https://github.com/atomantic/PortOS/issues?q=is%3Aissue+is%3Aopen+label%3Afuture)
  (speculative possibilities; promote to `plan` when one becomes a real, claimable task).

Managed by `/do:replan --issues`. For project goals, see [GOALS.md](./GOALS.md);
for completed work, see [.changelog/](./.changelog/) and `git log`.

This file no longer tracks individual tasks (so it stops generating merge
conflicts as work proceeds). Speculative ideas now live as `future`-labeled
issues rather than a list here, so they can each be promoted, refined, or closed
independently.

- [ ] [lora-training-caption-strips-invariant-identity] **Character-LoRA captions must NOT describe invariant identity — warn/strip it, or the trigger token never learns the character.** Surfaced 2026-06-14 training Freydis (dataset `90043893`). The first 4B LoRA (run `d36562a0`) rendered a generic auburn-haired woman from the bare trigger `freydis_of_quaervarr portrait`, but rendered Freydis *correctly* when the full identity description was in the prompt — proving her identity (white hair, circlet, tooth necklace, leather armor) got bound to the caption PHRASES, not the trigger token. Every one of the 25 captions repeated her invariant features. Fix path that worked: rewrite captions to keep only the trigger + what VARIES (pose, framing, weapon, setting, style) and drop invariant identity, so the trigger absorbs it (retrain ~35 min). **Caption source matters:** Freydis's images were `source: upload` with `captionSource: manual` — i.e. **Codex authored the captions externally (from the character canon/lore — hence "Northwind Cairnblade") and they were submitted through PortOS's manual-caption path** (`loraDatasets.js:393`), NOT produced by PortOS's vision auto-captioner. Codex described *the character* (identity-first), which is backwards for trigger-binding. The root fix is the external Codex prompt that authors these captions (it must follow the character-LoRA recipe: describe only what varies); PortOS's role is a safety net (lint/strip) since it can't control external caption authors. Product improvements: (a) PortOS's own auto-captioner (`server/services/loraDatasetCaption.js`) already says "do not mention the character's name" and scopes to pose/clothing/accessories — that path is fine; the canon-generation path is the offender; (b) add a caption-lint/warn in the dataset workbench that flags when the SAME descriptive fragment (hair/eyes/skin/signature items) appears across most captions — that's identity that should move to the trigger; (c) optionally offer a one-click "strip shared identity fragments" that diffs captions and removes the common invariant block (the manual rewrite done for Freydis, generalized); (d) document the character-LoRA caption recipe in the workbench UI ("describe what changes shot-to-shot, not who she is"). Estimated scope: medium.

- [ ] [lora-training-default-base-tier-too-heavy] **Reconsider the LoRA-training memory tier + default steps — unquantized 9B is ~25× slower than 4B on a 128GB box for a character LoRA.** Surfaced 2026-06-14 training Freydis (dataset `90043893`). `deriveMfluxMemoryConfig` (`server/services/loraTraining/runtimes.js`) maps ≥96GB → `{quantize:null}` (full bf16). On the M5 Max 128GB, a `flux2-klein-9b` bf16 run at 768px/600 steps **failed to complete a single real training step in 14 min** (the 32GB base materialized for the fused `value_and_grad` forward+backward hits memory/swap pressure), whereas `flux2-klein-4b` ran steady at **~3.5s/step → ~35min total** (run `d36562a0`). The base-model size dominated; `TRAINING_DEFAULTS.steps` 600→400 was minor. Proposed: (a) for single-character LoRAs default to the 4B base (or 9B-8bit) rather than unquantized 9B even when RAM allows it — speed/fidelity tradeoff strongly favors the lighter base; (b) revisit `TRAINING_DEFAULTS.steps` (600 may be 1.5-2× more than needed for 25-image faces — the checkpoint picker lets the user grab an earlier checkpoint); (c) re-measure whether `low_ram:true` is still warranted on a 128GB box once the base is lighter (the in-RAM-cache rejection was decided during the swap-thrash that coincided with the panics). Expose base-quant/low_ram as request params (currently only steps/rank/lr/resolution/checkpointEvery/sampleEvery are in `loraTrainingParamsSchema`) so a run can opt into a heavier base without a code change. Validate fidelity: confirm a 4B Freydis LoRA is acceptable before changing the default. Estimated scope: medium.

- [ ] [ref-watch-phosphene-model-weights-integrity-repair] **Verify downloaded model weights (header/size + optional checksum) and offer one-click Repair.** From `reference-watch` review of Phosphene (commits `173d2fd5` + `80752a70`, 2026-06-12). Upstream proved that corrupt/partial safetensors (right size, *wrong bytes*) silently decode to garbled "mosaic" video output that a clean re-download fixes — a leading non-Metal cause of bad renders that hits some machines and not others, so it's cross-platform-relevant for PortOS (mac and windows). PortOS currently only sums file sizes: `server/lib/hfCache.js#inspectModelCache()` returns `{ cached, sizeBytes }` with no integrity check. Fix: add a `verifyModelCache(repoId)` to `server/lib/hfCache.js` that does a cheap structural check first (read each safetensors header length prefix and confirm the file is at least header+declared-tensor-bytes long — catches truncation without loading tensors), and an optional deep check that hashes each weight and compares against HuggingFace's published LFS sha256 (from the repo's `git lfs` pointer / `/api/models/<repo>` metadata). Surface results on `GET /api/video-gen/status` (and the imageGen status it mirrors) as a `modelIntegrity` field, add `POST /api/video-gen/models/verify` (force re-scan, threaded/on-demand) and `POST /api/video-gen/models/repair` (delete flagged files + re-download via the existing resumable HF fetch path), and show a dismissible "Repair model" banner in `client/src/pages/VideoGen.jsx` when integrity reports bad/missing files. Estimated scope: large.

- [ ] [lora-training-phase-aware-soft-hang-stall-watchdog] **Phase-aware stall watchdog that SIGKILLs a wedged-but-not-rebooted mflux child and auto-resumes from the newest checkpoint.** From `reference-watch` review of Phosphene (2026-06-14). Phosphene proved an in-process daemon-thread watchdog can't fire during a Metal hang (the completion-handler dealloc chain holds the GIL — `docs/STATE.md:436`); the rescue must be out-of-process. PortOS's Node orchestrator is already out-of-process but its training idle watchdog is 30 min (`WATCHDOG_TRAINING_MS`). Add a **phase-aware** stall detector keyed off the wrapper's existing stage lines (`STAGE:load-model`, `STAGE:training`, `STEP:n:m:loss`, `STAGE:cooldown:heartbeat`): a tight budget during `STAGE:training` (consecutive `STEP:` lines should arrive within a few × the observed per-step time), and a generous/disabled budget during model load, dataset encode, preview generation, and cooldown — then SIGKILL + auto-resume from the newest checkpoint via the existing `--resume-checkpoint` path. **Important caveat / scope:** this does NOT address the observed *hard-reboot* kernel panics (the machine is gone before any userspace watchdog runs — checkpoint-resume already covers that); it only recovers *soft* GPU hangs faster, so it's defensive hygiene, not a fix for the incident. Must avoid false-killing a legitimately slow load/encode (derive the training budget from the first few observed step intervals, not a hardcoded 45s). Unit-test by feeding synthetic line streams. Estimated scope: medium.


- [ ] [chatgpt-import-stream-assets-to-disk] **Stream ChatGPT-export `.dat` assets directly to disk instead of buffering them all in memory.** From claude review of the ChatGPT-ZIP-import PR (#1264). `extractChatgptZip` (`server/services/chatgptZipImport.js`) buffers every `.dat` asset into `pendingAssets` (a `Map<assetId, {buffer}>`) and holds them until the ZIP stream closes, then resolves extensions + writes them in a second loop — because the `conversation_asset_file_names.json` map can arrive *after* the `.dat` members, so the friendly-name fallback isn't known per-asset mid-stream. Peak RAM ≈ sum of all asset bytes; an aggregate `MAX_TOTAL_BUFFERED_BYTES` (1 GB) guard now fails fast rather than OOMing, but a legitimately large export (the route allows up to 2 GB) would be rejected. Proper fix: write each asset to a temp/served path as it streams (sniff the extension from the first chunk, which is enough for all current magic-byte types), recording `{assetId → tempPath}`; after the stream closes, rename/finalize using the now-complete name map (extension is already known from the sniff; the friendly NAME is metadata only, not part of the served filename). This removes the aggregate ceiling and bounds peak RAM to one chunk. Keep the magic-byte sniff (the `.dat` files have no extension). Estimated scope: medium.
